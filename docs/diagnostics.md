# Диагностические средства KiCadSpoke

В этом документе описаны все диагностические инструменты, входящие в состав проекта или рекомендованные для отладки. Они помогают разработчикам и продвинутым пользователям проверять работу **KiCadSpoke**, отлаживать конфигурации, анализировать геометрию, тестировать отдельные операции IPC, а также выявлять и документировать критические баги в самом KiCad.

---

## 1. Встроенные диагностические скрипты (`kicadspoke/diagnostics/`)

Эти скрипты используют актуальный API `kicadspoke` (адаптер, геометрию, конфигурацию) и **не зависят** от устаревших модулей. Все они требуют открытого экземпляра KiCad с активной платой и запускаются из корня проекта командой `python -m kicadspoke.diagnostics.<имя_скрипта>`.

### 1.1. Скрипты для проверки геометрии и IPC

| Скрипт | Назначение | Использование |
|--------|------------|---------------|
| **`diagnostic_keepout.py`** | Загружает конфиг, планирует расстановку, строит keepout из падов IC и компонентов, проверяет попадание позиций компонентов и via в keepout. Выводит подробную информацию для отладки. | `python -m kicadspoke.diagnostics.diagnostic_keepout <config.yaml>` |
| **`get_pad_bbox.py`** | Выводит bounding box пада (размер, позиция) и размер медного слоя. Полезно для проверки геометрии падов. | `python -m kicadspoke.diagnostics.get_pad_bbox --ref IC1 --pad 17 --verbose` |
| **`get_selected_component.py`** | Выводит детальную информацию о выделенных компонентах: refdes, номинал, футпринт, позиция, угол, размер (bbox), список падов (номера, цепи, позиции, размеры) и поле `Role`. Корректно обрабатывает группы. | Выделите компоненты в PCB-редакторе, затем выполните `python -m kicadspoke.diagnostics.get_selected_component` |
| **`get_selection.py`** | Простой список всех выделенных объектов (футпринты, пады, треки, via) с типами и основными параметрами. | Выделите объекты, выполните `python -m kicadspoke.diagnostics.get_selection` |
| **`test_pad_mirror_convention.py`** | Проверяет конвенцию зеркалирования локального смещения пада при флипе (используется в `geometry/pad_projection.py`). Выполняет поворот и флип, сравнивает три кандидата, возвращает компонент в исходное состояние. | `python -m kicadspoke.diagnostics.test_pad_mirror_convention C6 --pad 2` |

### 1.2. Скрипты для тестирования операций IPC (мутирующие)

| Скрипт | Назначение | Использование |
|--------|------------|---------------|
| **`test_create_one_via.py`** | Создаёт одну via рядом с указанным компонентом. Сохраняет UUID в `.last_test_via.json` для последующего удаления. Позволяет проверить `create_items` и транзакции. | Создать: `python -m kicadspoke.diagnostics.test_create_one_via C5 --offset-mm 1.2` <br> Удалить последнюю: `python -m kicadspoke.diagnostics.test_create_one_via --remove` |
| **`test_move_one_cap.py`** | Перемещает один компонент на заданное расстояние по оси X. Позволяет изолировать проблемы с транзакциями (зависание `begin_commit`, `update_items`, `push_commit`). | Сдвинуть: `python -m kicadspoke.diagnostics.test_move_one_cap C5 --delta-mm 1.0` <br> Вернуть: `... --revert` |
| **`test_flip_one_cap.py`** | Проверяет «настоящий» флип компонента через GUI-действие `pcbnew.InteractiveEdit.flip`. Выводит состояние до и после. | `python -m kicadspoke.diagnostics.test_flip_one_cap C6` |

### 1.3. Скрипты для проверки пользовательских полей и ролей

| Скрипт | Назначение | Использование |
|--------|------------|---------------|
| **`test_custom_fields.py`** | Проверяет чтение пользовательского поля компонента (по умолчанию `Role`) через IPC. Выводит все тексты и поля (`Field`) компонента. Критично для проверки работы ролей. | `python -m kicadspoke.diagnostics.test_custom_fields C5 --field Role` |

### 1.4. Скрипт для воспроизведения краша KiCad #24966

| Скрипт | Назначение | Использование |
|--------|------------|---------------|
| **`diagnose_first_write_crash.py`** | Воспроизводит известный баг KiCad 10.0.4: падение на первой записи через IPC при открытом редакторе схем и отсутствии интерактивных правок. Выполняет ступенчатую лесенку (чтения → no-op запись) с замером времени, логированием и сверкой PID. | `python -m kicadspoke.diagnostics.diagnose_first_write_crash` (с опциями `--until 8`, `--delay 30`, `--repeat 3`, `--log file.log`) |

Этот скрипт позволил точно локализовать место краша (`API_HANDLER_EDITOR::checkForBusy`) и подтвердить, что проблема связана с диспетчеризацией запроса к обработчику схемы. Его вывод используется в отчётах об ошибках.

## 2. Внешний инструмент – Охотник за падениями (`hunt-proc.ps1`)

Для автоматического перехвата и анализа падений KiCad (и любых других процессов) разработан PowerShell скрипт **`hunt-proc.ps1`**. Он:

- Цепляет **ProcDump** (из Sysinternals) к каждому запущенному экземпляру отслеживаемых процессов (по умолчанию `kicad.exe`).
- Пишет **полный (Full)** или **мини (Mini)** дамп памяти при первом же фатальном исключении (`C0000005`, `C0000409`, `80000003`) и при завершении процесса (тихий выход тоже даёт снимок).
- Автоматически **вскрывает** дампы с помощью `cdb.exe` (отладчик Windows) – выполняет `!analyze -v`, извлекает `FAILURE_BUCKET_ID` и стек вызовов, сохраняет анализ рядом с дампом.
- При запуске подбирает **невскрытые дампы** прошлых сессий (сироты) и анализирует их в фоне.
- Работает в цикле, отслеживая новые процессы на лету.
- Поддерживает установку как **запланированной задачи** для автоматического запуска при входе в систему.

### 2.1. Конфигурация

Настройки хранятся в файле **`hunt-proc_config.psd1`** (формат PowerShell Data File), который должен лежать рядом со скриптом. Пример:

```powershell
@{
    ProcessNames      = @('kicad')
    DumpDir           = 'D:\Projects\WinDbg\KiCad'
    SymbolCache       = 'D:\Projects\WinDbg\SymbolsCache'
    ProcDumpPath      = 'D:\Utils\ProcDump.exe'
    CdbPath           = 'C:\Program Files (x86)\Windows Kits\10\Debuggers\x64\cdb.exe'
    ExceptionFilters  = @('C0000005', 'C0000409', '80000003')
    MaxDumpsPerAttach = 5
    DumpType          = 'Full'   # или 'Mini' для экономии места
    OrphanMaxAgeHours = 24
    SymbolServers     = @(
        'srv*https://msdl.microsoft.com/download/symbols',
        'srv*https://symbols.kicad.org/kicad-stable'
    )
}
```

### 2.2. Использование

```powershell
# Запуск охоты (основной режим)
.\hunt-proc.ps1

# Ручное вскрытие одного дампа
.\hunt-proc.ps1 -Analyze C:\path\to\dump.dmp

# Установка как запланированной задачи (автозапуск при входе)
.\hunt-proc.ps1 -Install

# Удаление задачи
.\hunt-proc.ps1 -Uninstall
```

### 2.3. Результаты работы

- Дампы сохраняются в `DumpDir` с именами вида `kicad.exe_<PID>_<дата>.dmp`.
- Рядом с каждым дампом создаётся файл `<имя>.dmp.analysis.txt` с полным разбором.
- В `hunter.log` пишутся все события: прицепление к процессу, исключения, вердикты анализа.

### 2.4. Роль в отладке KiCadSpoke и KiCad

Именно с помощью `hunt-proc.ps1` были перехвачены и задокументированы два критических бага KiCad 10.0.4:

- **#24966** – крах на первой API-записи при открытой схеме (`API_HANDLER_EDITOR::checkForBusy`).
- **Дополнительный баг** – null-pointer в `LIB_BUFFER::GetDerivedSymbolNames` во время интерактивного редактирования полей символов (также перехвачен тем же охотником).

Эти дампы и их анализ позволили точно определить место падения и предложить workaround (закрыть схему или сделать интерактивную правку перед запуском). Инструмент остаётся полезным для будущих версий KiCad и для отладки других процессов.

## 3. Рекомендации по использованию диагностических средств

- **Перед запуском мутирующих тестов** (move, flip, create_via) убедитесь, что у вас есть резервная копия платы.
- Для проверки чтения ролей используйте `test_custom_fields.py` – это самый быстрый способ убедиться, что поле `Role` перенеслось на плату.
- Если у вас возникают подозрения на баг KiCad, запустите `hunt-proc.ps1` в фоне перед воспроизведением – он автоматически захватит дамп при падении.
- Для анализа keepout и позиций используйте `diagnostic_keepout.py` с вашим конфигом – он не изменяет плату.
- Для изолированной проверки транзакций используйте `test_move_one_cap.py` – он покажет время каждого шага и позволит отличить проблемы IPC от проблем геометрии.

## 4. Расширение диагностических скриптов

Если вам необходимо добавить новый диагностический скрипт:

1. Разместите его в `kicadspoke/diagnostics/`.
2. Используйте актуальный API `kicadspoke` (адаптер, геометрию, конфигурацию).
3. Если скрипт мутирует плату, явно предупредите об этом в описании и добавьте защиту (например, флаг `--dry-run`).
4. Обновите этот документ, добавив описание скрипта в соответствующую группу.

---

## 5. Связь с отчётами об ошибках KiCad

- **Issue #24966** – подробное описание, стек и условия воспроизведения краша на первой записи IPC. Ссылка на скрипт `diagnose_first_write_crash.py` включена в отчёт.
- **Дополнительный баг** (null в `LIB_BUFFER::GetDerivedSymbolNames`) – также перехвачен охотником, описан отдельно.

Оба бага были обнаружены и задокументированы благодаря диагностическим инструментам KiCadSpoke, что подчёркивает их ценность не только для самого проекта, но и для сообщества KiCad в целом.
