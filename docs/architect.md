# Архитектура KiCadSpoke (v1.0.9)

## 1. Общее описание

**KiCadSpoke** — инструмент командной строки для автоматической расстановки компонентов и переходных отверстий (via) на печатных платах в **KiCad** на основе **шаблонов спиц**. Он подключается к открытому экземпляру KiCad через IPC (библиотека `kipy`) и выполняет:

- Перемещение компонентов (конденсаторов, резисторов, ферритов, кварцев и т.д.) на заданные позиции согласно шаблонам.
- Создание via как на уровне спицы, так и на уровне каждого компонента.
- **Клонирование** повторяющихся функциональных блоков (П-фильтры, каналы ЦАП) в произвольных местах платы с привязкой к якорю или абсолютным координатам.
- Автоматический подбор конкретных экземпляров компонентов по **ролям** и **цепям** без указания refdes.
- Идемпотентность: повторные запуски не дублируют уже правильно стоящие элементы.
- Откат последней операции (undo) через JSON-логирование.
- Извлечение шаблонов из текущего выделения на плате.
- Снятие снимков каналов для иерархических проектов через файловый клонер (без IPC).

В основе лежит понятие **спицы** (spoke) – набора компонентов и via, относящихся к одному выводу микросхемы или к произвольной точке на плате (для клонирования). Вся геометрия спицы описывается в **локальных координатах (along, across)** в виде **шаблона**, который затем можно повернуть, сдвинуть и применить к любому паду или точке.

## 2. Принципы архитектуры

| Принцип | Описание |
|---------|----------|
| **Разделение ответственности** | Чёткое разделение на слои: CLI, конфигурация, валидация, геометрия, адаптер KiCad, бизнес-логика, реестр, логирование, откат. |
| **Инкапсуляция сложности IPC** | Все вызовы к KiCad сосредоточены в адаптере (`KiCadBoardAdapter`), предоставляющем унифицированный интерфейс `IBoardAdapter`. |
| **Расширяемость** | Легко добавлять новые типы шаблонов, ролей, стратегии сопоставления (по выделению/по цепям) и геометрические преобразования. |
| **Безопасность** | Двухфазное выполнение (сначала перемещения, потом via) с промежуточным перечитыванием платы; валидация перед любыми изменениями; логирование и откат. |
| **Идемпотентность** | Повторный запуск не создаёт дублирующих via и не перемещает уже стоящие на месте компоненты благодаря реестру расстановки. |
| **Детерминизм** | Компоненты подбираются из пула в естественном числовом порядке (`C5 < C10`), что гарантирует воспроизводимость. |
| **Прозрачность** | Диагностические скрипты и подробное логирование позволяют легко отлаживать как конфигурацию, так и проблемы IPC. |

## 4. Слои архитектуры (снизу вверх)

### 4.1. Уровень доступа к KiCad (`kicad/`)
- **`adapter.py`** – реализует `KiCadBoardAdapter`, инкапсулирующий все вызовы к `kipy`. Предоставляет методы:
  - поиск компонентов, падов, зон, цепей;
  - получение bounding box'ов (батч-запрос);
  - транзакции (begin_commit, push_commit, drop_commit);
  - чтение пользовательских полей (`get_field_value`);
  - работа с выделением (`get_selected_items` с учётом групп);
  - создание via (`create_via`);
  - удаление по UUID (`remove_by_id`);
  - флип через GUI-экшен (`flip_selected`);
  - проверка на краш первой записи (`check_write_crash_risk` – обход issue #24966);
  - обёртки с ретраями для мутирующих операций.
- **`interfaces.py`** – определяет абстрактный интерфейс `IBoardAdapter`, позволяющий легко подменять адаптер (например, для мок-тестов).

### 4.2. Геометрические утилиты (`geometry/`)
Модули, работающие с координатами, не зависящие от KiCad:
- **`keepout.py`** – `Rect` (AABB), построение keepout из bounding box'ов с запасом, проверка точки на занятость (`point_is_clear`), поиск свободного места по спирали (`find_free_point`) или вдоль линии.
- **`pad_projection.py`** – предсказание позиции пада после перемещения компонента, поворота и (опционально) флипа (зеркалирование локального смещения по оси X).
- **`spoke_layout.py`** – преобразование локальных координат шаблона (`along/across`) в абсолютные для `ManualSpoke`: сначала плоский сдвиг от центра пада, затем поворот. Генерирует via (уровня спицы и компонента) с использованием `rule.net` как цепи по умолчанию.
- **`clone_geometry.py`** – аналогично для `ClonePlacement`, но без привязки к паду IC: используется абсолютная точка (или сдвиг от якоря), и цепи via резолвятся через `net_resolution` (с поддержкой плейсхолдеров).
- **`thermal_grid.py`** – генерация сетки термовиа под площадкой с учётом отступа и паттерна (grid/staggered).

### 4.3. Бизнес-логика (`placement/`)
- **`planner.py`** – класс `PlacementPlanner`, главный координатор:
  - вызывает `ManualPositionCalculator` для `rules` и `ClonePositionCalculator` для `clone_placements`;
  - применяет `skip_existing_components` к компонентам (сравнение позиции, угла, слоя);
  - формирует команды `MoveCommand` и `ViaCommand`.
- **`executor/`** – разбит на модули для читаемости и тестируемости:
  - `FlipManager` – групповой флип через GUI-экшен с батчированием;
  - `MoveExecutor` – исполнение перемещений с проверкой коллизий и записью исходного состояния;
  - `ViaExecutor` – исполнение создания via с регистрацией в реестре;
  - `OperationLogger` – запись JSON-логов (для undo);
  - `BatchExecutor` – фасад, объединяющий перемещения и via, управляющий логированием.
- **`interfaces.py`** – определяет абстрактные интерфейсы `IPositionCalculator` (расчёт позиций) и `IViaPlanner` (планирование via), что позволяет легко подменять реализации.
- **`services/`**:
  - `ComponentPool` – сбор компонентов по ролям и цепи, сортировка, раздача спицам (для `ManualSpoke`).
  - `CloneRoleResolver` – разрешение ролей для `ClonePlacement` в двух режимах:
    - **по выделению** – считывание ролей из выделенных компонентов (с проверкой уникальности и полноты);
    - **по цепям** – явное задание цепей через `nets` или `net_template` с подстановкой `params` и `net_overrides`.
  - `ManualPositionCalculator` – реализует `IPositionCalculator` для `rules`: строит пул, применяет `apply_spoke_geometry`, возвращает `PlacedComponentInfo` и `ViaCommand`.
  - `ClonePositionCalculator` – аналогично для `clone_placements`, используя `apply_clone_geometry` и резолвинг якоря (`_resolve_anchor`).
  - `ViaPlanner` – реализует `IViaPlanner`: отвечает только за термовиа и фильтрацию существующих via через реестр (`skip_existing_components`).

### 4.4. Реестр расстановки (`registry.py`)
- Обеспечивает идемпотентность via между прогонами.
- Сохраняет в JSON-файл информацию о созданных via (UUID, позиция, цепь, параметры) с составными ключами `anchor_id|template_name|role|via_index`.
  - Для `ManualSpoke` anchor_id = `pad:{pad}`.
  - Для `ClonePlacement` anchor_id = `name:{clone.name}`.
- При повторном запуске:
  - сверяет запланированные via с реестром;
  - удаляет устаревшие (изменившие позицию/параметры) по сохранённому UUID и создаёт новые;
  - удаляет via, ключи которых не встретились в новом плане (prune);
  - пропускает уже стоящие правильно.

### 4.5. Разрешение цепей для клонирования (`net_resolution.py`)
- Реализует трёхслойное разрешение имени цепи для `ClonePlacement`:
  1. **Литерал** – если нет плейсхолдеров.
  2. **Плейсхолдер** – подстановка из `params` (например, `{channel}` → `2`).
  3. **net_overrides** – финальная подмена итогового имени (для иерархических путей).

### 4.6. Извлечение шаблонов (`template_extraction.py`)
- Команда `extract`: из текущего выделения в PCB-редакторе извлекает шаблон спицы (компоненты + via) и формирует YAML-секцию `templates`.
- Проверяет уникальность ролей и наличие поля `Role` у компонентов.

### 4.7. Файловый клонер (`cloner/`)
- Позволяет анализировать иерархические проекты без IPC.
- Парсит `.net` и `.kicad_pcb`, строит карту близнецов по внутренним путям и UUID, формирует снимок канала (компоненты, дорожки, виа) для последующего использования в конфигах клонирования.

### 4.8. Валидация (`validation.py`)
- Предварительные проверки **до** любых изменений:
  - существование шаблонов и падов целевого компонента;
  - достаточность компонентов в пуле по ролям для каждой цепи;
  - для `clone_placements`: не более одного клона в режиме «по выделению» за прогон (т.к. в KiCad активно только одно выделение).
- Бросает `ValidationError` с полным списком проблем (не останавливаясь на первой).

### 4.9. Откат операций (`undo.py`)
- Функция `undo_last_operation()` читает последний JSON-лог из `logs/` и восстанавливает плату:
  - возвращает компоненты на исходные позиции, угол и слой (с флипом, если необходимо);
  - удаляет созданные via по UUID.

### 4.10. Пользовательский интерфейс (CLI) – `kicadspoke_cli.py`
- Обрабатывает подкоманды: `apply`, `undo`, `extract`, `clone-extract`.
- Настраивает логирование (консоль + файл).
- Загружает конфиг, подключается к KiCad, выполняет валидацию, планирование и исполнение.
- Поддерживает флаги `--dry-run`, `--clone-placement` (для обработки только одного клона), `--no-collision-check` и др.

## 5. Взаимодействие модулей (последовательность работы)

```
1. Запуск CLI (kicadspoke_cli.py)
   │
   ├── Разбор аргументов (argparse)
   ├── Настройка логирования
   │
2. Загрузка конфига (load_config)
   │
   ├── Чтение YAML
   ├── Проверка уникальности ролей в шаблонах
   ├── Создание объекта Config
   │
3. Подключение к KiCad
   │
   ├── Создание KiCadBoardAdapter
   ├── adapter.refresh_board() — получение платы
   ├── (Если --dry-run — возможно, не требуется)
   │
4. Валидация (run_all_checks)
   │
   ├── check_clone_templates_exist()
   ├── check_single_selection_based_clone()  # не более одного клона по выделению
   ├── check_templates_and_pads_exist()
   ├── check_role_pool_sufficiency()
   │
5. Планирование (PlacementPlanner)
   │
   ├── plan_moves()
   │   ├── Для каждого правила (rules):
   │   │   ├── Строит ComponentPool (по rule.net + ролям)
   │   │   ├── Для каждой спицы:
   │   │   │   ├── pop() — берёт refdes для каждой роли
   │   │   │   ├── apply_spoke_geometry() — вычисляет позиции (Pad.position + shift + rotation)
   │   │   │   └── генерирует ViaCommand (с ключами реестра) и PlacedComponentInfo
   │   │
   │   ├── Для каждого clone_placements:
   │   │   ├── Определяет режим (по выделению или по цепям)
   │   │   ├── Вызывает clone_role_resolver для получения role_to_ref
   │   │   ├── apply_clone_geometry() — вычисляет позиции (origin/anchor + rotation)
   │   │   └── генерирует ViaCommand и PlacedComponentInfo
   │   │
   │   └── Применяет skip_existing_components (проверка позиции/угла/слоя)
   │
   ├── (Если --dry-run — вывод плана и выход)
   │
6. Исполнение (двухфазное)
   │
   ├── Фаза 1: перемещения
   │   ├── executor.execute_moves()
   │   │   ├── Сбор исходных состояний (для undo)
   │   │   ├── Проверка коллизий (опционально, по реальным размерам через bounding box)
   │   │   ├── Флип (если слой изменился) через flip_selected()
   │   │   ├── Перемещение батчами через commit_with_retry (update_items)
   │   │   └── Сохранение лога перемещений
   │   │
   │   ├── adapter.refresh_board() — перечитываем плату
   │   │
   ├── Фаза 2: via
   │   ├── planner.plan_vias()
   │   │   ├── ViaPlanner.filter_existing_vias() — пропускает уже существующие via (skip_existing_components)
   │   │   ├── ViaPlanner.build_keepout() — строит keepout из падов IC и перемещённых компонентов
   │   │   ├── ViaPlanner.plan_thermal_vias() — генерирует термовиа с поиском свободных мест
   │   │   └── Возвращает итоговый список ViaCommand
   │   │
   │   ├── registry.reconcile() — сверяет с реестром:
   │   │   ├── удаляет устаревшие (изменившие позицию/параметры) по UUID
   │   │   ├── удаляет via, которых нет в плане (prune)
   │   │   └── возвращает подмножество via, которые реально нужно создать
   │   │
   │   ├── executor.execute_vias()
   │   │   ├── Создание via батчами через commit_with_retry (create_items)
   │   │   ├── Сбор UUID созданных via (для undo и реестра)
   │   │   ├── registry.record_created() — запись в реестр
   │   │   └── Запись единого JSON-лога операции
   │   │
   └── Вывод результата (успех/ошибки)
```

## 6. Ключевые архитектурные решения

| Решение | Обоснование |
|---------|-------------|
| **Отказ от зависимости via от живой платы** | Все via (кроме термовиа) вычисляются исключительно по геометрии шаблона. Это делает их независимыми от текущего состояния платы и позволяет планировать их до перемещений. |
| **Двухфазное выполнение** | Необходимо для термовиа (чтобы не мешать перемещениям) и для корректности `skip_existing_components` (via создаются после перемещений, чтобы проверить их наличие в финальных позициях). |
| **Идемпотентность через реестр** | Реестр хранит UUID созданных via, что позволяет при повторном запуске пропускать уже существующие, удалять устаревшие (при изменении позиции или prune). |
| **Автоматический подбор refdes по ролям** | Устраняет необходимость вручную прописывать имена компонентов в конфиге, делая его компактным и устойчивым к изменениям нумерации. |
| **Два режима клонирования (по выделению / по цепям)** | Обеспечивает гибкость: для штучных секций — простое выделение мышкой, для многократных блоков — явное задание цепей с параметризацией. |
| **Разрешение цепей через плейсхолдеры и net_overrides** | Позволяет переиспользовать шаблоны в разных контекстах (разные каналы, разные иерархические пути) без дублирования YAML. |
| **Извлечение шаблонов из выделения** | Позволяет создавать шаблоны визуально, без ручного вычисления координат. |
| **Логирование и undo** | Каждая операция сохраняется в JSON, что позволяет откатить изменения. |
| **Обход краша KiCad #24966** | В адаптере реализована проверка `check_write_crash_risk()` и ретраи с задержками, а также обработка `ConnectionError` для диагностики падения KiCad. |
| **Модульные и интеграционные тесты** | Модульные тесты на моках защищают от регрессий; интеграционные тесты с реальным KiCad проверяют сквозную работу. |

## 7. Используемые шаблоны проектирования

| Шаблон | Применение |
|--------|------------|
| **Адаптер** | `KiCadBoardAdapter` предоставляет унифицированный интерфейс для `kipy`. |
| **Команда** | `MoveCommand`, `ViaCommand`, `PlacedComponentInfo` инкапсулируют действия. |
| **Строитель** | `apply_spoke_geometry` и `apply_clone_geometry` собирают `SpokeLayout` из частей. |
| **Хранитель (Memento)** | JSON-логи хранят состояние для undo. |
| **Шаблонный метод** | `BatchExecutor` использует единый шаблон для перемещений и via (батчи, транзакции). |
| **Внедрение зависимости** | Все сервисы получают `adapter` и `config` через конструктор. |
| **Data Transfer Object (DTO)** | Dataclass'ы из `config.py` и `commands.py` — чистые DTO. |
| **Стратегия** | Два режима сопоставления ролей в `clone_role_resolver` (по выделению или по цепям). |
| **Фабричный метод** | `load_config` создаёт объекты конфигурации. |
| **Декоратор (обёртка)** | `_mutating_call` добавляет ретраи и проверку краша к мутирующим операциям. |

## 8. Использование нестабильных и недокументированных API

| Элемент | Статус | Причина использования |
|---------|--------|------------------------|
| `kicad.run_action("pcbnew.InteractiveEdit.flip")` | **Нестабильный** | Единственный способ выполнить «настоящий» переворот компонента с зеркалированием площадок и шелкографии. |
| `Group.proto.items` | **Недокументированный** | Для получения участников группы (свойство `.items` всегда пустое). |
| `footprint.texts_and_fields` | **Недокументированный** | Используется для чтения пользовательских полей (`Field`). |
| `footprint.definition.items` | **Недокументированный** | Используется для получения падов компонента. |
| `board.remove_items_by_id()` | **Новый, потенциально нестабильный** | Добавлен недавно (июль 2025), используется в реестре для удаления via. |
| `board.get_item_bounding_box(list)` | **Недокументированный** | Используется для батч-получения bounding box'ов (вместо несуществующего `fp.getBoundingBox()`). |
| `kicad.get_open_documents()` | **Официальный, но редко используемый** | Используется для проверки наличия открытой схемы (предупреждение о краше #24966). |

Все эти места **документированы** в коде и проверяются модульными тестами.

## 9. Зависимости и требования

- **Python 3.8+** (рекомендуется 3.9+).
- **KiCad 8.0+** (тестировалось на 10.0.4) с включённым IPC-сервером.
- **Библиотека `kipy`** – официальный Python-клиент для IPC KiCad (версия 0.7.1 или новее).
- **Дополнительные пакеты:**
  - `pyyaml` – для чтения/записи YAML.
  - `sexpdata` – для парсинга S-выражений (файловый клонер).
  - `psutil` – опционально, для диагностики (получение PID процессов).
- **Для тестов:** `pytest`, `numpy`, `scipy` (не требуются для работы).

## 10. Заключение

Архитектура KiCadSpoke обеспечивает:

- **Гибкость** – легко добавлять новые шаблоны, роли, стратегии клонирования и геометрические преобразования.
- **Надёжность** – двухфазное выполнение, валидация, идемпотентность, реестр и undo.
- **Удобство использования** – конфигурация без явных refdes, автоматический подбор компонентов, извлечение шаблонов из выделения, клонирование повторяющихся секций.
- **Тестируемость** – чёткое разделение слоёв, интерфейсы, модульные и интеграционные тесты.
- **Прозрачность** – диагностические скрипты, подробное логирование и обход известных багов KiCad.

Это делает инструмент пригодным для использования в реальных проектах по разработке печатных плат, значительно ускоряя рутинные операции размещения пассивных компонентов и фильтрующих цепей.
