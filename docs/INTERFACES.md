## 1. IPC (Inter‑Process Communication) – через библиотеку `kipy`

Используется для **взаимодействия с открытой платой в KiCad** в реальном времени. Все вызовы идут через **gRPC/IPC** к процессу `kicad.exe` / `kicad`.

### Где именно:

- **`kicad/adapter.py`** – весь класс `KiCadBoardAdapter` построен на `kipy`:
  - `refresh_board()` → `self._kicad.get_board()`
  - `get_footprint()`, `get_footprints()` → обращение к плате через `kipy`
  - `get_zone_by_name()` → `self._board.get_zones()`
  - `get_net_by_name()` → `self._board.get_nets()`
  - `get_bounding_boxes()` → `self._board.get_item_bounding_box(list(items))`
  - `begin_commit()`, `push_commit()`, `drop_commit()` – транзакции через IPC
  - `update_items()`, `create_items()` – изменение/создание объектов в плате
  - `flip_selected()` → вызывает GUI‑action `pcbnew.InteractiveEdit.flip` через IPC
  - `create_via()` – создание объекта Via (через IPC)

### Кто использует этот адаптер (т.е. IPC):

- **`placement/planner.py`** – для чтения текущего состояния платы:
  - получение целевого компонента, его падов,
  - получение зоны (`boundary_zone`),
  - получение цепей,
  - получение bounding box’ов падов для построения keepout.
- **`placement/executor.py`** – для применения изменений:
  - флип компонентов,
  - обновление позиций/углов,
  - создание via.
- **`placement/collision.py`** – для получения реальных bounding box’ов компонентов (через адаптер).

**Важно:** IPC используется **только в режиме `apply`** (основная расстановка). Он требует, чтобы KiCad был запущен и плата открыта.

## 2. Прямой парсинг файлов (без IPC)

Используется для **генерации правил** (подкоманда `generate`) – без подключения к KiCad, путём чтения файлов `.net` и `.kicad_pcb` напрямую.

### Где именно:

- **`rules/parser.py`** – модуль, который читает файлы:
  - `parse_net_file(net_path)` – открывает `.net` (S‑выражения) через `open()` и `sexpdata.load()`, извлекает цепи и пины.
  - `parse_pcb_file(pcb_path)` – открывает `.kicad_pcb` (S‑выражения) через `open()` и `sexpdata.load()`, извлекает позиции футпринтов и координаты падов (с учётом поворота).
- **`rules/generator.py`** – использует эти парсеры для построения структуры `Rule`/`Spoke`, затем генерирует YAML.

**Важно:** прямой парсинг **не требует** запущенного KiCad. Он работает только с файлами на диске. Используется исключительно для подготовки конфигурации.

---

## 3. Другие способы доступа к данным (не IPC, не парсинг)

- **`config.py`** – читает YAML-конфиг через `yaml.safe_load()`. Это просто чтение текстового файла.
- **`geometry/*.py`** – чистая математика, без внешнего API.
- **`placer.py`** – CLI, аргументы, логирование.

## Итоговая таблица

| Модуль / файл | Способ взаимодействия | Что делает |
|---------------|------------------------|------------|
| `kicad/adapter.py` | **IPC (kipy)** | Получение/изменение платы в реальном времени |
| `placement/planner.py` | **IPC** (через адаптер) | Планирование расстановки, keepout, via |
| `placement/executor.py` | **IPC** (через адаптер) | Применение команд, флип, создание via |
| `placement/collision.py` | **IPC** (через адаптер) | Проверка коллизий по bbox |
| `rules/parser.py` | **Прямой парсинг** файлов (sexpdata) | Чтение `.net` и `.kicad_pcb` |
| `rules/generator.py` | **Прямой парсинг** (через parser) | Генерация правил из файлов |
| `config.py` | Чтение YAML (pyyaml) | Загрузка конфигурации |
| `geometry/*.py` | Нет внешнего API | Математические расчёты |

---

## Вывод

- **IPC** используется исключительно для **применения расстановки** (режим `apply`), когда нужно взаимодействовать с живой платой.
- **Прямой парсинг** используется только для **генерации правил** (режим `generate`), когда нет необходимости в KiCad, а нужна лишь статическая информация из файлов.

Эти два способа **не пересекаются** в коде: `adapter.py` никогда не читает файлы, а `parser.py` никогда не вызывает `kipy`. Это правильное разделение ответственности.
