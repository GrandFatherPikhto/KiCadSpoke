# Модули верхнего уровня KiCadSpoke

В корневой директории `kicadspoke/` находятся ключевые модули, обеспечивающие загрузку конфигурации, обработку исключений, разрешение цепей, управление реестром, извлечение шаблонов, валидацию, откат операций и точку входа CLI. Каждый модуль решает конкретную задачу и взаимодействует с остальными через чёткие интерфейсы, что обеспечивает модульность, тестируемость и расширяемость.

---

## Обзор модулей

| Модуль | Назначение |
|--------|------------|
| `kicadspoke_cli.py` | CLI-точка входа (подкоманды `apply`, `undo`, `extract`, `clone-extract`). |
| `config.py` | Загрузка YAML-конфигурации в типизированные объекты Python с проверкой уникальности ролей. |
| `exceptions.py` | Иерархия пользовательских исключений и единый форматтер фатальных ошибок. |
| `net_resolution.py` | Трёхслойное разрешение имён цепей для клонируемых шаблонов (литерал → плейсхолдеры → overrides). |
| `registry.py` | Реестр расстановки via для идемпотентности (сохранение UUID, сверка, prune). |
| `template_extraction.py` | Извлечение шаблона спицы из текущего выделения в PCB-редакторе (команда `extract`). |
| `undo.py` | Откат последней операции по JSON-логу. |
| `validation.py` | Предварительные фатальные проверки конфигурации до изменения платы. |
| `constants.py` | Глобальные константы (имена полей, допуски, размеры батчей, таймауты). |
| `__init__.py` | Экспорт публичного API пакета (опционально). |

---

## 1. `kicadspoke_cli.py` – точка входа (CLI)

**Назначение:**  
Главный исполняемый скрипт, обрабатывающий аргументы командной строки, инициализирующий логирование, загружающий конфигурацию, подключающийся к KiCad, выполняющий валидацию, запускающий планировщик и исполнитель, а также поддерживающий команды `undo`, `extract`, `clone-extract` и опциональный флаг `--clone-placement`.

**Основные функции:**

| Функция | Описание |
|---------|----------|
| `setup_logging(verbose, log_file)` | Настраивает логирование: уровень (INFO/DEBUG), вывод в консоль и/или файл. Использует `FlushingFileHandler` для гарантированной записи даже при краше. |
| `cmd_apply(args)` | Команда `apply`: загружает конфиг, подключается к KiCad, запускает валидацию, планирование и двухфазное исполнение (перемещения → refresh → via). Поддерживает `--dry-run` и `--clone-placement` для обработки одного клона. |
| `cmd_extract(args)` | Команда `extract`: извлекает шаблон из текущего выделения на плате и записывает его в YAML-файл. |
| `cmd_undo(args)` | Команда `undo`: находит последний JSON-лог в папке `logs/`, вызывает `undo_last_operation()`. |
| `main()` | Парсит аргументы (поддерживает неявный `apply`), настраивает логирование, вызывает соответствующую команду, перехватывает исключения. |

**Ключевые зависимости:**  
`config.load_config`, `kicad.adapter.KiCadBoardAdapter`, `validation.run_all_checks`, `placement.planner.PlacementPlanner`, `placement.executor.BatchExecutor`, `registry.PlacementRegistry`, `template_extraction.extract_template_from_selection`, `undo.undo_last_operation`, `cloner.extract.extract_channel` (для `clone-extract`).

**Особенности:**
- Поддерживает четыре команды: `apply`, `undo`, `extract`, `clone-extract`.
- В режиме `apply` выполняет двухфазный процесс: сначала перемещения, затем перечитывание платы и создание via (необходимо для корректного учёта термовиа и реестра).
- Флаг `--clone-placement NAME` позволяет обработать только один клон в режиме «по выделению» – это полезно, когда в конфиге несколько клонов, а в KiCad активно только одно выделение.
- В режиме `extract` использует выделение в PCB-редакторе для создания шаблона; проверяет наличие поля `Role` и уникальность ролей.
- В режиме `clone-extract` запускает файловый клонер (без IPC), который анализирует `.net` и `.kicad_pcb`, извлекая снимок канала.
- Все исключения перехватываются и логируются; пользовательские (`PlacerError`) выводятся без стека, а неожиданные – с полным traceback.

## 2. `config.py` – загрузка и хранение конфигурации

**Назначение:**  
Определяет все структуры данных (dataclass'ы), описывающие конфигурацию расстановки, и предоставляет функцию `load_config()` для загрузки YAML-файла в типизированные объекты Python. Включает проверку уникальности ролей внутри шаблона (фатальная ошибка при дублировании) и поддержку новых сущностей: `ClonePlacement`, `TemplateComponentSlot.net_template` и обобщённых via.

**Основные классы (dataclass'ы):**

| Класс | Описание |
|-------|----------|
| `ThermalViaArrayConfig` | Настройки массива термовиа под термопадом (включён, целевой компонент, пад, цепь, сетка, размеры). |
| `TemplateVia` | Описание via в шаблоне: локальные координаты (`offset_along_mm`, `offset_across_mm`), цепь (может быть `None` – тогда наследуется), параметры отверстия. |
| `TemplateComponentSlot` | Слот компонента: роль, локальные координаты, угол, список via (уровня компонента), опциональный `net_template` (для ClonePlacement). |
| `SpokeTemplate` | Полный шаблон спицы: имя, список via (уровня спицы), список слотов компонентов. |
| `ManualSpoke` | Конкретная спица (привязка к паду IC): пад, шаблон, сдвиг, поворот, флаг `enabled`. |
| `Rule` | Правило для одной цепи: имя цепи и список спиц. |
| `ClonePlacement` | Клонируемое размещение (TemplatePlacer): имя, шаблон, абсолютная точка (или сдвиг от якоря), угол, словари `nets`, `params`, `net_overrides`, якорь (`anchor_ref`, `anchor_pad`). |
| `Config` | Главный объект: целевой компонент, сторона, шаблоны, термовиа, правила, клонирования, флаги (`place_components`, `skip_existing_components`, параметры поиска via). |

**Основные функции:**

| Функция | Описание |
|---------|----------|
| `load_config(path)` | Читает YAML, парсит все секции через вспомогательные `_load_*` функции, возвращает объект `Config`. При загрузке шаблонов проверяет уникальность ролей в `components`; при дублировании бросает `ValidationError` с подробным сообщением. |
| `_load_template_via(data)` | Загружает `TemplateVia`. |
| `_load_template_component_slot(data)` | Загружает `TemplateComponentSlot` с учётом `net_template`. |
| `_load_spoke_template(name, data)` | Загружает `SpokeTemplate` и проверяет уникальность ролей. |
| `_load_manual_spoke(data)` | Загружает `ManualSpoke`. |
| `_load_clone_placement(data)` | Загружает `ClonePlacement` с проверкой якоря и обязательности координат. |

**Особенности:**
- Проверка уникальности ролей внутри шаблона – дублирование недопустимо, так как ключ реестра включает роль.
- Поддержка `net_template` для клонирования – позволяет параметризовать имена цепей (например, `DAC{channel}_DB1`).
- Для `ClonePlacement` поддерживаются два режима сопоставления ролей: «по выделению» (без `nets`/`params`) и «по цепям» (с `nets` или `params`). Режим определяется в `clone_role_resolver.clone_uses_selection_mode()`.
- Наследование `net` для via: если `net` не указан в `TemplateVia`, то для `ManualSpoke` используется `rule.net`, а для `ClonePlacement` отсутствие `net` является фатальной ошибкой (нет дефолтной цепи).
- Для `ClonePlacement` поддерживается привязка к якорю (`anchor_ref` + опциональный `anchor_pad`) – в этом случае `origin_x/y` становятся сдвигом от якоря.

## 3. `exceptions.py` – иерархия исключений

**Назначение:**  
Определяет пользовательские исключения для проекта и единую функцию форматирования фатальных ошибок. Все исключения наследуются от базового `PlacerError`, что позволяет перехватывать все ошибки проекта одним блоком.

**Классы исключений:**

| Класс | Назначение |
|-------|------------|
| `PlacerError` | Базовое исключение для всех ошибок планера. |
| `BoardNotFoundError` | Не удалось получить плату из KiCad (обычно после `adapter.refresh_board()`). |
| `ComponentNotFoundError` | Компонент не найден на плате (например, целевой `target_ref`). |
| `GeometryError` | Ошибка в геометрических расчётах (например, слишком большой отступ для термовиа). |
| `ValidationError` | Фатальная ошибка предварительной проверки конфигурации – программа останавливается до изменения платы. |

**Вспомогательная функция:**

| Функция | Описание |
|---------|----------|
| `format_fatal_error(title, problems)` | Форматирует список проблем в единое многострочное сообщение с рамкой из `=`. Используется как в `config.py` (проверки на этапе загрузки YAML), так и в `validation.py` (проверки после подключения к KiCad). Живёт здесь, чтобы избежать циклических импортов. |

**Пример вывода:**
```
======================================================================
  ФАТАЛЬНАЯ ОШИБКА: не хватает компонентов для ролей шаблона
======================================================================
  ✗ цепь +1V2_VCCINT, роль LIGHT: нужно 5, найдено 3
  ✗ цепь +3V3_VCCIO, роль HEAVY: нужно 4, найдено 2
======================================================================
Расстановка остановлена, плата не тронута. Исправьте конфиг и запустите заново.
```

## 4. `net_resolution.py` – разрешение цепей для клонируемых шаблонов

**Назначение:**  
Обеспечивает трёхслойное разрешение имени цепи для `ClonePlacement` (TemplatePlacer). Позволяет подставлять плейсхолдеры из `params` и применять переопределения `net_overrides`. Это ключевой механизм для параметризации клонируемых блоков.

**Основная функция:**

| Функция | Описание |
|---------|----------|
| `resolve_net(net_template, params, net_overrides)` | Принимает шаблон имени цепи (возможно с `{placeholder}`), словарь параметров для подстановки и словарь переопределений. Возвращает итоговое имя цепи. В случае отсутствия параметра для плейсхолдера бросает `ValidationError` с понятным сообщением. |

**Принцип работы:**
1. Если в `net_template` нет плейсхолдеров – возвращает как есть.
2. Иначе выполняет `str.format(**params)`.
3. Затем применяет `net_overrides.get(resolved, resolved)` для точечной подмены (например, для иерархических путей типа `/STM32F4xx/BOOT0` → `/STM32F4xx_2/BOOT0`).

**Используется в:** `placement/services/clone_role_resolver.py` (при разрешении ролей для клонированных размещений) и в `geometry/clone_geometry.py` (при разрешении цепей via).

**Пример:**
```python
resolve_net("DAC{channel}_DB1", {"channel": 2}, {})  # -> "DAC2_DB1"
resolve_net("/STM32F4xx/BOOT0", {}, {"/STM32F4xx/BOOT0": "/STM32F4xx_2/BOOT0"})  # -> "/STM32F4xx_2/BOOT0"
```

## 5. `registry.py` – реестр расстановки via

**Назначение:**  
Обеспечивает идемпотентность расстановки via между прогонами. Сохраняет в JSON-файл рядом с конфигом информацию о созданных via (UUID, позиция, цепь, параметры). При повторном запуске сверяет запланированные via с реестром, удаляет устаревшие via (prune) и создаёт только новые или изменившие позицию.

**Основные функции и классы:**

| Класс/Функция | Описание |
|---------------|----------|
| `make_registry_key(anchor_id, template_name, role, via_index)` | Генерирует составной ключ для реестра по схеме `anchor_id|template_name|role|via_index`. |
| `registry_path_for_config(config_path)` | Возвращает путь к файлу реестра для данного конфига (`<config>.registry.json`). |
| `RegistryEntry` | Dataclass, хранящий информацию о via: UUID, позиция (в мм), цепь, параметры отверстия. |
| `load_registry(path)` | Загружает реестр из JSON, при ошибке возвращает пустой словарь (graceful degradation). |
| `save_registry(path, entries)` | Сохраняет реестр в JSON с отступами для читаемости. |
| `PlacementRegistry` | Класс, управляющий реестром в рамках одного прогона. |
| `reconcile(planned_vias)` | Сравнивает запланированные via с реестром, удаляет устаревшие (изменившие позицию/параметры), удаляет via, ключи которых не встретились (prune), возвращает список via для реального создания. |
| `record_created(via_cmd, created_uuid)` | Записывает созданную via в реестр и сохраняет файл. |

**Особенности:**
- Ключи реестра строятся по схеме: `anchor_id|template_name|role|via_index`.
- `anchor_id` для ManualSpoke – `f"pad:{pad}"`, для ClonePlacement – `f"name:{clone.name}"`.
- `role` для via уровня спицы – `__spoke__` (константа из `constants.py`).
- Допуск на позицию: 0.01 мм (из `POSITION_TOLERANCE_MM`).
- При изменении позиции/параметров via удаляется старая (по UUID) и создаётся новая.
- Если ключ исчез из конфига, via удаляется (prune) – это позволяет автоматически чистить устаревшие via.
- Реестр не проверяет наличие via на плате при ручном удалении – протухший UUID обрабатывается graceful (логируется предупреждение).

**Используется в:** `kicadspoke_cli.py` (при выполнении `apply`), `executor/via_executor.py` (при создании via), и `executor/batch_executor.py` (передаётся в via executor).

## 6. `template_extraction.py` – извлечение шаблона из выделения

**Назначение:**  
Реализует команду `extract`: из текущего выделения в PCB-редакторе KiCad извлекает шаблон спицы (компоненты + via) и формирует YAML-секцию `templates` для записи в конфигурационный файл. Это позволяет создавать шаблоны визуально, без ручного ввода координат.

**Основные функции:**

| Функция | Описание |
|---------|----------|
| `extract_template_from_selection(adapter, name)` | Основная функция. Читает выделение (с учётом групп), проверяет наличие и уникальность ролей, вычисляет origin, формирует список компонентов и via, возвращает словарь для YAML. |
| `_bbox_origin(footprints, vias)` | Вычисляет левый нижний угол bounding box'а выделения (`min_x`, `max_y`). В системе координат KiCad Y растёт вниз, поэтому левый нижний – это минимальный X и максимальный Y. |

**Алгоритм:**
1. Получает выделенные объекты через `adapter.get_selected_items()`.
2. Отфильтровывает только `FootprintInstance` и `Via`; остальные объекты игнорируются с предупреждением.
3. Проверяет наличие поля `Role` у каждого компонента – если отсутствует, выбрасывает `ValidationError`.
4. Проверяет уникальность ролей – дублирование недопустимо (иначе фатальная ошибка).
5. Вычисляет origin как `(min_x, max_y)` по всем компонентам и via.
6. Для каждого компонента вычисляет `along/across` как `(позиция - origin)` в мм, сохраняет угол и роль.
7. Для каждой via аналогично вычисляет локальные координаты, сохраняет `net`, `drill_mm`, `diameter_mm`.
8. Возвращает словарь `{name: {"vias": [...], "components": [...]}}`, готовый для записи в YAML.

**Используется в:** `kicadspoke_cli.py` (команда `extract`).

## 7. `undo.py` – откат последней операции

**Назначение:**  
Реализует команду `undo`, которая восстанавливает состояние платы до выполнения последней операции расстановки. Использует JSON-логи, создаваемые `executor/operation_logger.py` при каждом успешном применении изменений.

**Основная функция:**

| Функция | Описание |
|---------|----------|
| `undo_last_operation(json_path)` | Загружает JSON-лог, для каждого перемещённого компонента: определяет исходный слой (по строке), при необходимости выполняет флип через `adapter.flip_selected()`, затем восстанавливает позицию и угол. Для каждой созданной via удаляет её по UUID. После успешного отката удаляет JSON-файл. |

**Алгоритм восстановления слоя:**
- `original_layer` хранится в логе как строка `"F.Cu"` или `"B.Cu"`.
- Если текущий слой футпринта отличается от исходного, выполняется `adapter.flip_selected([fp])`, затем футпринт перечитывается через `adapter.get_footprint(ref)` (так как локальный объект устаревает после флипа).
- Затем восстанавливаются позиция и угол.

**Используется в:** `kicadspoke_cli.py` (команда `undo`).

## 8. `validation.py` – предварительные проверки конфигурации

**Назначение:**  
Выполняет фатальные проверки конфигурации **до** любых изменений на плате. Если обнаружена проблема – программа останавливается с подробным списком ошибок, не трогая плату. Все проверки собирают ошибки разом, а не останавливаются на первой.

**Основные функции:**

| Функция | Описание |
|---------|----------|
| `check_templates_and_pads_exist(adapter, cfg)` | Проверяет, что каждая включённая спица ссылается на существующий шаблон и существующую площадку целевого компонента. Пропускает отключённые спицы (`enabled=False`). |
| `check_role_pool_sufficiency(adapter, cfg)` | Для каждого правила строит `ComponentPool` и сверяет требуемое количество компонентов каждой роли с доступным. Если не хватает – сообщает все нехватки разом. |
| `check_clone_templates_exist(cfg)` | Проверяет, что каждый `ClonePlacement` ссылается на существующий шаблон (чисто конфиговая проверка, без подключения к KiCad). |
| `check_single_selection_based_clone(cfg)` | Проверяет, что в конфиге не более одного `ClonePlacement` в режиме «по выделению» (без `nets`/`params`), так как в KiCad активно только одно выделение. Подсказывает использовать `--clone-placement` для отладки. |
| `run_all_checks(adapter, cfg)` | Запускает все проверки по порядку: сначала дешёвые (без IPC), затем более тяжёлые. |

**Особенности:**
- Сбор всех проблем: проверки собирают список ошибок, а не останавливаются на первой – это экономит время пользователя.
- Использование `ComponentPool`: в `check_role_pool_sufficiency` строится пул для каждой цепи, чтобы проверить наличие компонентов по ролям.
- Для `ClonePlacement` проверяется, что не более одного клона в режиме «по выделению» (иначе фатальная ошибка с подсказкой).
- Форматирование ошибок через `format_fatal_error()` из `exceptions.py`.

**Используется в:** `kicadspoke_cli.py` (перед планированием).

## 9. `constants.py` – глобальные константы

**Назначение:**  
Содержит глобальные константы, используемые в различных модулях проекта, что упрощает их изменение и поддержку. Все константы собраны в одном месте и документированы.

| Константа | Значение | Использование |
|-----------|----------|---------------|
| `ROLE_FIELD_NAME` | `"Role"` | Имя пользовательского поля в схеме для ролей (используется в `component_pool.py`, `template_extraction.py`, `clone_role_resolver.py`). |
| `POSITION_TOLERANCE_NM` | `10_000` (0.01 мм) | Допуск по позиции для проверки «уже на месте» (используется в `planner.py`). |
| `ANGLE_TOLERANCE_DEG` | `0.1` | Допуск по углу для проверки «уже на месте» (используется в `planner.py`). |
| `POSITION_TOLERANCE_MM` | `0.01` | Допуск по позиции в миллиметрах для реестра (используется в `registry.py`). |
| `DEFAULT_BATCH_SIZE` | `10` | Размер батча по умолчанию для транзакций (используется в `executor/batch_executor.py` и `kicadspoke_cli.py`). |
| `DEFAULT_TIMEOUT_MS` | `20000` | Таймаут IPC по умолчанию (используется в `kicad/adapter.py` и `kicadspoke_cli.py`). |
| `DEFAULT_LOG_DIR` | `"logs"` | Папка для логов по умолчанию (используется в `executor/operation_logger.py`). |
| `SPOKE_LEVEL_ROLE_PLACEHOLDER` | `"__spoke__"` | Плейсхолдер для via уровня спицы в реестре (используется в `registry.py`). |

## Взаимосвязи модулей

```mermaid
graph TD
    CLI[kicadspoke_cli.py] --> Config[config.py]
    CLI --> Adapter[kicad/adapter.py]
    CLI --> Validation[validation.py]
    CLI --> Planner[placement/planner.py]
    CLI --> Executor[placement/executor/batch_executor.py]
    CLI --> Registry[registry.py]
    CLI --> Extract[template_extraction.py]
    CLI --> Undo[undo.py]
    CLI --> Constants[constants.py]

    Config --> Exceptions[exceptions.py]
    Validation --> Config
    Validation --> ComponentPool[placement/services/component_pool.py]
    Validation --> Exceptions

    Registry --> Config
    Registry --> Adapter
    Registry --> Exceptions

    Extract --> Adapter
    Extract --> Config
    Extract --> Exceptions

    Undo --> Adapter
    Undo --> Exceptions

    NetResolution[net_resolution.py] --> Exceptions

```

Каждый модуль решает свою задачу и взаимодействует с другими через чётко определённые интерфейсы, что обеспечивает модульность и тестируемость. Благодаря централизованным константам и единому форматтеру ошибок проект легко поддерживать и расширять.
