# `tests/` – Модульные и интеграционные тесты KiCadSpoke

## Назначение

Директория `tests/` содержит полный набор тестов для обеспечения корректности, стабильности и регрессионной защиты проекта **KiCadSpoke**. Тесты разделены на две категории:

1. **Модульные тесты** (unit tests) – выполняются **без подключения к KiCad**, используют моки (`unittest.mock`) и проверяют бизнес-логику модулей: геометрию, конфигурацию, валидацию, реестр, извлечение шаблонов, разрешение цепей, обработку ролей и т.д. Эти тесты быстрые, стабильные и могут запускаться в CI/CD без внешних зависимостей.

2. **Интеграционные тесты** – выполняются **с реальным экземпляром KiCad** через IPC и проверяют сквозную работу: подключение, создание via, перемещение/флип компонентов, работу реестра расстановки, извлечение шаблонов из выделения и клонирование (`ClonePlacement`). Они требуют открытого KiCad с активной платой и отмечены маркером `@pytest.mark.integration`, чтобы не запускаться случайно.

## Фикстуры

### Модульные тесты (корневой `conftest.py`)

Модульные тесты используют встроенные фикстуры `pytest` (например, `tmp_path`) и ручные моки через `unittest.mock`. Кастомные фикстуры обычно не требуются, так как вся логика изолирована.

### Интеграционные тесты (`integration_tests/conftest.py`)

Для работы с живым KiCad определены следующие фикстуры (все они автоматически восстанавливают состояние платы после теста):

| Фикстура | Область | Описание |
|----------|---------|----------|
| `adapter` | `session` | Один экземпляр `KiCadBoardAdapter` на всю сессию. |
| `board` | `session` | Доска, полученная через `adapter.refresh_board()`. |
| `test_config` | `session` | Загруженный тестовый конфиг (например, `kicadspoke_templates_example.yaml`). |
| `test_component_ref` | `function` | Refdes компонента для тестов (по умолчанию `C5`). |
| `test_pad_number` | `function` | Номер пада для тестов (по умолчанию `17`). |
| `temp_via` | `function` | Создаёт временную via на GND, удаляет после теста. Возвращает `(via_id, position, net)`. |
| `moved_component` | `function` | Перемещает компонент на 1 мм вправо, возвращает обратно. Возвращает `(ref, original_pos, new_pos)`. |
| `flipped_component` | `function` | Переворачивает компонент на другую сторону, возвращает обратно. Возвращает `(ref, original_layer, target_layer)`. |
| `registry` | `function` | Создаёт временный реестр расстановки в `tmp_path` (для тестов реестра). |

Все мутирующие фикстуры используют `adapter.commit_with_retry()` и завершаются блоком `finally` для гарантированного восстановления, даже если тест упал.

## Запуск тестов

### Модульные тесты (без KiCad)

```bash
# Все модульные тесты
pytest tests/ -v

# Исключить интеграционные (они в отдельной папке)
pytest tests/ -v -m "not integration"

# Конкретный файл
pytest tests/test_spoke_layout.py -v
```

### Интеграционные тесты (с реальным KiCad)

**Перед запуском:**
- Убедитесь, что KiCad открыт и активна тестовая плата (рекомендуется использовать копию проекта).
- Компоненты, используемые в тестах (например, `C5`, `IC1`), должны существовать на плате.
- Тесты не изменяют критичную разводку, но всё же рекомендуется запускать их на специальном тестовом проекте.

```bash
# Все интеграционные тесты
pytest tests/integration_tests/ -v -m integration

# Конкретный файл
pytest tests/integration_tests/test_via_ops.py -v -m integration

# С отключением перехвата вывода для отладки
pytest tests/integration_tests/ -v -s -m integration
```

## Описание модульных тестов

| Файл | Что тестирует |
|------|---------------|
| `test_clone_placement_config.py` | Загрузку `ClonePlacement` из YAML: проверку полей `name`, `template`, `origin_x_mm`, `origin_y_mm`, `rotation_deg`, `nets`, `params`, `net_overrides`, `enabled`, а также привязку к якорю (`anchor_ref`, `anchor_pad`). |
| `test_clone_role_resolver.py` | Разрешение ролей для `ClonePlacement` в двух режимах: по выделению (проверка уникальности и полноты) и по цепям (с использованием `net_resolution` для плейсхолдеров и `net_overrides`). |
| `test_execute_vias_owner_ref.py` | Корректность `owner_ref` в логах и вызов `registry.record_created()` с правильным UUID для каждой созданной via. |
| `test_full_pipeline_templates.py` | Сквозной тест конвейера для `ManualSpoke` на моках: построение `ComponentPool`, применение `apply_spoke_geometry`, формирование `PlacedComponentInfo` и `ViaCommand`. |
| `test_kicad.py` | Наличие всех методов интерфейса `IBoardAdapter` в `KiCadBoardAdapter`, корректность импорта и конструктора. |
| `test_net_resolution.py` | Разрешение цепей с плейсхолдерами: подстановка из `params`, применение `net_overrides`, а также выбрасывание `ValidationError` при отсутствии параметра. |
| `test_pad_projection.py` | Предсказание позиции пада после перемещения и поворота (без флипа и с флипом) – проверка геометрической корректности. |
| `test_registry_integration.py` | Полный цикл реестра на моках: создание записи, обновление позиции (удаление старой, создание новой), prune (удаление отсутствующих ключей). |
| `test_skip_existing.py` | Идемпотентность: пропуск компонентов, уже стоящих на целевой позиции (с учётом допусков), и пропуск via, уже существующих с той же цепью и позицией. |
| `test_spoke_layout.py` | Геометрическое преобразование локальных координат шаблона в глобальные, включая via уровня спицы и компонента; проверка корректности поворота и сдвига. |
| `test_template_extraction.py` | Извлечение шаблона из выделения: проверка наличия поля `Role`, уникальности ролей, вычисление origin, формирование YAML-структуры. |
| `test_two_phase_execution.py` | Двухфазное выполнение (перемещения → перечитывание платы → via) на моках – гарантирует, что via планируются после перемещений. |
| `test_undo_layer.py` | Сохранение и восстановление слоя компонента при undo: проверка, что `original_layer` корректно записывается в лог и правильно восстанавливается. |
| `test_unique_roles.py` | Проверка, что дублирование ролей внутри шаблона вызывает фатальную ошибку (`ValidationError`) при загрузке конфига. |
| `test_validation.py` | Предварительные проверки конфигурации: существование шаблонов и падов, достаточность компонентов по ролям, корректность `clone_placements` (не более одного клона в режиме «по выделению»). |

## Описание интеграционных тестов

| Файл | Что тестирует |
|------|---------------|
| `test_connection.py` | Подключение к KiCad, получение платы, поиск компонента по refdes, поиск цепи по имени, получение списка всех via. |
| `test_via_ops.py` | Создание via (с проверкой параметров), удаление via по UUID, работа реестра (`reconcile` и `record_created`) на реальной плате. |
| `test_component_ops.py` | Перемещение компонента на 1 мм по X (и возврат), флип компонента на другую сторону и восстановление слоя. |
| `test_extract.py` | Извлечение шаблона из текущего выделения: успешный сценарий (выделены компоненты с ролями) и ошибка при пустом выделении. |
| `test_registry.py` | Полный цикл реестра с реальным KiCad: создание via, идемпотентность (повторный запуск не создаёт дубли), обновление позиции (удаление старой, создание новой), prune (удаление via, которых нет в новом плане). |

Все интеграционные тесты используют фикстуры и автоматически восстанавливают исходное состояние платы, поэтому их можно безопасно запускать на тестовом проекте.

## Особенности тестирования клонирования (`ClonePlacement`)

Хотя интеграционные тесты напрямую не тестируют весь процесс клонирования (он требует создания сложного окружения), модульные тесты покрывают:

- Разрешение ролей по выделению и по цепям (`test_clone_role_resolver.py`).
- Генерацию ключей реестра для клонов (`test_registry_integration.py`).
- Геометрическое преобразование для `ClonePlacement` (`test_clone_placement_config.py` совместно с `spoke_layout` и `clone_geometry`).
- Валидацию корректности `ClonePlacement` (`test_validation.py`).

Для полноценного сквозного теста клонирования рекомендуется использовать ручное тестирование с реальным конфигом, описанным в `pi_filter_vccio.yaml`.

## Рекомендации по добавлению тестов

При добавлении нового функционала или исправлении багов:

1. **Модульные тесты:**
   - Создайте новый файл `test_<module>.py` в корне `tests/` или дополните существующий.
   - Используйте `unittest.mock` для изоляции от адаптера и внешних зависимостей.
   - Убедитесь, что тест быстрый (доли секунды) и не требует внешних ресурсов.

2. **Интеграционные тесты:**
   - Добавьте новый тест в соответствующий файл в `integration_tests/` или создайте новый файл.
   - Отметьте тест маркером `@pytest.mark.integration`.
   - Используйте готовые фикстуры (`adapter`, `board`, `temp_via`, `moved_component` и т.д.) для упрощения работы с состоянием платы.
   - Убедитесь, что тест восстанавливает исходное состояние даже при падении (используйте `try/finally` или фикстуры с `yield`).

3. **Проверка перед коммитом:**
   - Запустите все модульные тесты: `pytest tests/ -v`.
   - По возможности запустите интеграционные тесты на тестовой плате: `pytest tests/integration_tests/ -v -m integration`.

## Примечания

- Модульные тесты не требуют KiCad и могут запускаться в любом окружении.
- Интеграционные тесты требуют KiCad с открытой платой и отмечены маркером `integration`, чтобы их можно было исключить из обычного прогона.
- Для интеграционных тестов рекомендуется использовать специальный тестовый проект (например, скопировать `10CL006YE144C8G.kicad_pcb` и соответствующий `.kicad_sch`), чтобы не повредить рабочие файлы.
- При добавлении новых зависимостей в проект убедитесь, что они не нарушают работу существующих тестов.
- Все тесты должны проходить локально перед отправкой изменений в репозиторий.
