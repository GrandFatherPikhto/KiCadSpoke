# `tests/` – Модульные и интеграционные тесты KiCadSpoke

## Назначение

Директория `tests/` содержит набор автоматизированных тестов, которые проверяют корректность работы всех ключевых модулей и сценариев использования **KiCadSpoke**. Тесты выполняются **без подключения к реальному KiCad** – вместо этого используются моки (заглушки) для адаптера и объектов платы. Это обеспечивает:

- Быстрое выполнение (секунды, а не минуты).
- Независимость от внешнего окружения (не требуется открытый KiCad).
- Возможность запуска в CI/CD.
- Раннее обнаружение регрессий при изменениях в коде.

Тесты покрывают все ключевые аспекты новой архитектуры **KiCadSpoke**:
- Шаблоны спиц и обобщённые via (`spoke_layout`).
- Автоматический подбор компонентов по ролям (`component_pool`).
- Клонируемые размещения и разрешение цепей (`clone_role_resolver`, `net_resolution`).
- Реестр расстановки (`registry`).
- Извлечение шаблонов из выделения (`template_extraction`).
- Двухфазное выполнение (`two_phase_execution`).
- Идемпотентность и откат (`skip_existing`, `undo_layer`).
- Валидацию конфигурации (`validation`).
- Адаптер KiCad (`kicad`).
- Геометрические утилиты (`pad_projection`, `spoke_layout`).

## Описание файлов

### `test_clone_placement_config.py`

**Назначение:**  
Проверяет загрузку структуры `ClonePlacement` из YAML-конфига (модуль `config.py`).

**Что тестируется:**
- Корректное чтение всех полей `ClonePlacement`: `name`, `template`, `origin_x_mm`, `origin_y_mm`, `rotation_deg`, `params`, `nets`, `net_overrides`, `enabled`.
- Значения по умолчанию для отсутствующих полей (`rotation_deg=0`, `enabled=True`).
- Отсутствие `clone_placements` в конфиге даёт пустой список.

**Зачем:**  
Убедиться, что новая структура для клонируемых размещений загружается корректно.

---

### `test_clone_role_resolver.py`

**Назначение:**  
Тестирует модуль `clone_role_resolver.py`, который сопоставляет роли с refdes для клонируемых размещений (TemplatePlacer) двумя способами: по выделению (`resolve_roles_by_selection`) и по явным цепям (`resolve_roles_by_nets`).

**Что тестируется:**
- **По выделению:**
  - Точное совпадение всех ролей с выделенными компонентами.
  - Ошибка при отсутствии компонента с нужной ролью.
  - Ошибка при наличии лишней роли в выделении.
  - Ошибка при дублировании роли в выделении.
- **По цепям:**
  - Корректное разрешение ролей для трёх одинаковых П-фильтров (без пересечения refdes).
  - Подстановка плейсхолдеров из `params` в `net_template`.
  - Приоритет `nets` над `net_template`.
  - Ошибка при отсутствии источника цепи для роли.
  - Ошибка при неоднозначном совпадении (два компонента на одной цепи).
  - Ошибка при отсутствии компонента на указанной цепи.

**Зачем:**  
Гарантирует, что клонируемые шаблоны могут быть применены как к уникальным выделенным секциям, так и к многократно повторяющимся цепям (например, каналы ЦАП, фильтры).

---

### `test_execute_vias_owner_ref.py`

**Назначение:**  
Регрессионный тест на баг: `execute_vias()` записывал в JSON-лог приблизительный `owner_ref` (всегда от первого элемента батча) для каждой созданной via.

**Что тестируется:**
- В логе каждая via получает свой собственный `owner_ref`, а не `owner_ref` первого элемента батча.
- `PlacementRegistry.record_created()` вызывается для каждой via с правильным UUID.

**Зачем:**  
Это исправление гарантирует корректность логов для undo и реестра расстановки.

---

### `test_full_pipeline_templates.py`

**Назначение:**  
Сквозной интеграционный тест всего конвейера расстановки: от `PlacementPlanner` до `ViaPlanner` с использованием `ComponentPool` и шаблонов.

**Что тестируется:**
- Автоматический подбор refdes по ролям (`LIGHT`/`HEAVY`) и цепи.
- Расчёт позиций и углов для нескольких спиц с разными поворотами.
- Планирование via уровня спицы и уровня компонента (чистая геометрия).
- Правильное распределение компонентов из пула в порядке следования спиц.

**Зачем:**  
Это ключевой тест, который проверяет взаимодействие всех основных сервисов. Он гарантирует, что архитектура KiCadSpoke работает корректно на реальном примере (без моков платы, только логика).

---

### `test_kicad.py`

**Назначение:**  
Проверяет, что адаптер `KiCadBoardAdapter` имеет все методы, объявленные в интерфейсе `IBoardAdapter`, и что импорты работают.

**Что тестируется:**
- Наличие всех методов интерфейса.
- Конструктор адаптера не падает при создании (без вызова `refresh_board`).

**Зачем:**  
«Дымовой» тест, гарантирующий, что адаптер соответствует контракту и может быть использован в основном коде.

---

### `test_net_resolution.py`

**Назначение:**  
Тестирует модуль `net_resolution.py`, который разрешает имена цепей с плейсхолдерами для клонируемых шаблонов.

**Что тестируется:**
- Литералы передаются без изменений.
- Плейсхолдеры подставляются из `params`.
- `net_overrides` применяются к итоговому имени (после подстановки).
- Отсутствие параметра для плейсхолдера вызывает ошибку.
- Лишние `params` игнорируются.

**Зачем:**  
Обеспечивает гибкую параметризацию цепей для многократно повторяющихся секций.

---

### `test_pad_projection.py`

**Назначение:**  
Проверяет функцию `predict_pad_position`, которая предсказывает положение пада после перемещения и поворота компонента (с учётом флипа).

**Что тестируется:**
- Отсутствие изменений при нулевом переносе/повороте.
- Заметное изменение позиции пада при повороте на 90°.
- Зеркалирование локального X при флипе (needs_flip=True).

**Зачем:**  
Эта логика критична для предсказания положения падов (например, для via, привязанных к падам, хотя в текущей версии via вычисляются по геометрии шаблона, модуль оставлен для будущего использования).

---

### `test_registry_integration.py`

**Назначение:**  
Интеграционный тест реестра расстановки (`registry.py`) на протяжении нескольких последовательных прогонов.

**Что тестируется:**
- Первый прогон: чистый реестр → via создаётся.
- Второй прогон: конфиг не изменился → via пропускается.
- Третий прогон: изменён `offset_across_mm` → старая via удаляется, новая создаётся.
- Четвёртый прогон: спица удалена из конфига → via удаляется (prune).

**Зачем:**  
Гарантирует, что реестр расстановки работает корректно, обеспечивая идемпотентность и автоматическое обновление via при изменении конфига.

---

### `test_skip_existing.py`

**Назначение:**  
Проверяет механизм идемпотентности: пропуск уже стоящих на месте компонентов и уже существующих via.

**Что тестируется:**
- Пропуск перемещения компонента, если он уже на целевой позиции (с допуском).
- Отсутствие пропуска при выключенном флаге `skip_existing_components`.
- Пропуск via, если она уже существует в той же позиции с той же цепью (допуск 0.01 мм).
- Различение via по цепи и по позиции.

**Зачем:**  
Позволяет безопасно перезапускать скрипт для добавления новых via без повторного перемещения уже расставленных компонентов и без создания дублирующих via.

---

### `test_spoke_layout.py`

**Назначение:**  
Проверяет корректность преобразования локальных координат шаблона спицы в абсолютные координаты платы (модуль `geometry/spoke_layout.py`), включая обобщённые via.

**Что тестируется:**
- `rotate_local_offset` – поворот локального вектора (сверка с реальной формулой поворота `kipy`).
- `apply_spoke_geometry` – применение сдвига и поворота шаблона к конкретной спице.
- Via уровня спицы и уровня компонента вычисляются от нуля спицы (а не от позиции компонента).
- Случаи: нулевой поворот, разные углы, отсутствие via, пропуск ролей без refdes.
- Поддержка произвольного количества ролей (не только 2) и нескольких via на компонент.

**Зачем:**  
Геометрия – основа расстановки. Тест гарантирует, что все компоненты и via попадут точно в расчётные позиции, а обобщённые via считаются от правильной точки отсчёта.

---

### `test_template_extraction.py`

**Назначение:**  
Проверяет модуль `template_extraction.py` и метод `adapter.get_selected_items()`, которые используются для извлечения шаблонов из выделения на плате.

**Что тестируется:**
- Извлечение шаблона с тремя ролями и via (вычисление origin, локальных координат).
- Пустое выделение → ошибка.
- Отсутствие поля `Role` → ошибка с указанием refdes.
- Дублирование роли → ошибка с указанием обоих refdes.
- Объекты, не являющиеся футпринтами/via, игнорируются (не фатально).
- `get_selected_items()` корректно разворачивает группы (`Group`) через `proto.items`.

**Зачем:**  
Гарантирует, что команда `extract` работает корректно, создавая шаблоны из текущего выделения.

---

### `test_two_phase_execution.py`

**Назначение:**  
Регрессионный тест на двухфазное выполнение: `execute_moves()` → `adapter.refresh_board()` → `plan_vias()` → `execute_vias()`.

**Что тестируется:**
- Двухфазный поток отрабатывает целиком без ошибок.
- Позиции via геометрически верны и соответствуют независимому расчёту (чистая геометрия, без чтения живого пада).

**Зачем:**  
Гарантирует, что двухфазный процесс (обязательный для корректного учёта термовиа) работает правильно. В новой архитектуре via уже не зависят от живого пада, но структура потока сохраняется для термовиа и обновления реестра.

---

### `test_undo_layer.py`

**Назначение:**  
Регрессионный тест на баг: `undo.py` не сохранял исходный слой компонента в JSON-лог, из-за чего при откате компонент возвращался на неверный слой.

**Что тестируется:**
- Корректное сохранение слоя в JSON-логе (явное преобразование в строку `"F.Cu"`/`"B.Cu"`).
- Восстановление слоя при откате (проверяется через логику `undo.py`).

**Зачем:**  
Undo – важная функция безопасности. Этот тест предотвращает потерю информации о слое, что могло бы привести к неправильному восстановлению платы.

---

### `test_unique_roles.py`

**Назначение:**  
Проверяет, что в одном шаблоне не могут быть использованы одинаковые роли (это проверяется в `config.py` при загрузке).

**Что тестируется:**
- Ошибка при дублировании роли в шаблоне.
- Корректная обработка уникальных ролей.
- Все дублирующиеся роли упоминаются в сообщении об ошибке.

**Зачем:**  
Уникальность ролей необходима для однозначного сопоставления роли и refdes из пула.

---

### `test_validation.py`

**Назначение:**  
Тестирует модуль `validation.py`, который выполняет предварительные проверки конфигурации до изменения платы.

**Что тестируется:**
- Проверка существования шаблонов и падов.
- Проверка достаточности компонентов в пуле по ролям и цепям.
- Обработка выключенных спиц (`enabled=False`).
- Раздельная проверка для разных цепей (не путаются).

**Зачем:**  
Валидация предотвращает выполнение операций с неполным или ошибочным конфигом. Тесты гарантируют, что программа выдаст понятное сообщение и остановится до любых изменений на плате.

## Запуск тестов

Все тесты используют **pytest**. Для запуска выполните из корня проекта:

```bash
pytest tests/ -v
```

Для запуска конкретного файла:

```bash
pytest tests/test_spoke_layout.py -v
```

Для просмотра вывода `print()` (отключение перехвата):

```bash
pytest tests/ -v -s
```

## Покрытие и качество

- **Покрытие критических модулей:** `geometry/spoke_layout.py`, `validation.py`, `component_pool.py`, `manual_position_calculator.py`, `registry.py`, `template_extraction.py` покрыты тестами.
- **Регрессионные тесты:** все исправленные баги (`two_phase_execution`, `undo_layer`, `owner_ref`) имеют отдельные тесты.
- **Тесты на моках:** не требуют реального KiCad, быстрые и стабильные.
- **Чёткие assert'ы:** каждый тест проверяет конкретное условие.

---

## Примечания

- Тесты используют `unittest.mock` для имитации адаптера и объектов платы.
- В тестах имитируются пользовательские поля (`Role`) через `_role` в моках.
- Допуски для координат и углов соответствуют допускам, используемым в основном коде (0.01 мм, 0.1°).
- Тесты не создают реальных via и не изменяют плату – они работают только с моками.
