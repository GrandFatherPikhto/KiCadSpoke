# Архитектура KiCadSpoke (актуальная версия)

## 1. Общее описание

**KiCadSpoke** — это инструмент командной строки для автоматической расстановки компонентов, переходных отверстий (via) и дорожек (tracks) на печатных платах в среде **KiCad 10** с использованием **шаблонов спиц**. Это продвинутая скриптовая альтернатива плагину KiCad Replicate Layout, разработанная для сложных многоканальных проектов и автоматизированного повторного использования дизайна через IPC API.

Программа подключается к открытому экземплятору KiCad через IPC (библиотека `kipy`) и выполняет:

- Перемещение компонентов на заданные позиции согласно шаблонам спиц.
- Создание via на уровне спицы и на уровне компонента.
- **Создание дорожек (tracks)** — прямых отрезков медной разводки, входящих в шаблон (клонируются вместе с компонентами и via).
- Откат последней операции (undo) с восстановлением исходного состояния платы.
- Извлечение шаблонов из выделения на плате (команда `extract`) с возможностью сохранения в JSON и параметризации цепей.
- **Клонирование шаблонов** (`ClonePlacement`) – многократное применение одного шаблона в разных местах платы с разными цепями и параметрами.
- **Разрешение неоднозначности** при подборе компонентов по физической близости к якорю.

В основе подхода лежит понятие **спицы** (spoke) — набора компонентов, via и дорожек, физически относящихся к одному выводу микросхемы или к произвольной точке на плате (для клонируемых секций). Вся геометрия спицы описывается в **локальных координатах (along, across)** в виде **шаблона**, который можно применить к любому паду микросхемы (задав сдвиг и поворот) или к произвольной точке на плате (для ClonePlacement).

## 2. Принципы архитектуры

| Принцип | Описание |
|---------|----------|
| **Разделение ответственности** | Чёткое разделение на слои: CLI, конфигурация, валидация, геометрия, адаптер к KiCad, бизнес-логика, реестр, логирование, откат. |
| **Разделение данных и логики** | Шаблоны (геометрия) — машинные данные, создаются только `extract`; размещение (логика клонирования) — человеческая логика, пишется на Python или в YAML. Шаблоны выносятся в отдельные файлы (`templates_file`). |
| **Инкапсуляция сложности IPC** | Все вызовы к KiCad сосредоточены в адаптере, предоставляющем унифицированный интерфейс `IBoardAdapter`. |
| **Расширяемость** | Новые типы via, ролей, шаблонов и дорожек легко добавляются за счёт модульной структуры. |
| **Безопасность** | Двухфазное выполнение (сначала перемещения, потом via и треки) с промежуточным перечитыванием платы; валидация перед любыми изменениями; логирование и откат. |
| **Идемпотентность** | Повторный запуск не создаёт дублирующих via и треков, не перемещает уже стоящие на месте компоненты (через реестр расстановки и сверку с реальными объектами на плате). |
| **Детерминизм** | Автоматический подбор компонентов из пула по ролям и цепям, сортировка по естественному числовому порядку, разрешение неоднозначности по близости к якорю. |
| **Физический IR** | Внутренняя модель (IR) хранит только физические факты (координаты, геометрия падов); семантика (банки, роли) хранится в тегах/метаданных, не встроена в структуру. |

## 4. Слои архитектуры (снизу вверх)

### 4.1. Уровень доступа к KiCad (`kicad/`)

- **`adapter.py`** – реализует `KiCadBoardAdapter`, инкапсулирующий все вызовы к `kipy`. Предоставляет методы для поиска компонентов, падов, зон, цепей, получения bounding box'ов, транзакций (commit/rollback), создания via **и треков**, выполнения флипа, чтения пользовательских полей, получения выделения с учётом групп и удаления объектов по UUID.
- **`interfaces.py`** – определяет абстрактный интерфейс `IBoardAdapter`, позволяющий легко подменять адаптер (например, для тестирования с моками).

### 4.2. Геометрические утилиты (`geometry/`)

Модули, работающие с координатами, не зависящие от KiCad:

- **`keepout.py`** – класс `Rect` (AABB-прямоугольник) и функции для построения keepout-областей, проверки точек на занятость, поиска свободного места.
- **`pad_projection.py`** – предсказание позиции пада после перемещения и поворота компонента с учётом флипа.
- **`spoke_layout.py`** – преобразование локальных координат шаблона (`along/across`) в абсолютные координаты платы для `ManualSpoke` (привязка к паду IC, сдвиг и поворот). Генерирует via и **треки** (уровня спицы и компонента) с использованием `rule.net` как цепи по умолчанию.
- **`clone_geometry.py`** – аналогичное преобразование для `ClonePlacement`, но без привязки к паду IC: используется абсолютная точка `(origin_x, origin_y)`, и цепи via/треков резолвятся через `net_resolution` (с поддержкой плейсхолдеров). Поддерживает зеркалирование (`mirror`).
- **`thermal_grid.py`** – генерация сетки термовиа под площадкой.

### 4.3. Бизнес-логика (`placement/`)

- **`planner.py`** – класс `PlacementPlanner`, главный координатор. Вызывает `ManualPositionCalculator` для расчёта позиций компонентов и via на основе `rules`, и `ClonePositionCalculator` для `clone_placements` (который также возвращает треки). Затем формирует команды для перемещения, создания via и треков. Содержит логику пропуска уже стоящих на месте компонентов.
- **`executor/`** – разбит на модули для улучшения читаемости и тестируемости:
  - `FlipManager` – управление флипом.
  - `MoveExecutor` – исполнение перемещений.
  - `ViaExecutor` – исполнение создания via.
  - `TrackExecutor` – исполнение создания треков.
  - `OperationLogger` – запись JSON-логов (включая треки).
  - `BatchExecutor` – фасад, объединяющий все фазы и управляющий логированием.
- **`interfaces.py`** – определяет абстрактные интерфейсы `IPositionCalculator` (для расчёта позиций) и `IViaPlanner` (для планирования via), что позволяет легко подменять реализации.
- **`services/`**:
  - `ComponentPool` – сбор компонентов по ролям (`Role`) и цепи, сортировка, раздача спицам (для `ManualSpoke`).
  - `CloneRoleResolver` – разрешение ролей для `ClonePlacement` двумя режимами: «по выделению» (считывание ролей из выделенных компонентов) и «по цепям» (явное задание цепей с поддержкой плейсхолдеров через `net_resolution`). В режиме по цепям реализовано **разрешение неоднозначности** по физической близости к якорю (если кандидатов несколько, выбирается ближайший, если отрыв достаточный).
  - `ManualPositionCalculator` – реализует `IPositionCalculator` для `rules`: строит пул, применяет `apply_spoke_geometry`, возвращает `PlacedComponentInfo` и `ViaCommand`.
  - `ClonePositionCalculator` – реализует расчёт для `clone_placements`, используя `apply_clone_geometry` и `clone_role_resolver`, возвращает компоненты, via и треки.
  - `ViaPlanner` – реализует `IViaPlanner`: отвечает только за термовиа и фильтрацию существующих via через реестр (skip_existing_components).

### 4.4. Конфигурация и валидация

- **`config.py`** – определяет датаклассы для всех структур конфига (`Config`, `SpokeTemplate`, `ManualSpoke`, `ClonePlacement`, `TemplateVia`, `TemplateTrack`, `TemplateComponentSlot` и т.д.) и функцию `load_config()` для загрузки YAML. Поддерживает **`templates_file`** — внешний файл шаблонов (JSON или YAML), который загружается и объединяется с инлайновыми `templates`. Это позволяет выносить тяжёлую геометрию из основного конфига.
- **`validation.py`** – предварительные проверки: существование шаблонов и падов, достаточность компонентов в пуле по ролям, корректность `clone_placements` (не более одного клона в режиме «по выделению» за прогон), **а также резолвинг цепей via/треков и проверка их наличия на плате**. Бросает `ValidationError` с понятным списком проблем.

### 4.5. Реестр расстановки (`registry.py`)

- Обеспечивает идемпотентность via и треков между прогонами.
- Сохраняет в JSON-файлы информацию о созданных via (UUID, позиция, цепь, параметры) и треках (UUID, координаты, ширина, слой) с составными ключами `anchor_id|template_name|role|via_index` (и аналогично для треков).
- Для `ManualSpoke` anchor_id = `pad:{pad}`, для `ClonePlacement` anchor_id = `name:{clone.name}` (или `anchor:{ref}:{pad}` / `role:{role}:{sheet}:{pad}` для физических якорей).
- При повторном запуске **сверяет запланированные via и треки с реальными объектами на плате** (`adapter.get_vias()` / `adapter.get_tracks()`), а не только с JSON-записью. Это предотвращает рассинхронизацию при ручном удалении или сбоях.
- Удаляет устаревшие via/треки (prune), создаёт только новые или изменившие позицию/параметры.

### 4.6. Разрешение цепей для клонирования (`net_resolution.py`)

- Реализует трёхслойное разрешение имени цепи для `ClonePlacement`: литерал → подстановка плейсхолдеров из `params` → применение `net_overrides`.
- Также содержит **обратную параметризацию** (`parametrize_net`) для `extract`: по реальному имени цепи и карте `net_template_map` восстанавливает паттерн с плейсхолдером, проверяя round-trip.

### 4.7. Извлечение шаблонов (`template_extraction.py`)

- Реализует команду `extract`: из текущего выделения в PCB-редакторе извлекает шаблон спицы (компоненты, via и **треки**) и формирует структуру для записи в файл.
- Поддерживает **сохранение в JSON** (при расширении `.json`) — файл записывается как плоский словарь `{имя_шаблона: {...}}`, без обёртки `templates:`, что удобно для использования в `templates_file`.
- Поддерживает **параметризацию цепей** через `--net-template` и `--param`: заменяет литеральные имена цепей на паттерны с плейсхолдерами.
- Поддерживает **выбор origin** не по bbox, а по конкретному via или компоненту (`--origin-by-via-net`, `--origin-by-component-role`).
- Автоматически определяет слой шаблона и проставляет явные слои для элементов, лежащих на других слоях.

### 4.8. Откат операций (`undo.py`)

- Функция `undo_last_operation()` читает последний JSON-лог и восстанавливает плату (возвращает компоненты на исходные позиции и слой, удаляет созданные via и треки).

### 4.9. Пользовательский интерфейс (CLI)

- **`kicadspoke_cli.py`** – главный исполняемый файл, обрабатывающий аргументы командной строки, вызывающий загрузку конфига, подключение к KiCad, валидацию, планирование и выполнение (трёхфазное: перемещения → via → треки), а также команды `undo`, `extract`, `clone-extract` и опциональный флаг `--clone-placement` для обработки только одного клона.

## 5. Взаимодействие модулей (последовательность работы)

```
1. Запуск CLI (kicadspoke_cli.py)
   │
   ├── Разбор аргументов (argparse)
   ├── Настройка логирования
   │
2. Загрузка конфига (load_config)
   │
   ├── Чтение YAML
   ├── Загрузка внешнего файла шаблонов (templates_file), если указан
   ├── Слияние с инлайновыми templates
   ├── Проверка уникальности ролей в шаблонах
   ├── Создание объекта Config
   │
3. Подключение к KiCad
   │
   ├── Создание KiCadBoardAdapter
   ├── adapter.refresh_board() — получение платы
   │
4. Валидация (run_all_checks)
   │
   ├── check_clone_templates_exist()
   ├── check_single_selection_based_clone()
   ├── check_templates_and_pads_exist()
   ├── check_role_pool_sufficiency()
   ├── check_clone_nets_exist_on_board()  # проверка цепей via/треков на плате
   │
5. Планирование (PlacementPlanner)
   │
   ├── plan_moves()
   │   ├── Для каждого правила (rules):
   │   │   ├── Строит ComponentPool
   │   │   ├── Для каждой спицы:
   │   │   │   ├── pop() — берёт refdes для каждой роли
   │   │   │   ├── apply_spoke_geometry() — вычисляет позиции (и via)
   │   │   │   └── генерирует ViaCommand и PlacedComponentInfo
   │   │
   │   ├── Для каждого clone_placements:
   │   │   ├── Определяет режим (по выделению или по цепям)
   │   │   ├── Вызывает clone_role_resolver для получения role_to_ref (с учётом близости к якорю)
   │   │   ├── apply_clone_geometry() — вычисляет позиции, via и треки
   │   │   └── генерирует PlacedComponentInfo, ViaCommand, TrackCommand
   │   │
   │   └── Возвращает списки MoveCommand, ViaCommand, TrackCommand
   │
   ├── (Если --dry-run — вывод плана и выход)
   │
6. Исполнение (трёхфазное)
   │
   ├── Фаза 1: перемещения
   │   ├── executor.execute_moves()
   │   │   ├── Сбор исходных состояний (для undo)
   │   │   ├── Проверка коллизий (опционально)
   │   │   ├── Флип (если слой изменился)
   │   │   ├── Перемещение батчами через commit_with_retry
   │   │   └── Сохранение лога перемещений
   │   │
   │   ├── adapter.refresh_board() — перечитываем плату
   │   │
   ├── Фаза 2: via
   │   ├── planner.plan_vias() — применяет фильтрацию через реестр (и термовиа)
   │   ├── executor.execute_vias()
   │   │   ├── Создание via батчами через commit_with_retry
   │   │   ├── Сбор UUID созданных via (для undo и реестра)
   │   │   └── Регистрация в реестре via (record_created)
   │   │
   ├── Фаза 3: треки
   │   ├── planner.plan_tracks() — просто возвращает запланированные треки (без keepout)
   │   ├── executor.execute_tracks()
   │   │   ├── Создание треков батчами через commit_with_retry
   │   │   ├── Сбор UUID созданных треков (для undo и реестра)
   │   │   └── Регистрация в реестре треков
   │   │
   └── Запись единого JSON-лога операции (включая перемещения, via, треки)
       └── Вывод результата (успех/ошибки)
```

## 6. Ключевые архитектурные решения

| Решение | Обоснование |
|---------|-------------|
| **Отказ от зависимости via/треков от живой платы** | Все via и треки (кроме термовиа) вычисляются исключительно по геометрии шаблона. Это делает их независимыми от текущего состояния платы и позволяет планировать их до перемещений. |
| **Трёхфазное выполнение** | Необходимо для корректного учёта термовиа и для идемпотентности (via и треки создаются после перемещений, чтобы проверить их наличие в финальных позициях). |
| **Идемпотентность через реестр с живой сверкой** | Реестр хранит UUID созданных объектов, но при повторном запуске сверка идёт с реальными via/треками на плате, а не только по JSON. Это предотвращает рассинхронизацию при ручном вмешательстве или сбоях. |
| **Автоматический подбор refdes по ролям** | Устраняет необходимость вручную прописывать имена компонентов в конфиге, делая его компактным и устойчивым к изменениям нумерации. |
| **Разрешение неоднозначности по близости к якорю** | В режиме «по цепям» при нескольких кандидатах на одну роль программа выбирает ближайший к якорю (если отрыв достаточный). Это решает проблему фильтров питания на общей шине. |
| **Клонирование шаблонов с треками** | Позволяет копировать не только компоненты и via, но и дорожки, что сохраняет связность и сокращает ручную трассировку. |
| **Вынос шаблонов в отдельные файлы (`templates_file`)** | Тяжёлая геометрия (особенно треки) уходит из основного конфига, делая его читаемым. Шаблоны генерируются `extract` и никогда не правятся руками. |
| **JSON-формат для шаблонов** | Упрощает сериализацию и исключает проблемы с кавычками и отступами YAML. |
| **Логирование и undo** | Каждая операция сохраняется в JSON, что позволяет откатить изменения (включая удаление треков). |
| **Модульные и интеграционные тесты** | Модульные тесты на моках защищают от регрессий; интеграционные тесты с реальным KiCad проверяют сквозную работу. |

## 7. Используемые шаблоны проектирования

| Шаблон | Применение |
|--------|------------|
| **Адаптер** | `KiCadBoardAdapter` предоставляет унифицированный интерфейс для `kipy`. |
| **Команда** | `MoveCommand`, `ViaCommand`, `TrackCommand`, `PlacedComponentInfo` инкапсулируют действия. |
| **Строитель** | `apply_spoke_geometry` и `apply_clone_geometry` собирают `SpokeLayout` из частей (via, компоненты, треки). |
| **Хранитель (Memento)** | JSON-логи хранят состояние для undo. |
| **Шаблонный метод** | `BatchExecutor` использует единый шаблон для перемещений, via и треков (батчи, транзакции). |
| **Внедрение зависимости** | Все сервисы получают `adapter` и `config` через конструктор. |
| **Data Transfer Object (DTO)** | Dataclass'ы из `config.py` и `commands.py` — чистые DTO. |
| **Стратегия** | Два режима сопоставления ролей в `clone_role_resolver` (по выделению или по цепям). |
| **Фабрика** | `load_config` создаёт объекты конфигурации; `create_via`, `create_track` в адаптере. |

## 8. Использование нестабильных и недокументированных API

| Элемент | Статус | Причина использования |
|---------|--------|------------------------|
| `kicad.run_action("pcbnew.InteractiveEdit.flip")` | **Нестабильный** | Единственный способ выполнить «настоящий» переворот компонента с зеркалированием площадок и шелкографии. |
| `Group.proto.items` | **Недокументированный** | Для получения участников группы (свойство `.items` всегда пустое). |
| `footprint.texts_and_fields` | **Недокументированный** | Используется для чтения пользовательских полей (`Field`). |
| `footprint.definition.items` | **Недокументированный** | Используется для получения падов компонента. |
| `board.remove_items_by_id()` | **Новый, потенциально нестабильный** | Добавлен недавно (июль 2025), используется в реестре для удаления via и треков. |
| `board.get_tracks()` | **Недокументированный** | Используется для получения всех треков на плате для сверки реестра. |

Все эти места **документированы** в коде и проверяются модульными тестами.

## 10. Эволюция архитектуры (ход дискуссии)

В ходе архитектурного обсуждения (май–июль 2026) были приняты следующие решения, зафиксированные в текущей архитектуре:

1. **Разделение шаблонов и размещения** — шаблоны (геометрия) выносятся в отдельные файлы, размещение (логика) пишется на Python или в YAML с ссылкой на шаблоны.
2. **Отказ от lock-файла** — заменён на команду `diff` (план) и явное разделение `plan`/`apply` (в будущем).
3. **Физический IR** — внутренняя модель хранит только координаты и геометрию, семантика (роли, банки) — в тегах/метаданных.
4. **Поддержка треков** — добавлена как часть шаблонов для сохранения связности при клонировании (v1.22.0).
5. **Разрешение неоднозначности по близости к якорю** — реализовано в `clone_role_resolver`.
6. **JSON для шаблонов** — `extract` может сохранять шаблоны в JSON, что упрощает сериализацию и интеграцию с `templates_file`.
7. **Живая сверка реестра** — `PlacementRegistry` и `TrackRegistry` сверяются с реальными объектами на плате, а не только по JSON.
8. **Явные поля `by_selection`, `anchor_role`, `refs`** — для улучшения управления клонированием.

Эти решения обеспечивают гибкость, надёжность и удобство использования для сложных проектов.

## 11. Заключение

Архитектура KiCadSpoke обеспечивает:

- **Гибкость** – легко добавлять новые шаблоны, роли, типы via и треки, а также клонируемые секции.
- **Надёжность** – трёхфазное выполнение, валидация, идемпотентность с живой сверкой, реестр и undo.
- **Удобство использования** – конфигурация без явных refdes, автоматический подбор компонентов, извлечение шаблонов из выделения, клонирование повторяющихся секций с дорожками.
- **Тестируемость** – чёткое разделение слоёв, интерфейсы, модульные и интеграционные тесты.
- **Масштабируемость** – вынос шаблонов в отдельные файлы, поддержка JSON.

Это делает инструмент пригодным для использования в реальных проектах по разработке печатных плат, особенно в многоканальных системах с высокой степенью повторяемости.
