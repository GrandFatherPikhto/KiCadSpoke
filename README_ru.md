# KiCadSpoke v1.25.0

**KiCadSpoke** — это инструмент командной строки для автоматизации разводки и **клонирования блоков** печатных плат в **KiCad 10**. Он позволяет **реплицировать** (клонировать) повторяющиеся участки схемы и трассировки, автоматически размещать компоненты и переходные отверстия на основе **шаблонов** и **ролей**. Подключается к открытому экземпляру KiCad через IPC и выполняет:

- Перемещение компонентов (конденсаторов, резисторов, ферритов, кварцев и т.д.) на заданные позиции.
- Создание via и **дорожек (треков)** с привязкой как к спице в целом, так и к отдельным компонентам.
- **Клонирование** повторяющихся функциональных блоков (П-фильтров, каналов ЦАП) в разных местах платы.
- Автоматический подбор компонентов по **ролям** и **цепям** без указания конкретных refdes.
- Идемпотентность: повторные запуски не дублируют уже правильно стоящие элементы.
- Откат последней операции (undo).
- Извлечение шаблонов из текущего выделения на плате с возможностью параметризации цепей и выбора точки привязки.
- Снятие снимков каналов (для иерархических проектов) через файловый клонер (`clone-extract`).

**Продвинутая альтернатива традиционному плагину KiCad Replicate Layout**, созданная для сложных многоканальных проектов и автоматизированного повторного использования дизайна через IPC API.

## Основные возможности

- **Шаблонный подход** – геометрия описывается один раз в локальных координатах и может быть повёрнута/сдвинута для каждого применения.
- **Автоматический подбор компонентов** – вместо refdes используются роли (`LIGHT`, `HEAVY`, `PI_FILTER_C1` и т.п.). Конкретные экземпляры берутся из пула по цепи и полю `Role` в схеме.
- **Клонирование секций** (`ClonePlacement`) – поддерживает два режима:
  - **по выделению** – для штучных экземпляров (одна MCU);
  - **по цепям** – для многократно повторяющихся блоков (с параметризацией имён цепей через плейсхолдеры и `params`). В этом режиме неоднозначность разрешается по физической близости к якорю (если компоненты расположены рядом с разными якорями и электрически неразличимы). Также можно использовать явные `refs` для крайних случаев.
- **Обобщённые via и дорожки** – все элементы шаблона (via, дорожки, компоненты) задаются в локальной системе координат и преобразуются единым образом (сдвиг, поворот, зеркалирование).
- **Реестр расстановки** – сохраняет UUID созданных via и треков, обеспечивая идемпотентность и автоматическую чистку устаревших. Теперь сверка выполняется с реальными элементами на плате, а не только по JSON-записи, что решает проблемы рассинхронизации.
- **Предварительная валидация** – проверка конфигурации до любых изменений на плате:
  - существование шаблонов, падов и якорных компонентов;
  - достаточность компонентов в пулах по ролям;
  - уникальность якорей и имён `clone_placements`;
  - корректность резолва цепей via (проверка против реальных цепей платы);
  - допустимость сочетания `layer`/`mirror` (зеркалирование только при смене слоя);
  - не более одного `clone_placement` в режиме "по выделению" за один прогон (в KiCad активно только одно выделение).
- **Диагностика** – набор скриптов для отладки IPC, геометрии и чтения полей.
- **Файловый клонер** (`clone-extract`) – анализирует `.net` и `.kicad_pcb` без IPC, строит карту близнецов каналов для иерархических проектов.
- **Дорожки (треки) в шаблонах** – с версии 1.22.0 шаблоны могут содержать прямые отрезки дорожек (ломаные поддерживаются через последовательность сегментов). Коллизии дорожек не проверяются автоматически (полагаемся на DRC KiCad).

## Почему это не просто конфиг, а инструмент клонирования PCB

KiCadSpoke изначально создавался для решения конкретной проблемы: как быстро и без ошибок размножить десятки одинаковых блоков на сложных платах (FPGA, многоканальные ЦАП, массивы питания). В отличие от ручного копирования (Ctrl+C/Ctrl+V) в KiCad, наш подход:

- **Автоматически подбирает компоненты** по ролям, а не по refdes – вам не нужно править сотни ссылок при реаннотации.
- **Сохраняет связность** – via и дорожки клонируются вместе с компонентами, что исключает ручную перерисовку.
- **Параметризует цепи** – один шаблон можно применить к разным напряжениям, просто задав параметры (`PWR_IN`, `PWR_OUT`).
- **Идемпотентен** – повторный запуск не создаёт дубликатов, а только обновляет изменившиеся позиции.
- **Интегрируется с Git** – благодаря разделению шаблонов (JSON) и размещения (YAML/генераторы) изменения легко отслеживать в системе контроля версий.
- **Поддерживает дорожки** – теперь можно клонировать не только компоненты и via, но и готовую трассировку внутри блока.

Это делает KiCadSpoke идеальным выбором для проектов с высокой степенью повторяемости: от плат питания до многоканальных измерительных систем.

## Установка и зависимости

### Требования
- Python 3.8 или новее.
- KiCad 10.0.4 или новее (с включённым IPC API).
- Установленная библиотека **kipy** (обёртка над KiCad IPC).

### Установка
```bash
pip install kipy pyyaml sexpdata
```
(Для диагностики может потребоваться `psutil`.)

### Настройка ролей в схеме (Eeschema)
Перед использованием необходимо добавить пользовательское поле `Role` для компонентов, которые должны участвовать в расстановке:
1. Откройте символ в Eeschema.
2. Добавьте поле с именем **Role** и значением, соответствующим роли в шаблоне (например, `LIGHT`, `HEAVY`).
3. Выполните **Update PCB from Schematic**, чтобы поле перенеслось на плату.
4. Проверьте читаемость поля диагностическим скриптом:
   ```bash
   python -m kicadspoke.diagnostics.test_custom_field C5 --field Role
   ```

## Ключевые концепции

### Шаблон (SpokeTemplate)
Шаблон описывает **локальную геометрию** одного «борта» – набора компонентов, via и дорожек относительно локального нуля (0,0) в системе координат `along/across`. Он содержит:
- **`vias`** – via на уровне спицы (обычно цепь питания).
- **`components`** – список слотов, каждый с `role`, локальными координатами, углом и списком via (обычно на GND).
- **`tracks`** – прямые отрезки дорожек (слой, ширина, цепь).

Все координаты задаются **один раз** при `rotation_deg=0`; при применении шаблон поворачивается целиком.

#### Слой шаблона (`layer`)
Теперь каждый шаблон имеет абсолютный слой (`F.Cu` или `B.Cu`), который автоматически проставляется при извлечении (`extract`). Компоненты, лежащие на другом слое, получают явный `layer` в своём слоте.

#### Параметризация цепей при извлечении
При извлечении шаблона можно сразу заменить литеральные имена цепей на паттерны с плейсхолдерами, используя опции `--net-template` и `--param` команды `extract`. Это избавляет от ручной правки YAML.

### Спица (ManualSpoke)
Привязка шаблона к конкретному выводу микросхемы:
- `pad` – номер пада целевого компонента.
- `shift_x_mm`, `shift_y_mm` – плоский сдвиг от центра пада к нулю шаблона.
- `rotation_deg` – поворот всего шаблона.

**Важно:** в новых версиях конфига у каждого правила (`rules`) должен быть указан свой `anchor_ref` (якорный компонент). Глобальный `target_ref` упразднён.

### Роли и пул компонентов
Вместо refdes в конфиге используются **роли**. Для каждой цепи (`rule.net`) строится пул компонентов, у которых:
- Есть поле `Role` с нужным значением.
- Хотя бы один пад подключён к этой цепи.

Компоненты сортируются в естественном численном порядке (`C5` < `C10`) и разбираются по порядку следования спиц.

### Клонирование (ClonePlacement) – новые поля
Позволяет применить шаблон в произвольной точке платы, не привязываясь к падам IC. Поддерживает:

- **Режим по выделению** – программа считывает роли из текущего выделения в PCB-редакторе. Для этого достаточно не задавать `nets` и `params` (или явно указать `by_selection: true`). В одном прогоне может быть обработан только один такой `clone_placement` (из-за ограничения KiCad на одно выделение).
- **Режим по цепям** – для каждой роли задаётся цепь (через `nets` или `net_template` с плейсхолдерами, разрешаемыми через `params` и `net_overrides`). Если несколько кандидатов электрически неразличимы, программа может выбрать ближайший к якорю (при достаточном отрыве по расстоянию). Это полезно для фильтров питания на общей шине.
- **Якорь по роли** (`anchor_role`) – альтернатива `anchor_ref`: вместо refdes можно указать поле `Role` якорного компонента. Это устойчиво к перенумерации. Дополнительно можно сузить поиск с помощью `anchor_sheet` (префикс локальной цепи) или `anchor_pad`.
- **Явные refs** (`refs`) – крайняя мера, когда кандидаты неразличимы ни по цепям, ни по выделению, ни по близости к якорю. Указывается сопоставление роль → refdes.

### Реестр расстановки (PlacementRegistry + TrackRegistry)
Для via и треков ведутся отдельные реестры (JSON-файлы рядом с конфигом). При повторном запуске:
- уже стоящие правильно – пропускаются;
- изменившие позицию/параметры – удаляются и пересоздаются;
- устаревшие (ключ отсутствует в новом плане) – удаляются (prune).

**Важно:** сверка выполняется с реальными элементами на плате (`adapter.get_vias()`/`adapter.get_tracks()`), а не только по JSON, что предотвращает рассинхронизацию при ручном удалении или сбоях между записью в реестр и коммитом на плату.

### Разрешение цепей (net_resolution)
Для клонируемых шаблонов имена цепей проходят трёхступенчатое разрешение:
1. **Литерал** – если нет плейсхолдеров.
2. **Плейсхолдер** – подстановка из `params` (например, `{channel}` → `2`).
3. **net_overrides** – финальная подмена итогового имени (для иерархических путей).

При извлечении шаблона доступна обратная операция (`--net-template`), которая превращает литералы в паттерны.

## Формат конфигурационного файла (YAML)

### Корневые параметры (новое)

| Поле | Тип | Описание |
|------|-----|----------|
| `layer` | строка | Глобальный слой для ManualSpoke-правил: `"F.Cu"` или `"B.Cu"` (вместо устаревшего `side`). |
| `templates` | словарь | Именованные шаблоны спиц (можно заменить `templates_file`). |
| `templates_file` | строка | Путь к внешнему файлу шаблонов (JSON или YAML). Инлайновые `templates` дополняют/переопределяют его. |
| `rules` | список | Правила для ручных спиц (ManualSpoke), каждое с `anchor_ref`. |
| `clone_placements` | список | Клонируемые размещения (TemplatePlacer). |
| `thermal_via_array` | словарь | Настройки термовиа (опционально). |
| `place_components` | булево | Включить перемещение компонентов (по умолчанию `true`). |
| `skip_existing_components` | булево | Пропускать уже стоящие на месте компоненты и via/треки. |
| `via_keepout_clearance_mm`, `via_search_step_mm`, `via_search_max_radius_mm`, `via_search_n_directions` | числа | Параметры поиска места для термовиа. |

**Устаревшие поля:** `target_ref` и `side` в корне больше не поддерживаются (фатальная ошибка при загрузке).

### Шаблон (`templates`) – теперь с `layer` и `tracks`

```yaml
templates:
  cap_pair_standard:
    layer: B.Cu       # абсолютный слой шаблона
    vias:
      - offset_along_mm: 0.0
        offset_across_mm: -1.5
        drill_mm: 0.3
        diameter_mm: 0.6
        # net не указан – для ManualSpoke берётся rule.net, для ClonePlacement должен быть задан явно
    tracks:
      - start_along_mm: 0.0
        start_across_mm: 0.0
        end_along_mm: 1.0
        end_across_mm: 1.0
        width_mm: 0.25
        net: GND
        layer: F.Cu    # опционально, наследуется от шаблона
    components:
      - role: LIGHT
        offset_along_mm: 1.0
        offset_across_mm: -1.0
        angle_deg: 90.0
        vias:
          - offset_along_mm: 0.0
            offset_across_mm: -1.0
            net: GND
            drill_mm: 0.3
            diameter_mm: 0.6
      - role: HEAVY
        offset_along_mm: 1.0
        offset_across_mm: 2.0
        angle_deg: 270.0
        vias:
          - offset_along_mm: 0.0
            offset_across_mm: 1.3
            net: GND
            drill_mm: 0.3
            diameter_mm: 0.6
```

### Ручные спицы (`rules`) – теперь с `anchor_ref`

```yaml
rules:
- net: +1V2_VCCINT
  anchor_ref: IC1       # обязательное поле – якорь для этого правила
  spokes:
  - pad: '109'
    template: cap_pair_standard
    shift_x_mm: 0.0
    shift_y_mm: 0.0
    rotation_deg: 90.0
  - pad: '62'
    template: cap_pair_standard
    shift_x_mm: 0.4
    shift_y_mm: 0.0
    rotation_deg: 270.0
```

### Клонируемые размещения (`clone_placements`) – все новые поля

```yaml
clone_placements:
  - name: dac_channel_2
    template: dac_channel
    anchor_ref: IC1
    anchor_pad: '17'      # опционально
    origin_x_mm: -10.0    # сдвиг от якоря, если anchor_ref задан, иначе абсолютная точка
    origin_y_mm: -10.0
    rotation_deg: 90.0
    params:
      channel: 2
    nets:
      CAP_IN: "GPIO12"
      CAP_OUT: "GPIO12_FILTERED"
    net_overrides:
      "/STM32F4xx/BOOT0": "/STM32F4xx_2/BOOT0"
    layer: B.Cu          # явный слой размещения (если отличается от слоя шаблона)
    mirror: true         # зеркальное отражение всей конструкции (требует смены слоя)
    by_selection: false  # явный запрос режима "по выделению" (приоритетнее, чем отсутствие nets)
    refs:                # явное сопоставление роль->ref (крайняя мера)
      DAC_PI_FILTER_C1: C601
    enabled: true
```

- Если задан `anchor_ref` (или `anchor_role`), `origin_x/y` становятся **сдвигом** от якоря.
- Поля `layer` и `mirror` позволяют размещать шаблон на другой стороне платы с зеркалированием.
- `by_selection: true` заставляет интерпретировать размещение как режим выделения, даже если заданы `params` (которые могут использоваться для резолва via, но не для ролей).

### Термовиа (`thermal_via_array`) – переименован `target_ref` в `anchor_ref`

```yaml
thermal_via_array:
  enabled: true
  anchor_ref: IC1       # было target_ref
  pad: '145'
  net: GND
  rows: 4
  cols: 4
  margin_mm: 0.5
  pattern: grid
  drill_mm: 0.3
  diameter_mm: 0.5
```

## Команды CLI

Все команды выполняются через `kicadspoke_cli.py`. Если подкоманда не указана, подразумевается `apply`.

### `apply` – применить расстановку

```bash
python kicadspoke_cli.py apply config.yaml [options]
```

Опции:
- `--dry-run` – только показать план, ничего не изменять.
- `--timeout-ms` – таймаут IPC (по умолчанию 20000 мс).
- `--batch-size` – размер батча для коммитов (по умолчанию 10).
- `--verbose` – подробный вывод (DEBUG).
- `--log-file` – сохранять логи в файл.
- `--no-collision-check` – отключить проверку коллизий.
- `--collision-margin` – зазор при проверке коллизий (мм, по умолчанию 0.2).
- `--clone-placement NAME` – обработать только указанный клон (полезно для отладки).

### `extract` – извлечь шаблон из выделения (расширенный)

```bash
python kicadspoke_cli.py extract --name имя_шаблона --output config.yaml [--verbose] [--log-file] [--param KEY=VALUE] [--net-template ЛИТЕРАЛ=ПАТТЕРН] [--origin-by-via-net NET] [--origin-by-component-role ROLE]
```

Новые опции:
- `--param KEY=VALUE` – задаёт параметр для проверки `--net-template` (например, `channel=1`). В шаблон не пишется, нужен только для верификации.
- `--net-template ЛИТЕРАЛ=ПАТТЕРН` – заменяет реальную цепь на паттерн с плейсхолдером (например, `DAC1_DB1=DAC{channel}_DB1`). Можно указывать несколько раз.
- `--origin-by-via-net NET` – задаёт origin шаблона по позиции via с указанной цепью (вместо левого нижнего угла bbox). Фатально, если такой via нет или она не единственна.
- `--origin-by-component-role ROLE` – задаёт origin по позиции компонента с указанной ролью.

**Важно:** расширение `--output` определяет формат: `.json` → JSON, иначе YAML. При этом файл записывается **без обёртки `templates:`** (плоский словарь), что упрощает использование как `templates_file`.

Перед запуском выделите в PCB-редакторе нужные компоненты и via. Роли должны быть уникальны.

### `undo` – откатить последнюю операцию

```bash
python kicadspoke_cli.py undo [--verbose] [--log-file]
```

### `clone-extract` – снятие снимка канала (файловый клонер)

```bash
python kicadspoke_cli.py clone-extract --net project.net --pcb project.kicad_pcb --channel Channel_0 --output snapshot.yaml [--verbose]
```

Анализирует иерархический проект, извлекает все компоненты, дорожки и via, относящиеся к указанному каналу, и сохраняет снимок в YAML. Полезно для изучения структуры перед написанием конфига клонирования.

## Примеры использования

### 1. Стандартный запуск
```bash
python kicadspoke_cli.py 10CL006YE144C8G.yaml
```

### 2. Просмотр плана без изменений
```bash
python kicadspoke_cli.py config.yaml --dry-run
```

### 3. Обработка одного клона при нескольких в режиме выделения
```bash
python kicadspoke_cli.py config.yaml --clone-placement pi_filter_vccio
```

### 4. Извлечение шаблона с параметризацией и origin по via
Выделите элементы на плате, затем:
```bash
python kicadspoke_cli.py extract --name my_filter --output my_config.json --net-template "DAC1_DB1=DAC{channel}_DB1" --param channel=1 --origin-by-via-net "/Channel_0/DAC/+3V3_CLKVDD" --verbose
```

### 5. Откат
```bash
python kicadspoke_cli.py undo --verbose
```

## Диагностика и известные проблемы

### Баг KiCad #24966 (крах на первой записи IPC)
При открытом редакторе схем и отсутствии интерактивных правок в текущей сессии KiCad вызов `Board.update_items()` (даже без изменений) может привести к падению KiCad (null pointer в `API_HANDLER_EDITOR::checkForBusy`). Это связано с тем, что запрос направляется в обработчик схемы, который не инициализирован.

**Симптомы:** KiCad молча закрывается, клиент получает `ConnectionError: Error receiving reply from KiCad: Timed out`.

**Обход:** перед запуском `apply` либо закройте редактор схем, либо выполните любое интерактивное действие в PCB (подвиньте компонент и сохраните). В коде предусмотрено предупреждение (`check_write_crash_risk`) и повторные попытки с задержкой, но крах остаётся возможным – это дефект самого KiCad.

### Диагностические скрипты
В папке `kicadspoke/diagnostics/` находятся скрипты для отладки:
- `diagnose_first_write_crash.py` – воспроизводит лесенку чтений/записи для локализации краша.
- `test_custom_fields.py` – проверяет чтение поля `Role`.
- `test_move_one_cap.py` – минимальный тест перемещения компонента.
- `test_flip_one_cap.py` – тест флипа через GUI action.
- `test_create_one_via.py` – тест создания via.
- `test_pad_mirror_convention.py` – эмпирическая проверка зеркалирования падов при флипе.
- `get_selected_component.py` – выводит информацию о выделенных компонентах.
- `get_pad_bbox.py`, `diagnostic_keepout.py` – вспомогательные.

## Структура проекта (кратко)

```
kicadspoke/
├── kicadspoke_cli.py          # CLI точка входа
├── config.py                  # Загрузка YAML → датаклассы (поддерживает templates_file)
├── constants.py               # Глобальные константы
├── exceptions.py              # Иерархия исключений
├── validation.py              # Предварительные проверки (включая проверку цепей via, уникальность якорей, режим выделения)
├── registry.py                # Реестр via и треков (сверка с живыми элементами)
├── net_resolution.py          # Разрешение цепей для ClonePlacement и обратная параметризация
├── template_extraction.py     # Извлечение шаблона из выделения (с новыми опциями)
├── undo.py                    # Откат операции
├── geometry/                  # Геометрические расчёты (spoke_layout, keepout, thermal_grid, pad_projection, clone_geometry)
├── kicad/                     # Адаптер к KiCad IPC
├── placement/                 # Планировщик, исполнители, сервисы (collision, planner, executor, services)
│   ├── services/              # Включая clone_role_resolver с разрешением по близости к якорю и выделению
├── cloner/                    # Файловый клонер (extract, netlist, pcb, models, sexp)
├── diagnostics/               # Диагностические скрипты
└── tests/                     # Модульные и интеграционные тесты
```
