> Это рабочие заметки, а не спецификация. Ничего здесь не "утверждено" и не версионируется.
> Задача файла — не забыть, до чего мы додумались, а не зафиксировать это в бетоне.

## Проблема, из которой всё началось

YAML-конфиг (шаблоны + правила размещения в одном файле) стал громоздким и рискованным
для ручной правки. Настоящая причина — не формат, а то, что в одном файле смешаны две
разные по природе вещи: машинные факты и человеческая логика.

## Твёрдо решено (проверено рассуждениями нескольких независимых точек зрения,
## но ещё не кодом — статус "сильная гипотеза", не "доказано")

### Две сущности, разная природа

| Сущность | Кто создаёт | Формат | Характер |
|---|---|---|---|
| **Шаблон** (геометрия спицы) | `extract` (машина) | JSON/YAML | Инертные факты, никто не правит руками |
| **Размещение** (правила клонирования) | Человек | Python-сценарий | Логика: циклы, условия, повторы |

Шаблоны не редактируются руками — только через `extract`. Размещение пишется как код.

### Уровни внутренней модели (IR)

| Уровень | Что входит | Стабильность | Где живёт |
|---|---|---|---|
| **Физический слой** | Координаты, углы, слои, имена сетей, геометрия падов | стабилен — это факты KiCad | структура IR |
| **Семантический слой** | Банки питания, роли, группы симметрии, оси зеркала | будет меняться | НЕ в структуре IR |
| **Метаданные / теги** | Связь физики и семантики | меняется медленно | навешивается на объекты IR сверху |

Принцип: IR держим максимально физическим (то, что измеримо в KiCad). Смысл — тегами,
а не полями структуры. Так ядро остаётся стабильным, даже когда меняется словарь
понятий сверху.

### Lock-файл заменяем на `diff` + явный `apply`

Lock-файл существовал бы только ради читаемого git-диффа — слабое основание для новой
сущности. Вместо этого: `diff()` показывает план (сравнение сценария с реальной платой),
`apply()` — явно применяет. По аналогии с `terraform plan`/`apply`.

### Свой DSL — не делать

Год разработки лексера/парсера ради того, что Python даёт бесплатно (циклы, условия,
готовая экосистема IDE). Экономически не оправдано для этой задачи.

### API — не проектировать заранее, растить из повторения

Функции в `generate_config.py` пишутся как узкие, специализированные, даже "некрасивые"
глаголы (`clone_dac_filters`, `mirror_power_stage`). Когда одна и та же конструкция
встретится 5-7 раз в разных сценариях — тогда она превращается в устойчивую функцию
в общем модуле. Не раньше.

## Гипотезы для проверки на практике (НЕ API, просто рабочие названия для обсуждения)

Ниже — глаголы, которые звучали в разборе реальных плат (FPGA, ЦАП). Это не сигнатуры
для реализации, а слова-кандидаты, которые нужно проверить на 3-4 разных платах, прежде
чем фиксировать их форму:

- `clone` — разместить копию шаблона
- `distribute` — размножить шаблон по списку падов/точек
- `mirror` — отразить группу
- `fanout` — развести via/выводы веером
- `extract_template` — вытащить геометрию эталонного блока

Конкретные параметры и порядок аргументов у них пока не фиксируем — они меняются каждый
раз, когда мы честно пишем "код мечты" для новой платы.

## Открытые вопросы / известные дыры в IR

- **Геометрия падов якорного компонента.** Нужен способ спросить "где пад №N у компонента X"
  с учётом текущего поворота компонента на плате.
- **Нормаль пада** (куда "смотрит" вывод). Рабочая формула:
  `normal_local = normalize(pad.position - footprint.origin)`, затем повернуть на угол
  футпринта на плате. Ограничение: работает для периметрийных корпусов (QFN/QFP/BGA),
  **не работает** для DIP/SOIC (пады не радиальны относительно центра) — это надо явно
  документировать как ограничение метода, а не решать как общий случай.
- **Земляные зоны (AGND/DGND).** Простое переименование сети (`net_mapping["GND"] = "AGND"`)
  не решает задачу, если AGND/DGND — физически разные полигоны на плате. IR должен уметь
  привязать via к правильной зоне по геометрии, а не только по имени сети. Это отдельная,
  пока нерешённая, задача — не пытаться закрыть её "на будущее", ждать реального кейса.
- **Стадия commit.** Нужно явно решить: `diff()` — это просмотр до применения (нужен
  отдельный `apply()`), или модель транзакции устроена иначе. Пока не проверено на коде.

## Ход обсуждения (для памяти, не для действия)

| Вопрос | К чему пришли |
|---|---|
| YAML стал громоздким | Проблема не в формате, а в смешении данных и логики |
| Альтернативы формату (JSON5/TOML/Python/свой DSL) | Свой DSL отклонён, Python — для логики размещения |
| Как стабилизировать IR | Разделение на физический (стабилен) и семантический (эволюционирует через теги) слои |
| Lock-файл или что-то другое | `diff` + явный `apply`, без lock-файла |
| Когда проектировать API | Не заранее — расти из повторяющихся глаголов в `generate_config.py` |
| Тестовый кейс для проверки гипотез | 8-канальный ЦАП (смешанная топология, AGND/DGND, веерное размещение) выбран как первый |
