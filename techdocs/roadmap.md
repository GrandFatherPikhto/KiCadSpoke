# Roadmap развития KiCadSpoke

> **Версия:** 1.0  
> **Горизонт:** 6–12 месяцев  
> **Статус:** Утверждена по итогам архитектурного обсуждения (июль 2026)  
> **Связь:** Основана на финальной архитектуре (см. `architect.md`)

---

## 1. Введение

Данный документ фиксирует план развития KiCadSpoke на ближайшие 6–12 месяцев. Дорожная карта построена на основе архитектурных решений, принятых в ходе дискуссии («Симпозиум»), и отражает эволюционный подход: мы не проектируем всё заранее, а выращиваем API из реальной практики.

**Ключевые принципы, заложенные в roadmap:**

1. **Разделение шаблонов и размещения** (машина vs человек).
2. **Эволюционный дизайн API** — глаголы рождаются из повторяющихся сценариев.
3. **Физический IR** — ядро хранит только измеримые факты, семантика — в тегах.
4. **Прозрачность** — `diff` вместо lock-файлов, явные `plan`/`apply`.

---

## 2. Структура дорожной карты

Roadmap разделена на фазы, каждая из которых имеет:

- **Цель** — что достигается в этой фазе.
- **Задачи** — конкретные шаги.
- **Критерии готовности** — как понять, что фаза завершена.
- **Срок** — оценочный (может корректироваться).

Фазы могут перекрываться, а задачи внутри фазы выполняться параллельно.

---

## 3. Фаза 0: Исследование и прототипирование (сейчас – 1 месяц)

### Цель
Наработать практику написания сценариев для реальных плат, выявить устойчивые глаголы и дыры в IR до того, как фиксировать API.

### Задачи

| № | Задача | Описание |
|---|--------|----------|
| 0.1 | Создать `generate_config.py` как лабораторию | Реализовать скелет с классами `Board`, методами `clone`, `diff`, `apply`. Использовать существующий `kipy`-адаптер для получения геометрии падов (`get_pad_geometry`). Пока генерировать YAML и запускать `kicadspoke_cli.py apply` через `subprocess`. |
| 0.2 | Написать сценарии для 3–4 различных плат | FPGA с банками питания, многоканальный ЦАП, полумост, (опционально) плата с повторяющимися аналоговыми фильтрами. |
| 0.3 | Собирать повторяющиеся конструкции | Оформлять их как функции-глаголы в отдельном модуле (`actions.py`). Документировать, какие операции повторяются чаще всего. |
| 0.4 | Зафиксировать «код мечты» | Создать файл `techdocs/future_boards_examples.py` с идеальными сценариями (как в обсуждении). Это поможет сверять реальный API с идеальным. |

### Критерии готовности

- [ ] Есть 3 рабочих сценария на Python для разных типов плат.
- [ ] Выявлены 5–7 повторяющихся глаголов (например, `clone`, `distribute`, `mirror`, `fanout`).
- [ ] `get_pad_geometry` реализована через реальный адаптер (не заглушка) и протестирована на реальной плате.
- [ ] Задокументированы первые «дыры» в IR (чего не хватает для реализации всех сценариев).

### Связь с дискуссией

Фаза 0 напрямую реализует принцип «API должно вырастать из практики, а не придумываться заранее». Мы сознательно отказываемся от проектирования абстракций до того, как увидим реальные сценарии.

## 4. Фаза 1: Перенос ключевых глаголов в ядро (1–2 месяца)

### Цель
Сделать наиболее востребованные операции частью самого KiCadSpoke, чтобы пользователь мог писать сценарии без промежуточной генерации YAML.

### Задачи

| № | Задача | Описание |
|---|--------|----------|
| 1.1 | Реализовать `get_pad_geometry` в адаптере | Добавить метод в `KiCadBoardAdapter`, возвращающий позицию пада и нормаль (вектор от центра футпринта к паду). Документировать ограничение: работает для периметрийных корпусов (QFN, QFP, BGA), но не для DIP/SOIC. |
| 1.2 | Реализовать команду `diff` | `kicadspoke_cli.py diff <config.py>` — сравнивает сценарий с текущим состоянием платы, выводит список изменений в читаемом виде. |
| 1.3 | Реализовать нативную загрузку Python-конфигов | Добавить в CLI поддержку `.py` файлов через `importlib` (соглашение: переменная `config`). Это позволяет применять сценарии без генерации YAML. |
| 1.4 | Перенести `apply` в ядро | Ядро должно уметь принимать Python-объекты напрямую. Сохранить обратную совместимость с YAML. |
| 1.5 | Реализовать базовые глаголы | `clone`, `mirror` (простое зеркалирование), `distribute` (для простых случаев, без сложной логики). |

### Критерии готовности

- [ ] `get_pad_geometry` работает на реальной плате и возвращает корректные нормали.
- [ ] `kicadspoke_cli.py diff config.py` показывает осмысленный план.
- [ ] `kicadspoke_cli.py apply config.py` применяет изменения без промежуточного YAML.
- [ ] Базовые глаголы работают для 2–3 типовых сценариев.

### Связь с дискуссией

Фаза 1 реализует решение о замене lock-файла на `diff` (план) и добавляет нативную поддержку Python-конфигов (отказ от промежуточного генератора как основного способа работы).

## 5. Фаза 2: Поддержка семантических тегов и зон (2–3 месяца)

### Цель
Решить проблемы, связанные с разделением AGND/DGND и другими семантическими связями, которые не сводятся к простой геометрии.

### Задачи

| № | Задача | Описание |
|---|--------|----------|
| 2.1 | Ввести понятие «зона» (zone) в IR | Via и компоненты могут быть привязаны к конкретной зоне (полигону) на плате. При клонировании зона наследуется или переопределяется через `net_mapping`. |
| 2.2 | Реализовать механизм тегов | Пользователь может навешивать на объекты произвольные теги (`metadata_tags`). Теги не влияют на геометрию, но используются для валидации и фильтрации. |
| 2.3 | Добавить поддержку зеркалирования групп | Зеркалирование с учётом тегов (чтобы сохранять семантику при отражении). |
| 2.4 | Реализовать более сложные глаголы | `distribute` с учётом зон, `fanout` (разводка via веером), `align` (выравнивание групп). |

### Критерии готовности

- [ ] Зоны работают: via клонированного блока попадают в правильные полигоны.
- [ ] Теги сохраняются при клонировании и зеркалировании.
- [ ] Сценарий для ЦАП с AGND/DGND работает корректно.
- [ ] Глаголы `distribute` и `fanout` реализованы и протестированы.

### Связь с дискуссией

Фаза 2 решает проблему, вскрытую в ходе обсуждения ЦАП: `net_mapping["GND"] = "AGND"` недостаточно, если AGND и DGND — физически разные зоны. Также реализует принцип «IR держать физическим, семантику — тегами сверху».

## 6. Фаза 3: Расширение на другие типы плат (3–4 месяца)

### Цель
Адаптировать инструмент под различные семейства плат (RF, цифровые, силовые) и собрать библиотеку готовых решений.

### Задачи

| № | Задача | Описание |
|---|--------|----------|
| 3.1 | Собрать библиотеку готовых шаблонов | Типовые блоки: LDO, фильтры, драйверы, согласующие цепи. Шаблоны хранятся в JSON, прилагаются примеры использования. |
| 3.2 | Добавить поддержку радиальных массивов | Размещение компонентов по кругу (полярные координаты) с автоматическим расчётом углов. |
| 3.3 | Реализовать `panelize` | Создание панелей из нескольких плат (если потребуется). |
| 3.4 | Улучшить `diff` | Добавить цветной вывод, группировку изменений по типам, экспорт в HTML. |

### Критерии готовности

- [ ] Библиотека содержит не менее 10 готовых шаблонов.
- [ ] Радиальные массивы работают для ЦАП и других круговых топологий.
- [ ] `panelize` реализован (если актуально).
- [ ] `diff` стал удобным инструментом для аудита.

### Связь с дискуссией

Фаза 3 расширяет инструмент на те сценарии, которые были упомянуты в обсуждении (радиальные массивы, многоканальные системы). Библиотека шаблонов реализует принцип переиспользования.

### Цель
Сделать инструмент доступным для сообщества, собрать обратную связь и продолжить эволюцию.

### Задачи

| № | Задача | Описание |
|---|--------|----------|
| 4.1 | Написать руководство пользователя | Инструкции по установке, примеры для типовых задач, описание API. |
| 4.2 | Создать репозиторий с примерами | Готовые сценарии для популярных плат (FPGA, ЦАП, полумост). |
| 4.3 | Организовать обратную связь | Issues, Discussions, опросы для сбора новых глаголов и улучшений. |
| 4.4 | Поддерживать обратную совместимость | Старые YAML-конфиги продолжают работать, новые функции не ломают существующие сценарии. |

### Критерии готовности

- [ ] Документация покрывает все основные сценарии.
- [ ] Есть 5–10 примеров в репозитории.
- [ ] Активное сообщество (не менее 5 внешних контрибьюторов или активных пользователей).
- [ ] Нет критических багов, связанных с обратной совместимостью.

### Связь с дискуссией

Фаза 4 закрепляет открытость проекта и готовность к развитию на основе реального опыта пользователей, а не только автора.

## 9. Риски и их смягчение

| Риск | Вероятность | Влияние | Смягчение |
|------|-------------|---------|-----------|
| Слишком ранняя фиксация API | Средняя | Высокое | Использовать `generate_config.py` как полигон, переносить в ядро только устойчивые глаголы (принцип «3 раза → задуматься, 7 раз → оформить») |
| Сложность поддержки зон и тегов | Средняя | Среднее | Внедрять постепенно, начиная с простых случаев (сначала только AGND/DGND) |
| Зависимость от KiCad IPC | Низкая | Высокое | Использовать существующий адаптер, тестировать на реальных платах, документировать ограничения |
| Разрастание кода | Средняя | Среднее | Модульная структура: ядро (IR), адаптер (KiCad), действия (глаголы) — разделены |
| Низкая вовлечённость сообщества | Средняя | Низкое | Активная документация, примеры, быстрая обратная связь на issues |
| Изменение API KiCad | Низкая | Среднее | Использовать стабильные методы `kipy`, иметь fallback для устаревших версий |

## 10. Заключение

Дорожная карта построена так, чтобы минимизировать риски и давать быструю обратную связь. На каждом этапе мы опираемся на реальный опыт использования, а не на гипотетические сценарии. Это позволяет сохранить гибкость и не перегружать ядро преждевременными абстракциями.

Успех проекта зависит от:

1. **Готовности инженеров писать сценарии и делиться ими** — через `generate_config.py` мы собираем практику.
2. **Обратной связи сообщества** — новые глаголы и улучшения рождаются из реальных задач.
3. **Дисциплины в реализации** — не добавлять функциональность, пока она не понадобилась 7 раз.

Мы открыты к изменениям и уточнениям по мере накопления практики. Следующий пересмотр roadmap — через 6 месяцев или после завершения Фазы 1.
