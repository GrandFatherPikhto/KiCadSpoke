# Варианты развития конфигурации KiCadSpoke

## Введение: две сущности с разными потребностями

В экосистеме KiCadSpoke есть **два принципиально разных типа данных**:

| Сущность | Кто создаёт | Характер | Требования |
|----------|-------------|----------|------------|
| **Шаблоны** (геометрия спиц) | Только `extract` (машина) | Инертные факты: координаты, цепи, параметры via/треков | Надёжная сериализация, предсказуемый diff, **никто не правит руками** |
| **Размещение** (правила `clone_placements`) | Человек (руками или через логику) | Логика: циклы, условия, параметризация, повторы | Читаемость, отсутствие рутины, возможность автоматизации |

Смешивать их в одном формате — значит создавать конфликт: машине нужна строгость, человеку — гибкость.

## Детальный разбор каждого варианта

### 1️⃣ YAML + Промежуточный генератор (`generate_config.py`)

**Суть:**  
Вы оставляете ядро KiCadSpoke «глупым» (принимает YAML), а для сложных плат пишете Python-скрипт, который генерирует этот YAML. Скрипт может использовать циклы, математику, читать шаблоны из JSON.

**✅ Что хорошо сейчас:**
- Работает **уже сегодня**, не требует изменений в коде KiCadSpoke.
- Даёт всю мощь Python для генерации логики.
- На выходе — валидный YAML, который можно глазами проверить перед запуском.
- Идеальная безопасность: исполняется только генератор, ядро получает инертные данные.

**⚠️ Что будет тормозить в будущем:**
- **Проблема двух источников правды:** если вы вручную поправите YAML (или в KiCad), а потом перезапустите генератор — ручные правки затираются.
- **Слепые Git-диффы:** при изменении одной формулы в генераторе пересчитываются сотни координат, и в Git появляется простыня из тысяч строк, которую невозможно осмысленно прочитать.
- **Отсутствие обратной связи:** изменения, сделанные вручную в KiCad, не могут быть «втянуты» обратно в генератор.
- **Сложность отладки:** ошибка валидации ссылается на строку в сгенерированном YAML, а не на логику в генераторе.
- **Порог входа:** каждый новый пользователь должен писать свой генератор, нет единого стандарта.

**Когда применять:** для быстрых прототипов, если плата небольшая и логика не меняется часто. **Не подходит** для долгосрочных проектов с частыми итерациями.

---

### 2️⃣ Раздельный TOML/YAML (шаблоны в JSON, размещение в плоском TOML)

**Суть:**  
Шаблоны (геометрия) остаются в машиночитаемом формате (JSON), а правила размещения (`clone_placements`) пишутся человеком в TOML (или плоском YAML) без вложенных структур. Отступы и кавычки больше не мучают, но нет циклов.

**✅ Что хорошо сейчас:**
- **Архитектурная чистота:** данные машины и человека разделены.
- **Безопасность:** всё ещё инертные данные, нет исполняемого кода.
- **Прозрачные диффы:** каждое изменение видно точечно.
- **Простота:** TOML не требует кавычек для ключей и поддерживает комментарии.

**⚠️ Что будет тормозить в будущем:**
- **Отсутствие циклов:** для 16 каналов питания придётся копировать блок 16 раз. Это рутина, которая ведёт к ошибкам.
- **Нет параметризации:** нельзя автоматически генерировать имена цепей (CH0, CH1, …) — только руками.
- **Сложность поддержки:** при изменении структуры размещения нужно править все блоки вручную.
- **Нет автодополнения:** IDE не подскажет доступные поля.

**Когда применять:** для очень простых плат с 1–3 клонами, где циклы не нужны, и важна максимальная безопасность данных. **Не подходит** для многоканальных систем (FPGA, ЦАП, массивы питания).

---

### 3️⃣ Python-конфиг (динамическая загрузка через `importlib`)

**Суть:**  
KiCadSpoke умеет загружать `.py`-файлы, исполнять их и извлекать объект `config: BoardConfig`. Пользователь пишет конфиг на чистом Python с циклами, условиями, импортами. Для прозрачности диффов вводится **Lock-файл** (слепок финальных координат в JSON).

**✅ Что хорошо сейчас:**
- **Максимальная гибкость:** циклы, условия, математика, импорт библиотек — всё доступно.
- **Автодополнение и проверка типов в IDE:** благодаря dataclass/pydantic IDE подсветит ошибки до запуска.
- **Переиспользование:** можно выносить общие правила в отдельные модули и импортировать в разные проекты.
- **Lock-файл решает проблему слепых диффов:** в Git видно и логику (`.py`), и финальные координаты (`.lock.json`).
- **Единый интерфейс:** пользователь запускает `kicadspoke_cli.py apply board.py` — никаких промежуточных генераторов.

**⚠️ Что будет тормозить в будущем:**
- **Исполняемый код вместо данных:** конфиг перестаёт быть инертным, это программа. Загрузка чужого конфига — выполнение чужого кода (риск безопасности).
- **Необходимость поддерживать стабильность API:** если вы переименуете класс `ClonePlacement`, старые конфиги сломаются. Нужна версионность или миграции.
- **Сложность для непрограммистов:** не каждый инженер-электронщик захочет писать Python-код для простой платы.
- **Медленная загрузка при тяжёлой логике:** если пользователь напишет ресурсоёмкий расчёт, это затормозит запуск.

**Когда применять:** **рекомендуемый вариант для развития проекта**. Идеален для сложных, многоканальных плат, где требуется автоматизация. Lock-файл делает его безопасным для Git-истории. Подходит как для личного использования, так и для опенсорса (с чёткими предупреждениями о безопасности).

### 4️⃣ Свой DSL (создание собственного языка конфигурации)

**Суть:**  
Разработать специальный язык (например, на основе Lark или PLY) для описания размещения, с собственным синтаксисом, без лишнего шума. Например:  
`clone pi_filter_4 at IC1.102 angle 90 params(PWR_IN=+1V2)`

**✅ Что хорошо сейчас:**
- **Абсолютный контроль над синтаксисом:** язык будет делать ровно то, что нужно для плат.
- **Безопасность и инертность:** остаются данными, не исполняются как код.
- **Потенциально лаконичнее YAML** для простых случаев.

**⚠️ Что будет тормозить в будущем:**
- **Год разработки:** написать стабильный парсер с поддержкой переменных, циклов, условий, вложенности — огромный проект.
- **Нет экосистемы:** нет подсветки синтаксиса, автодополнения, форматирования в IDE.
- **Нет проверок типов:** ошибки будут всплывать только во время выполнения.
- **Сложность поддержки:** каждый новый синтаксический сахар требует доработки парсера.
- **Низкая гибкость:** если потребуется математика, тригонометрия, импорт внешних данных — придётся расширять язык, а не просто использовать готовую библиотеку.

**Когда применять:** **никогда.** Это классическая ловушка «изобретения велосипеда». Python уже даёт всё, что нужно, бесплатно и с готовой экосистемой. Создавать свой DSL для этой задачи — неоправданные затраты.

## 📌 Итоговая рекомендация

**Выбирайте Вариант 3 (Python-конфиг с Lock-файлом) как основной вектор развития.**

- **Для шаблонов (геометрии):** оставьте YAML/JSON — они генерируются `extract`, их никто не пишет руками.
- **Для размещения (clone_placements):** переходите на Python. Это даёт максимальную гибкость при минимальной цене.
- **Lock-файл** решает проблему слепых диффов и даёт полную прозрачность изменений в Git.
- Для простых плат можно оставить поддержку YAML (обратная совместимость), но рекомендовать Python как основной способ.

**Чего делать не стоит:**
- Не создавайте свой DSL — это гигантский объём работы, не имеющий отношения к основной задаче.
- Не останавливайтесь на варианте 1 (генератор) — он быстро приводит к проблемам синхронизации и плохим диффам.
