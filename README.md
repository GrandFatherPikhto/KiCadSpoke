# Общее назначение программы

`decap_placer` — это инструмент командной строки для **автоматизированной расстановки развязывающих конденсаторов** (декаплинговых конденсаторов) и **переходных отверстий (via)** в печатных платах, разработанных в среде **KiCad**.

Программа подключается к открытому экземпляру KiCad через IPC (протокол взаимодействия между процессами) и выполняет:

- Перемещение и поворот компонентов (конденсаторов) на заданные позиции.
- Создание переходных отверстий трёх типов:
  - **Power via** – для подвода питания к выводам микросхемы (IC).
  - **GND via** – для подключения земляных выводов конденсаторов к земляному слою.
  - **Thermal via** – массив отверстий под термопадом для отвода тепла.
- Откат последней операции (undo) с восстановлением исходного состояния платы.

Ключевая особенность – **ручная стратегия позиционирования**: пользователь один раз описывает геометрию спицы (шаблон) в локальной системе координат, а затем применяет этот шаблон к конкретным падам микросхемы, задавая сдвиг и поворот для подгонки под конкретный корпус. Это делает конфигурацию компактной, переиспользуемой и понятной.

```bash
# Установить зависимости (пример)
```

## Запуск

### Основная команда – применить расстановку

```bash
python decap_placer.py apply <конфиг.yaml> [опции]
```

```bash
python decap_placer.py undo
```

## Формат конфигурационного файла (YAML)

Конфигурация состоит из трёх основных разделов:

1. **`templates`** – описание шаблонов спиц (локальные координаты и параметры via).
2. **`rules`** – привязка шаблонов к конкретным падам микросхемы с указанием сдвига, поворота и refdes компонентов.
3. **`thermal_via_array`** – настройки массива термовиа (опционально).

**Минимальный пример:**

```yaml
target_ref: IC1
side: back
templates:
  default:
    power_via:
      offset_along_mm: 1.5
      offset_across_mm: 0.0
    component1:
      offset_along_mm: 1.0
      offset_across_mm: 1.0
      angle_deg: 0.0
rules:
- net: +3V3_VCCIO
  spokes:
  - pad: '17'
    template: default
    shift_x_mm: 0.0
    shift_y_mm: 0.0
    rotation_deg: 0.0
    component1_ref: C5
```

Полное описание формата см. в документации `config.py` или в примерах в репозитории.

---

## Примечания

- Программа требует **открытого экземпляра KiCad** с активной платой.
- Все координаты в конфигурации задаются в **миллиметрах**.
- Углы задаются в **градусах** (поворот шаблона `rotation_deg` и угол компонента `angle_deg` суммируются).
- Команда `undo` откатывает **только последнюю выполненную операцию** (логи хранятся в папке `logs/`).
- В режиме `--dry-run` via не показываются (их позиции зависят от реального размещения компонентов, которое не происходит).

# `decap_placer.py` – Главная точка входа

## Назначение

`decap_placer.py` – это основной исполняемый скрипт, который запускает процесс расстановки развязывающих конденсаторов и переходных отверстий согласно конфигурационному YAML-файлу. Он обеспечивает:

- Парсинг аргументов командной строки.
- Загрузку конфигурации.
- Подключение к KiCad через IPC.
- Планирование и выполнение операций (перемещение компонентов, создание via).
- Логирование операций в JSON для последующего отката (`undo`).
- Поддержку режима `--dry-run` для предварительного просмотра.

Скрипт реализует **двухфазный процесс**:
1. **Фаза перемещений** – компоненты перемещаются на новые позиции и поворачиваются.
2. **Фаза via** – после перемещений плата перечитывается, и via создаются с учётом реального положения компонентов (особенно важно для GND via, которые привязаны к земляным падам).

Это обеспечивает корректное позиционирование via относительно уже размещённых компонентов.

## Внутреннее устройство

### Основные функции

| Функция | Описание |
|---------|----------|
| `setup_logging(verbose, log_file)` | Настраивает логирование: уровень (INFO/DEBUG), вывод в консоль и/или файл. |
| `cmd_apply(args)` | Выполняет основную расстановку: загружает конфиг, подключается к KiCad, планирует и выполняет перемещения и via. |
| `cmd_undo(args)` | Откатывает последнюю операцию, используя JSON-лог из папки `logs/`. |
| `main()` | Парсит аргументы, настраивает логирование и вызывает соответствующую команду. |

### Двухфазное выполнение

1. **Планирование перемещений** – `planner.plan_moves()` возвращает список `MoveCommand`.
2. **Применение перемещений** – `executor.execute_moves()` применяет их на плате.
3. **Перечитывание платы** – `adapter.refresh_board()` обновляет данные после перемещений.
4. **Планирование via** – `planner.plan_vias()` рассчитывает позиции via уже по реально размещённым компонентам.
5. **Применение via** – `executor.execute_vias()` создаёт via на плате.
6. **Логирование операции** – JSON-файл сохраняется в `logs/operation_YYYYMMDD_HHMMSS.json`.

### Логирование и откат

- Каждая успешная операция сохраняется в JSON-файл с полной информацией о перемещениях и созданных via (включая UUID).
- Команда `undo` читает последний JSON-файл, удаляет все созданные via и возвращает компоненты на исходные позиции и слои.
- После успешного отката JSON-файл удаляется, чтобы нельзя было откатить одну операцию дважды.

### Обработка ошибок

- Скрипт перехватывает пользовательские исключения (`PlacerError`) и выводит понятное сообщение.
- Неожиданные исключения логируются с полным стектрейсом.
- В случае ошибки во время транзакции выполняется автоматический откат (через `drop_commit` внутри адаптера и `commit_with_retry`).

## Примечания

- **Dry-run** не показывает via, так как их позиции зависят от реального положения компонентов после перемещения. Для точного предпросмотра необходимо запускать без `--dry-run` (или сначала сделать перемещения, а затем отдельно via, но это требует отдельного механизма, который пока не реализован).
- Проверка коллизий основана на упрощённой модели (круги) и может давать ложные срабатывания. Рекомендуется использовать `--no-collision-check`, если KiCad не выдаёт ошибок при размещении.
- Логи операций сохраняются в папку `logs/`, которая создаётся автоматически при первом запуске.
- Для успешного отката необходимо, чтобы JSON-файл операции был целым и соответствовал текущему состоянию платы (не рекомендуется редактировать файлы вручную).
# `config.py` – Загрузка и хранение конфигурации

## Назначение

Модуль `config.py` определяет структуру данных для всего проекта (dataclass'ы) и обеспечивает загрузку конфигурации из YAML-файла. Конфигурация содержит описание целевого компонента (IC), шаблонов спиц, правил привязки к падам, а также параметры для термовиа и опций выполнения (например, пропуск уже расставленных компонентов).

Модуль является единственным местом, где задаётся формат входного YAML-файла, и централизованно преобразует его в типизированные объекты Python, которые затем используются во всех остальных частях программы.

---

## Структура данных (dataclass'ы)

Модуль определяет иерархию классов, полностью описывающую конфигурацию:

- **`ThermalViaArrayConfig`** – параметры массива тепловых via под термопадом IC.
- **`TemplatePowerVia`** – описание power via в шаблоне спицы (локальные координаты along/across, цепь, размеры).
- **`TemplateComponentSlot`** – описание одного слота (компонента) в шаблоне: локальные координаты, угол, смещение для GND via.
- **`SpokeTemplate`** – полный шаблон спицы: имя, опциональные power via и два слота для компонентов.
- **`ManualSpoke`** – конкретная спица на плате: номер пада, ссылка на шаблон, сдвиг (shift_x, shift_y), поворот (rotation_deg), refdes компонентов.
- **`Rule`** – правило для одной цепи питания: имя цепи и список спиц (`ManualSpoke`).
- **`Config`** – главный объект: целевой компонент, сторона платы, словарь шаблонов, массив термовиа, список правил, флаги (`place_components`, `skip_existing_components`) и параметры поиска свободного места для термовиа.

---

## Функции

| Функция | Описание |
|---------|----------|
| `_load_template_power_via(data)` | Загружает `TemplatePowerVia` из словаря, возвращает `None`, если данных нет. |
| `_load_template_component_slot(data)` | Загружает `TemplateComponentSlot`, возвращает `None` при отсутствии. |
| `_load_spoke_template(name, data)` | Загружает полный `SpokeTemplate` по имени из секции `templates`. |
| `_load_manual_spoke(data)` | Загружает `ManualSpoke` из секции `spokes` внутри правила. |
| **`load_config(path)`** | **Основная функция**: принимает путь к YAML-файлу, парсит его и возвращает экземпляр `Config`. Обрабатывает все вложенные структуры, устанавливает значения по умолчанию, логирует результат. |

---

## Параметры YAML-конфигурации (корневые)

| Параметр | Тип | Описание | Пример |
|----------|-----|----------|--------|
| `target_ref` | строка | Рефдес целевого компонента (микросхемы) | `"IC1"` |
| `side` | строка | Сторона платы (`"back"` или `"front"`) | `"back"` |
| `templates` | словарь | Шаблоны спиц (ключ – имя шаблона, значение – структура) | см. ниже |
| `thermal_via_array` | словарь | Параметры термовиа | `enabled: true`, `pad: '145'` и т.д. |
| `rules` | список | Правила для цепей питания | см. ниже |
| `place_components` | булево | Перемещать компоненты (если `false` – только via) | `true` |
| `skip_existing_components` | булево | Пропускать уже расставленные компоненты | `false` |
| `via_keepout_clearance_mm` | число | Зазор для keepout (только для термовиа) | `0.2` |
| `via_search_step_mm` | число | Шаг поиска свободного места | `0.1` |
| `via_search_max_radius_mm` | число | Максимальный радиус поиска | `3.0` |
| `via_search_n_directions` | число | Количество направлений для поиска | `8` |

---

### Структура `templates`

```yaml
templates:
  имя_шаблона:
    power_via:               # опционально
      offset_along_mm: 0.0
      offset_across_mm: 0.0
      net: "+3V3_VCCIO"      # если не указан, берётся из rule.net
      drill_mm: 0.3
      diameter_mm: 0.6
    component1:              # опционально
      offset_along_mm: 0.0
      offset_across_mm: 1.0
      angle_deg: 0.0
      gnd_via_offset_along_mm: 0.0
      gnd_via_offset_across_mm: 0.0
      gnd_via_net: "GND"
      gnd_via_drill_mm: 0.3
      gnd_via_diameter_mm: 0.6
    component2:              # аналогично component1
      ...
```

---

### Структура `rules`

```yaml
rules:
- net: "+3V3_VCCIO"          # имя цепи
  spokes:
  - pad: '17'                # номер пада FPGA
    template: "имя_шаблона"
    shift_x_mm: 0.0
    shift_y_mm: 0.0
    rotation_deg: 0.0
    component1_ref: "C5"     # refdes компонента для первого слота
    component2_ref: "C31"    # refdes компонента для второго слота
    enabled: true
  - pad: '26'
    ...
```

## Типовые комбинации

### 1. Базовая расстановка (с перемещением компонентов и via)

```yaml
target_ref: IC1
side: back
place_components: true
skip_existing_components: false
templates:
  default:
    power_via:
      offset_along_mm: 1.5
      offset_across_mm: 0.0
      drill_mm: 0.3
      diameter_mm: 0.6
    component1:
      offset_along_mm: 1.0
      offset_across_mm: 1.0
      angle_deg: 0.0
      gnd_via_offset_along_mm: 0.0
      gnd_via_offset_across_mm: 0.0
rules:
- net: "+3V3_VCCIO"
  spokes:
  - pad: '17'
    template: default
    shift_x_mm: 0.0
    shift_y_mm: 0.0
    rotation_deg: 0.0
    component1_ref: C5
```

### 2. Только via (без перемещения компонентов)

```yaml
place_components: false
rules:
- net: "+3V3_VCCIO"
  spokes:
  - pad: '17'
    template: default
    power_via:
      enabled: true
      ...
```

### 3. Поворот и сдвиг спицы для подгонки под конкретный корпус

```yaml
- pad: '17'
  template: default
  shift_x_mm: 0.5
  shift_y_mm: -0.3
  rotation_deg: 5.0
  component1_ref: C5
```

---

## Примечания

- Все геометрические параметры (`offset_along_mm`, `offset_across_mm`, `shift_x_mm`, `shift_y_mm`) задаются в **миллиметрах**.
- Углы (`rotation_deg`, `angle_deg`) задаются в **градусах**.
- Если в `TemplatePowerVia` не указан `net`, используется цепь из правила (`rule.net`).
- Параметры `via_keepout_clearance_mm`, `via_search_*` используются **только** для термовиа (все остальные via ставятся строго по координатам без поиска).
- Поле `enabled: false` в спице позволяет временно отключить её без удаления из конфига.

# `geometry/` – Геометрические утилиты

## Назначение

Модули в директории `geometry/` предоставляют набор низкоуровневых геометрических функций и классов для работы с координатами платы, ограничивающими областями (keepout), предсказанием позиции падов и поворотом локальных координат шаблонов спиц. Они используются во всех этапах расстановки: от построения зон занятости до вычисления абсолютных позиций via и компонентов.

---

## Структура

```
geometry/
├── __init__.py             # Экспорт публичных функций
├── keepout.py              # Ограничивающие прямоугольники и поиск свободного места
├── pad_projection.py       # Предсказание позиции пада после перемещения/поворота компонента
├── spoke_layout.py         # Преобразование локальных координат шаблона спицы в глобальные
├── thermal_grid.py         # Генерация сетки термовиа под площадкой
└── placement.py            # [УСТАРЕЛ] – оставлен для совместимости, не используется
```

---

## Файлы и функции

### `keepout.py`

**Назначение:**  
Определяет класс `Rect` (выровненный по осям прямоугольник) и предоставляет функции для построения keepout-областей из bounding box'ов, проверки точки на занятость и поиска свободного места для via.

**Основные функции:**

| Функция | Описание |
|---------|----------|
| `Rect.from_bbox(bbox, clearance)` | Создаёт прямоугольник из Box2 (от адаптера) с добавлением зазора `clearance` со всех сторон. |
| `Rect.from_circle(center, radius)` | Аппроксимирует окружность квадратом (для быстрой проверки via). |
| `point_is_clear(point, via_radius, keepout)` | Проверяет, свободна ли точка (окружность радиуса `via_radius` не пересекает ни один keepout-прямоугольник). |
| `build_keepout(bboxes, clearance_mm)` | Принимает список bounding box'ов, строит список `Rect` с учётом зазора. |
| `find_free_point(ideal, keepout, via_radius, ...)` | Ищет ближайшую свободную точку вокруг `ideal` по спирали (кольцами). Учитывает предпочтительное направление. |
| `find_free_point_along_line(ideal, keepout, via_radius, line_direction, ...)` | Ищет свободную точку вдоль прямой (в обе стороны от `ideal`). |

**Используется в:** `via_planner.py` для поиска места под термовиа (и ранее – для stitching via).

---

### `pad_projection.py`

**Назначение:**  
Предсказывает абсолютную позицию конкретного пада компонента, если компонент будет перемещён в новое место и повёрнут на заданный угол. Это критически важно для правильного расчёта позиции GND via, которая привязана к реальному положению земляного пада конденсатора.

**Основные функции:**

| Функция | Описание |
|---------|----------|
| `local_pad_offset(fp, pad)` | Возвращает смещение пада относительно центра футпринта в **неповёрнутой** локальной системе координат (постоянная геометрия). |
| `predict_pad_position(fp, pad, dest, angle_deg, needs_flip)` | Предсказывает абсолютную позицию пада после перемещения футпринта в `dest` и поворота на `angle_deg` с учётом возможного зеркалирования при флипе на обратную сторону. |

**Используется в:** `via_planner.py` (ранее для stitching via, теперь для GND via – через реальный, уже размещённый компонент).

---

### `spoke_layout.py`

**Назначение:**  
Преобразует локальные координаты шаблона спицы (`along`, `across`) в абсолютные координаты платы с учётом сдвига `(shift_x, shift_y)` и поворота `rotation_deg` конкретной спицы. Также вычисляет итоговый угол поворота компонента и сохраняет параметры GND via из шаблона.

**Основные классы и функции:**

| Класс/Функция | Описание |
|---------------|----------|
| `rotate_local_offset(along_mm, across_mm, rotation_deg)` | Поворачивает локальный вектор `(along, across)` на заданный угол вокруг нуля. |
| `local_to_absolute(origin, along_mm, across_mm, rotation_deg)` | Преобразует локальный вектор в абсолютную позицию относительно `origin` с учётом поворота. |
| `ComponentLayout` | Структура, описывающая размещение одного компонента: ref, позиция, угол, параметры GND via. |
| `SpokeLayout` | Структура, описывающая все элементы спицы: origin, power via (если есть), компоненты. |
| `apply_spoke_geometry(pad_position, spoke, template, rule_net)` | Основная функция – принимает позицию пада FPGA, данные спицы (`ManualSpoke`), шаблон (`SpokeTemplate`) и цепь правила, возвращает `SpokeLayout` с абсолютными координатами. |

**Используется в:** `manual_position_calculator.py` для расчёта позиций компонентов и в `via_planner.py` для power via.

---

### `thermal_grid.py`

**Назначение:**  
Вычисляет абсолютные координаты для массива термовиа под термопадом компонента (например, `IC1`). Учитывает размер пада, отступы от края, количество строк/столбцов и возможность шахматного порядка (stagger).

**Основные функции:**

| Функция | Описание |
|---------|----------|
| `get_pad_size(pad)` | Возвращает размеры медного слоя пада `(width, height)` в нанометрах. |
| `compute_thermal_via_grid(pad, rows, cols, margin_mm, stagger)` | Генерирует список абсолютных позиций для via, размещённых равномерно внутри площадки с учётом отступов `margin_mm`. |

**Используется в:** `via_planner.py` для планирования термовиа.

---

### `placement.py` – **УСТАРЕЛ**

Этот файл содержит старую функцию `compute_position`, которая использовалась для автоматического размещения компонентов относительно границы зоны (`boundary`). После перехода на ручную стратегию (шаблоны) этот файл больше не используется и оставлен только для обратной совместимости со старыми тестами. В ближайшее время будет удалён.

---

## Взаимосвязи с другими модулями

- `keepout.py` и `thermal_grid.py` используются в `via_planner.py` (при построении via).
- `pad_projection.py` используется в `via_planner.py` для предсказания позиции GND-пада (но теперь `via_planner` работает с уже перемещёнными компонентами, поэтому эта функция больше не применяется; код оставлен на случай, если понадобится в будущем).
- `spoke_layout.py` – ключевой модуль для новой ручной стратегии; используется в `manual_position_calculator.py` и `via_planner.py` для преобразования шаблонов в реальные координаты.

---

## Примечания по использованию

- Все координаты и размеры во всех функциях ожидаются в **нанометрах**, если не указано иное (поля `_mm` принимают миллиметры и конвертируются внутри).
- Для конвертации используйте константу `MM = 1_000_000` из `utils/units.py`.
- Функции из `keepout.py` (особенно `find_free_point`) сейчас используются только для термовиа, поскольку все остальные via размещаются строго по заданным координатам без поиска.

---

Если потребуется расширить геометрические возможности (например, добавить поддержку повёрнутых прямоугольников для коллизий), новые функции следует размещать здесь же, сохраняя единообразие стиля.

# `kicad/` – Адаптер для взаимодействия с KiCad через IPC

## Назначение

Модули в директории `kicad/` предоставляют унифицированный интерфейс для работы с платой в KiCad через библиотеку `kipy` (KiCad Python IPC). Они скрывают детали низкоуровневого протокола и обеспечивают устойчивость к сбоям связи (повторные попытки при ошибках «KiCad is busy»), а также упрощают выполнение транзакций (commit/rollback) и доступ к объектам платы.

---

## Структура

```
kicad/
├── __init__.py        # Экспорт публичного API
├── interfaces.py      # Абстрактный интерфейс IBoardAdapter
└── adapter.py         # Реализация KiCadBoardAdapter
```

### `interfaces.py`

Определяет абстрактный базовый класс `IBoardAdapter`, который описывает контракт для всех адаптеров к плате. Это позволяет легко подменять реализацию (например, для тестирования).  
Методы интерфейса:

- `refresh_board()` – обновить данные платы из KiCad.
- `get_footprint(ref)` – получить футпринт по референсу.
- `get_footprints()` – получить все футпринты.
- `get_footprint_pads(fp)` – получить площадки (пады) футпринта.
- `get_pad_by_number(fp, number)` – найти площадку по номеру.
- `get_zone_by_name(name)` – найти зону по имени.
- `get_net_by_name(name)` – найти цепь по имени.
- `get_bounding_boxes(items)` – получить ограничивающие прямоугольники для списка объектов.
- `begin_commit()` – начать транзакцию.
- `push_commit(commit, description)` – применить транзакцию.
- `drop_commit(commit)` – откатить транзакцию.
- `update_items(items)` – обновить объекты на плате.
- `create_items(items)` – создать новые объекты.
- `flip_selected(footprints)` – перевернуть выделенные футпринты (зеркалирование на другую сторону).
- `commit_with_retry(description, work_fn, retries)` – выполнить работу в транзакции с автоматическим повторением при ошибках.
- `create_via(position, net, drill_mm, diameter_mm)` – создать переходное отверстие.

### `adapter.py`

Реализация `KiCadBoardAdapter`, наследующего `IBoardAdapter`.

**Особенности реализации:**
- Подключается к KiCad через `kipy.KiCad` с заданным таймаутом (по умолчанию 20 с).
- Все методы чтения (get_*) используют повторные попытки при возникновении `ApiError` с сообщением «KiCad is busy».
- Транзакции (`begin_commit`, `push_commit`, `drop_commit`) работают напрямую с объектом платы.
- Метод `commit_with_retry` оборачивает произвольную функцию в транзакцию с автоматическим откатом при ошибке.
- `flip_selected` использует GUI-действие `pcbnew.InteractiveEdit.flip` для реального переворота компонента (зеркалирование площадок и шелкографии).

**Ключевые методы (кроме интерфейсных):**

- `refresh_board()` – перечитывает плату из KiCad. Обязательно вызывать после операций, изменяющих плату «в обход» `update_items` (например, после `flip_selected`), чтобы локальные объекты не устарели.
- `get_bounding_boxes(items)` – оптимизированный батч-запрос для получения Box2 для нескольких элементов (экономит вызовы IPC).
- `create_via(position, net, drill_mm, diameter_mm)` – создаёт объект `Via` (но не добавляет на плату; для добавления нужен `create_items`).

### `__init__.py`

Экспортирует публичные классы:
```python
from .adapter import KiCadBoardAdapter
from .interfaces import IBoardAdapter
```

---

## Использование в проекте

Адаптер используется во всех модулях, которые взаимодействуют с KiCad:
- **`placement/planner.py`** – для получения целевого компонента (IC) и его падов.
- **`placement/executor.py`** – для выполнения перемещений и создания via.
- **`placement/services/manual_position_calculator.py`** – для поиска футпринтов по refdes.
- **`placement/services/via_planner.py`** – для чтения падов и создания via.
- **`diagnostics/`** – для диагностических скриптов (получение bbox, отладка).

Благодаря интерфейсу `IBoardAdapter`, код не зависит от конкретной реализации и может быть легко протестирован с мок-объектами.

---

## Зависимости

- `kipy` – официальная Python-библиотека для IPC KiCad (устанавливается отдельно).
- `kipy.board_types` – типы данных KiCad (FootprintInstance, Pad, Via, Net и др.).
- `kipy.geometry` – геометрические примитивы (Vector2, Box2, Angle).

---

## Примечания

- Все координаты и размеры в адаптере ожидаются в **нанометрах** (внутренние единицы KiCad). Для перевода из мм в нм используется константа `MM = 1_000_000` из `utils/units.py`.
- В транзакциях важно перехватывать исключения и корректно откатывать `commit`, чтобы не оставлять незавершённых изменений.
- Метод `flip_selected` не обновляет локальный объект футпринта – после его вызова необходимо перечитать футпринт через `refresh_board()` и повторно получить объект.

# `placement/` – Планирование и исполнение расстановки

## Назначение

Модули в директории `placement/` реализуют основную логику расстановки: от расчёта позиций компонентов и via до их применения на плате через IPC. Они используют данные из конфигурации (шаблоны спиц) и адаптер к KiCad для выполнения операций.

---

## Структура

```
placement/
├── __init__.py                 # Экспорт публичных компонентов (опционально)
├── collision.py                # Проверка коллизий компонентов (упрощённая)
├── commands.py                 # Структуры данных для команд (MoveCommand, ViaCommand)
├── executor.py                 # Исполнитель команд: флип, перемещение, создание via
├── planner.py                  # Главный планировщик (координирует расчёт и выполнение)
└── services/                   # Сервисные классы
    ├── __init__.py
    ├── manual_position_calculator.py   # Расчёт позиций по шаблонам спиц
    └── via_planner.py                  # Планирование всех via (power, GND, thermal)
```

---

## Файлы и функции

### `commands.py`

**Назначение:**  
Определяет структуры данных, используемые для передачи команд между планировщиком и исполнителем.

**Классы:**

| Класс | Поля | Назначение |
|-------|------|------------|
| `MoveCommand` | `ref: str`, `position: Vector2`, `angle: Angle`, `layer: BoardLayer` | Команда перемещения/поворота компонента. |
| `ViaCommand` | `position: Vector2`, `drill_mm: float`, `diameter_mm: float`, `net_name: str`, `owner_ref: str` | Команда создания переходного отверстия. |
| `PlacedComponentInfo` | `ref: str`, `dest: Vector2`, `angle_deg: float`, `rotation_deg: float`, `gnd_via_offset_along_mm: float`, `gnd_via_offset_across_mm: float`, `gnd_via_net: str`, `gnd_via_drill_mm: float`, `gnd_via_diameter_mm: float` | Информация о размещённом компоненте (передаётся от калькулятора к via-планировщику). |

**Используется в:** `planner.py`, `executor.py`, `manual_position_calculator.py`, `via_planner.py`.

---

### `collision.py`

**Назначение:**  
Упрощённая проверка коллизий между компонентами (по кругам-приближениям). Использует реальные bounding box'ы через адаптер.

**Основные функции:**

| Функция | Описание |
|---------|----------|
| `compute_radii(footprints, adapter)` | Вычисляет радиусы (половина диагонали bbox) для каждого футпринта (батч-запрос). |
| `footprints_overlap(pos1, r1, pos2, r2, margin_mm)` | Проверяет перекрытие двух кругов с запасом. |
| `check_collisions(moves, all_footprints, adapter, ignore_refs, margin_mm)` | Проверяет коллизии между перемещаемыми компонентами и остальными. Возвращает список конфликтных пар. |

**Используется в:** `executor.py` (опционально, при включённой проверке).

---

### `executor.py`

**Назначение:**  
Применяет команды перемещения и создания via к реальной плате через адаптер. Разделяет фазы (сначала перемещения, потом via) с обязательным перечитыванием платы между ними для корректного расчёта GND via по реальным позициям.

**Класс `BatchExecutor`:**

| Метод | Описание |
|-------|----------|
| `__init__(adapter, config, batch_size)` | Инициализация с размером батча для коммитов. |
| `execute_moves(moves, check_collisions, collision_margin_mm)` | Применяет только перемещения. Возвращает список refdes, которые не удалось применить. |
| `execute_vias(vias)` | Создаёт via. Возвращает список owner_ref'ов, для которых via не создались. |
| `execute(moves, vias, ...)` | Обратно совместимая обёртка (выполняет обе фазы без перечитывания). Не рекомендуется для боевого использования. |
| `_needs_flip(cmd, fp_by_ref)` | Проверяет, нужно ли перевернуть компонент на целевой слой. |
| `_flip_in_batches(refs, fp_by_ref)` | Переворачивает компоненты батчами через GUI-action. |
| `_write_operation_log(move_log, via_log)` | Сохраняет JSON-лог операции для undo. |

**Важно:**  
После `execute_moves()` вызывающий код обязан выполнить `adapter.refresh_board()`, чтобы `execute_vias()` работала с актуальными данными (GND via рассчитываются от реально размещённых компонентов).

**Используется в:** `decap_placer.py` (главный скрипт) для применения плана.

---

### `planner.py`

**Назначение:**  
Главный планировщик – координирует расчёт позиций (через `ManualPositionCalculator`) и via (через `ViaPlanner`). Содержит двухфазную логику: сначала планируются перемещения, затем via.

**Класс `PlacementPlanner`:**

| Метод | Описание |
|-------|----------|
| `__init__(adapter, config)` | Инициализация, поиск целевого компонента (IC), определение слоя. |
| `plan_moves()` | Использует `ManualPositionCalculator` для расчёта позиций компонентов. Возвращает список `MoveCommand`. |
| `plan_vias()` | Использует `ViaPlanner` для расчёта via. Возвращает список `ViaCommand`. Вызывается **после** применения перемещений и перечитывания платы. |
| `plan()` | Обратно совместимая обёртка (вызывает `plan_moves()` и `plan_vias()` подряд). Для боевого использования не рекомендуется (см. `decap_placer.py`). |

**Используется в:** `decap_placer.py` для получения плана.

---

### `services/`

#### `manual_position_calculator.py`

**Назначение:**  
Расчёт абсолютных позиций компонентов на основе шаблонов спиц (`SpokeTemplate`) и конкретных данных спицы (`ManualSpoke`). Не использует геометрию зоны – только пад FPGA + сдвиг/поворот + локальные координаты шаблона.

**Класс `ManualPositionCalculator`:**

| Метод | Описание |
|-------|----------|
| `__init__(adapter, config)` | Инициализация с адаптером и конфигом. |
| `compute_raw_positions(target_fp, rules, side)` | Для каждой спицы применяет `apply_spoke_geometry` и возвращает список `PlacedComponentInfo`. Пропускает отключённые спицы и компоненты без refdes. |

**Используется в:** `planner.py`.

---

#### `via_planner.py`

**Назначение:**  
Планирует создание всех via:
- **Power via** – из шаблона спицы (локальные координаты, повёрнутые на `rotation_deg`).
- **GND via** – по локальному смещению от **реального** земляного пада компонента (используя уже размещённый компонент).
- **Thermal via** – сетка под термопадом (поиск свободных точек через keepout).

**Класс `ViaPlanner`:**

| Метод | Описание |
|-------|----------|
| `__init__(adapter, config)` | Инициализация. |
| `plan_vias(planned, target_fp, rules, target_layer)` | Основной метод. Строит keepout (для термовиа), вызывает `_plan_power_vias`, `_plan_gnd_vias`, `_plan_thermal_vias`. Возвращает список `ViaCommand`. |
| `_plan_power_vias(target_fp, rules)` | Power via – из шаблонов спиц. |
| `_plan_gnd_vias(planned)` | GND via – от реального GND-пада компонента + локальное смещение, повёрнутое на `rotation_deg`. |
| `_plan_thermal_vias(planned, target_fp, keepout)` | Термовиа – сетка с поиском свободных точек через `find_free_point`. |
| `_build_keepout(target_fp, planned, exclude)` | Строит keepout из падов IC и компонентов (для термовиа). |

**Используется в:** `planner.py` (после перемещений).

## Взаимосвязи с другими модулями

- **`kicad/adapter.py`** – используется для всех операций с платой (чтение, запись, транзакции).
- **`geometry/spoke_layout.py`** – используется в `manual_position_calculator.py` для преобразования локальных координат в глобальные.
- **`geometry/thermal_grid.py`** и **`geometry/keepout.py`** – используются в `via_planner.py` для термовиа и keepout.
- **`config.py`** – предоставляет структуры данных (Config, ManualSpoke, SpokeTemplate и т.д.).
- **`utils/units.py`** – константа `MM` для перевода миллиметров в нанометры.

---

## Примечания по использованию

- **Двухфазный процесс**:  
  1. Сначала выполняются перемещения (`execute_moves`).  
  2. Затем плата перечитывается (`adapter.refresh_board()`).  
  3. Планируются и выполняются via (`execute_vias`).  
  Это необходимо, чтобы GND via рассчитывались по реальному положению компонентов, а не по расчётному (которое может отличаться из-за флипа или других факторов).

- **Коллизии** – проверка включена по умолчанию, но может давать ложные срабатывания. Отключается флагом `--no-collision-check`.

- **Логирование операции** – сохраняется в `logs/operation_*.json` и используется командой `undo`.

- **Dry-run** – отображает только перемещения; via не показываются, так как их позиции зависят от реального размещения компонентов (которое в dry-run не происходит).

# `utils/` – Вспомогательные утилиты

## Назначение

Директория `utils/` содержит общеиспользуемые вспомогательные модули и константы, которые не относятся к основной логике расстановки, но необходимы для работы проекта. В основном это конвертация единиц измерения и вспомогательные функции для упрощения кода в других модулях.

---

## Структура

```
utils/
├── __init__.py        # Экспорт публичных функций
└── units.py           # Константы для перевода единиц измерения
```

---

## Файлы и функции

### `units.py`

**Назначение:**  
Содержит константу для перевода миллиметров в нанометры – внутреннюю систему координат KiCad (и библиотеки `kipy`). Все координаты и размеры в проекте хранятся в нанометрах, а пользовательские значения в конфигурации задаются в миллиметрах.

**Константа:**

| Имя | Значение | Описание |
|-----|----------|----------|
| `MM` | `1_000_000` | Количество нанометров в одном миллиметре. Используется для перевода ввода/вывода и в геометрических расчётах. |

**Используется во всех модулях, работающих с координатами:**  
- `kicad/adapter.py` – при создании via, обновлении позиций.
- `geometry/*.py` – во всех геометрических вычислениях.
- `placement/*.py` – при расчёте позиций и via.
- `diagnostics/*.py` – для отладки и вывода значений в мм.

---

### `__init__.py`

Экспортирует публичные функции (пока только `MM`).  
Обычно содержит:
```python
from .units import MM
```

Это позволяет использовать `from decap_placer.utils import MM` вместо полного пути.

---

## Использование в проекте

Во всех модулях, где требуется пересчёт между миллиметрами и нанометрами, используется константа `MM`:

```python
from decap_placer.utils import MM

# Перевести миллиметры в нанометры
value_nm = value_mm * MM

# Перевести нанометры в миллиметры (для вывода)
value_mm = value_nm / MM
```

---

## Примечания

- Все координаты в `kipy` (и в адаптере) ожидаются в нанометрах, поэтому конвертация выполняется **всегда** при передаче данных из конфига (в мм) в API KiCad.
- При отладке и логировании для удобства чтения координаты часто выводятся в мм, для чего используется деление на `MM`.
- Константа `MM` является целым числом, поэтому умножение/деление выполняется точно без потери точности (если значения заданы в мм с достаточной точностью).
