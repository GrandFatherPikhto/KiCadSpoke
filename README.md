# KiCad Decap Placer v4.0

Автоматическая расстановка развязывающих конденсаторов и переходных отверстий в KiCad с использованием шаблонов и ролей.

---

## Назначение

**KiCad Decap Placer** – это инструмент командной строки, который:

- Перемещает компоненты (конденсаторы) на заданные позиции.
- Создаёт переходные отверстия (via) трёх типов:
  - **Power via** – для подвода питания к выводам микросхемы.
  - **GND via** – для подключения земляных выводов конденсаторов.
  - **Thermal via** – массив отверстий под термопадом.
- Позволяет откатить последнюю операцию (undo).

Вместо ручного указания refdes программа автоматически подбирает компоненты по **ролям** (например, `LIGHT`/`HEAVY`) и цепи питания, используя пользовательские поля в схеме. Это делает конфигурацию компактной, переиспользуемой и устойчивой к изменениям в нумерации компонентов.

---

## Установка

### Требования

- Python 3.9 или новее
- KiCad 8 или новее (с включённым IPC-сервером)
- Установленная библиотека `kipy` (KiCad Python IPC)

### Зависимости

Установите необходимые пакеты:

```bash
pip install kipy pyyaml sexpdata numpy scipy
```

### Запуск

Клонируйте репозиторий и перейдите в директорию проекта:

```bash
git clone <url>
cd KiCadDecapPlacer
```

## Использование

### Основная команда – применить расстановку

```bash
python decap_placer.py apply <конфиг.yaml> [опции]
```

Если команда `apply` не указана, она подразумевается:

```bash
python decap_placer.py decap_placement.yaml
```

### Откат последней операции

```bash
python decap_placer.py undo
```

### Параметры командной строки (для `apply`)

| Аргумент | Описание | Пример |
|----------|----------|--------|

## Формат конфигурационного файла (YAML)

Конфигурация состоит из трёх основных разделов:

1. **`templates`** – шаблоны спиц (локальная геометрия, роли).
2. **`rules`** – привязка шаблонов к конкретным падам микросхемы.
3. **`thermal_via_array`** – настройки массива термовиа (опционально).

### Пример минимального конфига

```yaml
target_ref: IC1
side: back

templates:
  my_template:
    power_via:
      offset_along_mm: 1.5
      offset_across_mm: 0.0
      drill_mm: 0.3
      diameter_mm: 0.6
    components:
      - role: LIGHT
        offset_along_mm: 1.0
        offset_across_mm: 1.0
        angle_deg: 0.0
        gnd_via_offset_along_mm: 0.0
        gnd_via_offset_across_mm: 0.0
        gnd_via_net: GND

rules:
- net: +3V3_VCCIO
  spokes:
  - pad: '17'
    template: my_template
    shift_x_mm: 0.0
    shift_y_mm: 0.0
    rotation_deg: 0.0
```

---

### Структура `templates`

Шаблон описывает **локальную геометрию** одной спицы. Все смещения задаются в координатах `(along, across)` относительно **нуля шаблона** (который затем сдвигается и поворачивается для каждой конкретной спицы).

#### Параметры шаблона

| Поле | Тип | Описание |
|------|-----|----------|
| `power_via` | словарь, опционально | Параметры силового via |
| `components` | список, опционально | Слоты для компонентов (роли) |

#### Секция `power_via`

| Поле | Тип | Описание |
|------|-----|----------|
| `offset_along_mm` | число | Смещение вдоль оси шаблона (мм) |
| `offset_across_mm` | число | Смещение поперёк оси шаблона (мм) |
| `net` | строка, опционально | Имя цепи (если не указано – берётся из `rule.net`) |
| `drill_mm` | число | Диаметр сверления (мм) |
| `diameter_mm` | число | Внешний диаметр via (мм) |

#### Секция `components` (слот компонента)

Каждый слот описывает один компонент с **ролью**.

| Поле | Тип | Описание |
|------|-----|----------|
| `role` | строка **обязательно** | Имя роли (например, `LIGHT`, `HEAVY`, `XTAL`) |
| `offset_along_mm` | число | Смещение вдоль оси шаблона (мм) |
| `offset_across_mm` | число | Смещение поперёк оси шаблона (мм) |
| `angle_deg` | число | Угол поворота компонента (градусы) |
| `gnd_via_offset_along_mm` | число | Смещение GND via от **реального** GND-пада компонента (вдоль оси шаблона) |
| `gnd_via_offset_across_mm` | число | Смещение GND via поперёк оси шаблона |
| `gnd_via_net` | строка | Имя цепи для GND via (обычно `GND`) |
| `gnd_via_drill_mm` | число | Диаметр сверления GND via (мм) |
| `gnd_via_diameter_mm` | число | Внешний диаметр GND via (мм) |

---

### Структура `rules`

Правила группируют спицы по цепям питания.

#### Секция `rule`

| Поле | Тип | Описание |
|------|-----|----------|
| `net` | строка | Имя цепи (например, `+3V3_VCCIO`) |
| `spokes` | список | Список спиц для этой цепи |

#### Секция `spoke`

| Поле | Тип | Описание |
|------|-----|----------|
| `pad` | строка | Номер пада микросхемы (целевого компонента) |
| `template` | строка | Имя шаблона из секции `templates` |
| `shift_x_mm` | число | Сдвиг нуля шаблона по X (в координатах KiCad, мм) |
| `shift_y_mm` | число | Сдвиг нуля шаблона по Y (в координатах KiCad, мм) |
| `rotation_deg` | число | Поворот шаблона (в градусах) |
| `enabled` | булево, опционально | Включена ли спица (по умолчанию `true`) |

**Порядок применения:**
1. Нулевая точка шаблона = центр пада + `(shift_x, shift_y)`.
2. Весь шаблон поворачивается на `rotation_deg` как единое целое.

## Автоматический подбор компонентов по ролям

В **Decap Placer v4.0** refdes компонентов **не прописываются в конфиге**. Вместо этого программа сама находит компоненты на плате по двум критериям:

1. **Цепь** – компонент должен быть подключён к той же цепи, что и правило (`rule.net`).
2. **Роль** – на компоненте должно быть задано пользовательское поле `Role` (в Eeschema), значение которого совпадает с одной из ролей, указанных в шаблоне.

### Как задать роли в схеме

1. Откройте символ компонента (например, конденсатора) в редакторе схем (Eeschema).
2. Добавьте новое пользовательское поле:
   - **Имя:** `Role`
   - **Значение:** например, `LIGHT` или `HEAVY`
3. Повторите для всех конденсаторов, которые должны участвовать в расстановке.
4. Выполните **Update PCB from Schematic**, чтобы поля перенеслись на плату.

### Как происходит распределение

- Для каждого правила (`rule`) программа собирает все компоненты на плате, у которых есть поле `Role` и которые подключены к цепи `rule.net`.
- Компоненты каждой роли сортируются в **естественном численном порядке** (например, `C5`, `C10`, `C30`).
- Спицы разбирают пул **по очереди** – первая спица берёт первый доступный компонент каждой роли, вторая – следующий, и т.д.
- Если компонентов какой-то роли не хватает для всех спиц, программа остановится с ошибкой **до** каких-либо изменений на плате.

---

## Термовиа (thermal via array)

Массив отверстий под термопадом задаётся в корневой секции `thermal_via_array`:

| Поле | Тип | Описание |
|------|-----|----------|
| `enabled` | булево | Включить массив |
| `target_ref` | строка | Refdes компонента с термопадом (обычно `IC1`) |
| `pad` | строка | Номер пада термопада |
| `net` | строка | Цепь для термовиа (обычно `GND`) |
| `rows` | число | Количество рядов |
| `cols` | число | Количество столбцов |
| `margin_mm` | число | Отступ от края площадки (мм) |
| `pattern` | строка | `grid` или `staggered` |
| `drill_mm` | число | Диаметр сверления |
| `diameter_mm` | число | Внешний диаметр via |

---

## Проверка конфигурации (валидация)

Перед выполнением программа запускает две проверки:

1. **Существование шаблонов и падов** – все `template` и `pad` должны существовать.
2. **Достаточность компонентов по ролям** – для каждой роли должно быть достаточно компонентов на соответствующей цепи.

Если хотя бы одна проверка не проходит, программа выдаёт полный список ошибок и завершается, **не трогая плату**.

## Пример полного конфига

```yaml
target_ref: IC1
side: back

thermal_via_array:
  enabled: true
  target_ref: IC1
  pad: '145'
  net: GND
  rows: 4
  cols: 4
  margin_mm: 0.5
  pattern: grid
  drill_mm: 0.3
  diameter_mm: 0.5

templates:
  cap_pair:
    power_via:
      offset_along_mm: 0.0
      offset_across_mm: -1.5
    components:
      - role: LIGHT
        offset_along_mm: 1.0
        offset_across_mm: -1.0
        angle_deg: 90.0
        gnd_via_offset_along_mm: 0.0
        gnd_via_offset_across_mm: -1.0
      - role: HEAVY
        offset_along_mm: 1.0
        offset_across_mm: 2.0
        angle_deg: 270.0
        gnd_via_offset_along_mm: 0.0
        gnd_via_offset_across_mm: 1.3

rules:
- net: +1V2_VCCINT
  spokes:
  - pad: '109'
    template: cap_pair
    shift_x_mm: 0.0
    shift_y_mm: 0.0
    rotation_deg: 90.0
  - pad: '62'
    template: cap_pair
    shift_x_mm: 0.4
    shift_y_mm: 0.0
    rotation_deg: 270.0
```

## Примечания по использованию

- **Dry-run** не показывает via – их позиции зависят от реального положения компонентов после перемещения, которое в dry-run не происходит.
- **Двухфазное выполнение** – сначала перемещаются компоненты, затем плата перечитывается, и только потом создаются via. Это гарантирует, что GND via рассчитываются по реальному положению компонентов.
- **Откат (undo)** – каждая успешная операция логируется в `logs/operation_*.json`. Команда `undo` восстанавливает исходное состояние.
- **Коллизии** – проверка основана на упрощённой модели и может давать ложные срабатывания. Используйте `--no-collision-check`, если KiCad не выдаёт ошибок.
- **Роли в схеме** – поле `Role` должно быть добавлено в Eeschema и перенесено на плату через **Update PCB from Schematic** перед запуском.

---

## Устранение неполадок

| Симптом | Вероятная причина | Решение |
|---------|-------------------|---------|
| `ValidationError: не хватает компонентов с ролью ...` | Недостаточно компонентов с данной ролью на нужной цепи | Проверьте поле `Role` в схеме и подключение к цепи |
| `ValidationError: шаблон ... не найден` | Опечатка в имени шаблона | Исправьте имя в `template` |
| `KiCad is busy` | IPC-соединение занято | Закройте диалоги в KiCad, перезапустите скрипт |
| `Много предупреждений о коллизиях` | Ложные срабатывания | Используйте `--no-collision-check` |
| `Undo не работает` | Файл лога повреждён или удалён | Проверьте папку `logs/` – там должен быть файл операции |

# Модули верхнего уровня

## `config.py` – Загрузка и хранение конфигурации

### Назначение

Модуль определяет все структуры данных (dataclass'ы), описывающие конфигурацию расстановки, и предоставляет функцию `load_config()` для загрузки YAML-файла в типизированные объекты Python. Это единственное место, где задаётся формат входного файла.

### Структуры данных (dataclass'ы)

| Класс | Описание |
|-------|----------|
| `ThermalViaArrayConfig` | Настройки массива термовиа под термопадом |
| `TemplatePowerVia` | Описание power via внутри шаблона спицы (локальные координаты `along/across`, параметры via) |
| `TemplateComponentSlot` | Описание одного слота (компонента) в шаблоне: роль, локальные координаты, угол, смещение для GND via |
| `SpokeTemplate` | Полный шаблон спицы: имя, опциональный power via и список слотов компонентов |
| `ManualSpoke` | Конкретная спица на плате: пад, ссылка на шаблон, сдвиг (`shift_x`, `shift_y`), поворот (`rotation_deg`), флаг `enabled` |
| `Rule` | Правило для одной цепи питания: имя цепи и список спиц |
| `Config` | Главный объект: целевой компонент, сторона, словарь шаблонов, массив термовиа, правила, флаги (`place_components`, `skip_existing_components`) и параметры поиска свободного места для термовиа |

### Ключевые функции

| Функция | Описание |
|---------|----------|
| `load_config(path)` | Загружает YAML-файл, парсит его и возвращает экземпляр `Config`. Устанавливает значения по умолчанию, логирует результат. |
| `_load_template_power_via(data)` | Загружает `TemplatePowerVia` из словаря |
| `_load_template_component_slot(data)` | Загружает `TemplateComponentSlot` |
| `_load_spoke_template(name, data)` | Загружает `SpokeTemplate` |
| `_load_manual_spoke(data)` | Загружает `ManualSpoke` |

### Примечания

- Все координаты задаются в **миллиметрах**.
- Углы задаются в **градусах**.
- Поле `net` в `TemplatePowerVia` опционально – если не указано, берётся из `rule.net`.
- Слоты компонентов теперь используют **роли** вместо конкретных refdes – refdes подбираются автоматически из пула (см. `component_pool.py`).

## `exceptions.py` – Пользовательские исключения

### Назначение

Определяет иерархию исключений для проекта, чтобы отличать ошибки, связанные с логикой расстановки, от системных. Все пользовательские исключения наследуются от базового `PlacerError`.

### Классы исключений

| Класс | Назначение |
|-------|------------|
| `PlacerError` | Базовое исключение для всех ошибок планера |
| `BoardNotFoundError` | Не удалось получить плату из KiCad |
| `ComponentNotFoundError` | Компонент не найден на плате |
| `GeometryError` | Ошибка в геометрических расчётах |
| `ValidationError` | Фатальная ошибка предварительной проверки конфигурации – программа останавливается до изменения платы |

### Использование

- `ValidationError` выбрасывается в `validation.py` и перехватывается в `decap_placer.py` для вывода понятного сообщения.
- Все остальные исключения перехватываются на верхнем уровне и логируются.

## `undo.py` – Откат последней операции

### Назначение

Реализует команду `undo`, которая восстанавливает состояние платы до выполнения последней операции расстановки. Использует JSON-логи, создаваемые `executor.py` при каждом успешном применении изменений.

### Основная функция

`undo_last_operation(json_path: Path) -> bool`

**Алгоритм:**
1. Загружает JSON-файл лога.
2. Для каждого перемещённого компонента:
   - Определяет исходный слой (`original_layer`).
   - Если текущий слой отличается – выполняет флип (через `adapter.flip_selected`).
   - Восстанавливает исходную позицию и угол (через `update_items`).
3. Для каждой созданной via:
   - Удаляет её по UUID (через `board.remove_items_by_id`).
4. Удаляет файл лога (чтобы нельзя было откатить одну операцию дважды).

### Примечания

- Файлы логов хранятся в папке `logs/` с именем `operation_YYYYMMDD_HHMMSS.json`.
- Лог содержит два раздела: `moves` (информация о перемещениях) и `created_vias` (информация о созданных via).
- Для корректного восстановления слоя используется явное преобразование `BoardLayer` в строку `"F.Cu"`/`"B.Cu"` (вместо `str()`).

## `validation.py` – Предварительные проверки конфигурации

### Назначение

Выполняет фатальные проверки конфигурации **до** любых изменений на плате. Если обнаружена проблема – программа останавливается с подробным списком ошибок, не трогая плату.

### Основные функции

| Функция | Описание |
|---------|----------|
| `check_templates_and_pads_exist(adapter, cfg)` | Проверяет, что каждая спица ссылается на существующий шаблон и существующую площадку целевого компонента. Выключает отключённые спицы (`enabled=False`). |
| `check_role_pool_sufficiency(adapter, cfg)` | Для каждого правила считает, сколько компонентов каждой роли требуется всеми спицами, и сверяет с реальным количеством компонентов на плате (по цепи и полю `Role`). Если не хватает – сообщает все нехватки разом. |
| `run_all_checks(adapter, cfg)` | Запускает обе проверки по порядку. |

### Как работает `check_role_pool_sufficiency`

1. Для каждого правила (`rule`) собирает все роли, необходимые его спицам.
2. Строит `ComponentPool` для этой цепи (читает все компоненты на плате, у которых есть поле `Role` и которые подключены к этой цепи).
3. Для каждой роли сравнивает требуемое количество с доступным в пуле.
4. Если хотя бы одно несовпадение – выбрасывает `ValidationError` с полным списком проблем.

### Использование в `decap_placer.py`

Перед вызовом `planner.plan_moves()`:
```python
validation.run_all_checks(adapter, cfg)
```

Это гарантирует, что ни одна операция не будет выполнена, если конфиг содержит ошибки.

### Примечания

- Проверки выполняются **до** подключения к KiCad? Нет, после подключения, но до любых изменений.
- Ошибки выводятся в формате, понятном пользователю – с указанием цепи, роли и недостающего количества.
- Если компонентов с нужной ролью нет вообще, программа сообщит об этом и остановится.

---

## Взаимосвязи

- `config.py` загружается в `decap_placer.py:cmd_apply()`.
- `validation.py` использует `adapter` и `cfg` для проверок.
- `undo.py` использует `adapter` для восстановления состояния.
- Все модули используют `exceptions.py` для сигнализации об ошибках.

# `geometry/` – Геометрические утилиты

## Назначение

Модули в директории `geometry/` предоставляют низкоуровневые геометрические функции и классы, используемые для расчёта позиций компонентов и via, построения ограничивающих областей (keepout), генерации сетки термовиа и преобразования локальных координат шаблонов в глобальные.

Эти модули **не зависят** от KiCad или адаптера – они оперируют только координатами и векторами, что делает их легко тестируемыми и переиспользуемыми.

---

## Структура

```
geometry/
├── __init__.py             # Экспорт публичных функций
├── keepout.py              # Ограничивающие прямоугольники и поиск свободного места
├── pad_projection.py       # Предсказание позиции пада после перемещения/поворота компонента
├── placement.py            # [УСТАРЕЛ] – оставлен для совместимости, не используется
├── spoke_layout.py         # Преобразование локальных координат шаблона спицы в глобальные
└── thermal_grid.py         # Генерация сетки термовиа под площадкой
```

---

## Файлы и функции

### `keepout.py`

**Назначение:**  
Определяет класс `Rect` (выровненный по осям прямоугольник) и предоставляет функции для:
- Построения keepout-областей из bounding box'ов (с запасом `clearance`).
- Проверки точки на занятость (с учётом радиуса via).
- Поиска свободного места вокруг идеальной позиции (по спирали или вдоль прямой).

**Основные классы и функции:**

| Имя | Описание |
|-----|----------|
| `Rect` | AABB-прямоугольник с методами `from_bbox`, `from_circle`, `intersects`. |
| `point_is_clear(point, via_radius, keepout)` | Проверяет, свободна ли точка (окружность радиуса `via_radius` не пересекает keepout). |
| `build_keepout(bboxes, clearance_mm, mm_per_unit)` | Принимает список Box2 (от адаптера), строит список `Rect` с учётом зазора. |
| `find_free_point(ideal, keepout, via_radius, ...)` | Ищет ближайшую свободную точку по спирали (кольцами). Учитывает предпочтительное направление. |
| `find_free_point_along_line(ideal, keepout, via_radius, line_direction, ...)` | Ищет свободную точку вдоль прямой (в обе стороны от `ideal`). |

**Используется в:** `via_planner.py` для поиска места под термовиа (и ранее – для stitching via, но теперь via ставятся строго по шаблону).

---

### `pad_projection.py`

**Назначение:**  
Предсказывает абсолютную позицию конкретного пада компонента, если компонент будет перемещён в новое место и повёрнут на заданный угол (с учётом возможного зеркалирования при флипе на обратную сторону). Это критически важно для правильного расчёта позиции GND via, которая привязана к земляному паду конденсатора.

**Основные функции:**

| Имя | Описание |
|-----|----------|
| `local_pad_offset(fp, pad)` | Возвращает смещение пада относительно центра футпринта в **неповёрнутой** локальной системе координат (постоянная геометрия). |
| `predict_pad_position(fp, pad, dest, angle_deg, needs_flip)` | Предсказывает абсолютную позицию пада после перемещения футпринта в `dest` и поворота на `angle_deg` с учётом возможного зеркалирования при флипе. |

**Используется в:** `via_planner.py` (ранее для stitching via, теперь – только для расчёта GND via по уже перемещённому компоненту, но фактически `via_planner` использует реальный пад, а не предсказание; функция оставлена для потенциального использования в будущем).

---

### `spoke_layout.py`

**Назначение:**  
Преобразует локальные координаты шаблона спицы (`along`, `across`) в абсолютные координаты платы с учётом сдвига `(shift_x, shift_y)` и поворота `rotation_deg` конкретной спицы. Также вычисляет итоговый угол поворота компонента и сохраняет параметры GND via из шаблона.

Этот модуль является **ключевым** для новой ручной стратегии (DecapPlacer 4.0).

**Основные классы и функции:**

| Имя | Описание |
|-----|----------|
| `rotate_local_offset(along_mm, across_mm, rotation_deg)` | Поворачивает локальный вектор `(along, across)` на заданный угол вокруг нуля (без переноса). |
| `local_to_absolute(origin, along_mm, across_mm, rotation_deg)` | Преобразует локальный вектор в абсолютную позицию относительно `origin` с учётом поворота. |
| `ComponentLayout` | Структура, описывающая размещение одного компонента: ref, роль, позиция, угол, параметры GND via. |
| `SpokeLayout` | Структура, описывающая все элементы спицы: origin, power via (если есть), список компонентов. |
| `apply_spoke_geometry(pad_position, spoke, template, rule_net, role_to_ref)` | Основная функция – принимает позицию пада FPGA, данные спицы (`ManualSpoke`), шаблон (`SpokeTemplate`), цепь правила и сопоставление роль→ref. Возвращает `SpokeLayout` с абсолютными координатами. |

**Используется в:** `manual_position_calculator.py` (для расчёта позиций компонентов) и `via_planner.py` (для power via).

---

### `thermal_grid.py`

**Назначение:**  
Вычисляет абсолютные координаты для массива термовиа под термопадом компонента (например, `IC1`). Учитывает размер пада, отступы от края, количество строк/столбцов и возможность шахматного порядка (stagger).

**Основные функции:**

| Имя | Описание |
|-----|----------|
| `get_pad_size(pad)` | Возвращает размеры медного слоя пада `(width, height)` в нанометрах. |
| `compute_thermal_via_grid(pad, rows, cols, margin_mm, stagger)` | Генерирует список абсолютных позиций для via, размещённых равномерно внутри площадки с учётом отступов `margin_mm`. |

**Используется в:** `via_planner.py` для планирования термовиа.

---

### `placement.py` – **УСТАРЕЛ**

Этот файл содержит старую функцию `compute_position`, которая использовалась для автоматического размещения компонентов относительно границы зоны (`boundary`). После перехода на ручную стратегию (шаблоны) этот файл больше не используется и оставлен только для обратной совместимости со старыми тестами. В ближайшее время будет удалён.

- Все координаты и размеры во всех функциях ожидаются в **нанометрах**, если не указано иное (поля `_mm` принимают миллиметры и конвертируются внутри с помощью `MM = 1_000_000` из `utils/units.py`).

# `kicad/` – Адаптер для взаимодействия с KiCad через IPC

## Назначение

Модули в директории `kicad/` предоставляют унифицированный интерфейс для работы с платой в KiCad через библиотеку `kipy` (KiCad Python IPC). Они скрывают детали низкоуровневого протокола, обеспечивают устойчивость к сбоям связи (повторные попытки при ошибках «KiCad is busy»), упрощают выполнение транзакций (commit/rollback) и доступ к объектам платы.

Ключевые возможности:
- Поиск компонентов (футпринтов), падов, зон, цепей.
- Получение bounding box'ов объектов (для keepout и коллизий).
- Транзакционное обновление и создание объектов.
- Чтение пользовательских полей компонентов (например, `Role` для DecapPlacer 4.0).
- Переворот компонентов на другую сторону (flip) через GUI-действие.

---

## Структура

```
kicad/
├── __init__.py        # Экспорт публичного API
├── interfaces.py      # Абстрактный интерфейс IBoardAdapter
└── adapter.py         # Реализация KiCadBoardAdapter
```

---

## Файлы и функции

### `interfaces.py`

**Назначение:**  
Определяет абстрактный базовый класс `IBoardAdapter`, который описывает контракт для всех адаптеров к плате. Это позволяет легко подменять реализацию (например, для тестирования с моками).

**Методы интерфейса:**

| Метод | Описание |
|-------|----------|
| `refresh_board()` | Обновить данные платы из KiCad (перечитать все объекты). |
| `get_footprint(ref)` | Получить футпринт по референсу (например, `IC1`). |
| `get_footprints()` | Получить все футпринты платы. |
| `get_vias()` | Получить все существующие переходные отверстия (via). |
| `get_field_value(fp, field_name)` | Прочитать значение пользовательского поля компонента (например, `Role`). |
| `get_footprint_pads(fp)` | Получить площадки (пады) футпринта. |
| `get_pad_by_number(fp, number)` | Найти площадку по номеру (например, `'1'`, `'145'`). |
| `get_zone_by_name(name)` | Найти зону по имени (Rule Area). |
| `get_net_by_name(name)` | Найти цепь по имени. |
| `get_bounding_boxes(items)` | Получить ограничивающие прямоугольники (Box2) для списка объектов (батч-запрос). |
| `begin_commit()` | Начать транзакцию. |
| `push_commit(commit, description)` | Применить транзакцию. |
| `drop_commit(commit)` | Откатить транзакцию. |
| `update_items(items)` | Обновить существующие объекты на плате. |
| `create_items(items)` | Создать новые объекты на плате. |
| `flip_selected(footprints)` | Перевернуть выделенные футпринты (зеркалирование на другую сторону). |
| `commit_with_retry(description, work_fn, retries)` | Выполнить работу в транзакции с автоматическим повторением при ошибках IPC. |
| `create_via(position, net, drill_mm, diameter_mm)` | Создать объект `Via` (но не добавлять на плату – для добавления нужен `create_items`). |

---

### `adapter.py`

**Назначение:**  
Реализация `KiCadBoardAdapter`, наследующего `IBoardAdapter`. Подключается к KiCad через `kipy.KiCad` и предоставляет все методы интерфейса, а также дополнительные внутренние утилиты.

**Особенности реализации:**

- **Устойчивость к сбоям:** все методы чтения (`get_*`) используют повторные попытки при возникновении `ApiError` с сообщением «KiCad is busy».
- **Батч-запросы:** `get_bounding_boxes()` принимает список объектов и возвращает Box2 для всех одним запросом (экономит вызовы IPC).
- **Транзакции:** `commit_with_retry` оборачивает произвольную функцию в транзакцию с автоматическим откатом при ошибке.
- **Чтение пользовательских полей:** `get_field_value()` фильтрует `texts_and_fields` по типу `Field` (а не `BoardText`) и возвращает значение поля с указанным именем.
- **Флип:** `flip_selected` использует GUI-действие `pcbnew.InteractiveEdit.flip` для реального переворота компонента (зеркалирование площадок и шелкографии). После вызова необходимо перечитать футпринты через `refresh_board()`.
- **Создание via:** `create_via` заполняет объект `Via` (тип, позиция, цепь, диаметры) – объект не добавляется на плату, пока не будет передан в `create_items`.

**Ключевые методы (дополнительно к интерфейсу):**

| Метод | Описание |
|-------|----------|
| `refresh_board()` | Перечитывает плату из KiCad. Обязательно вызывать после операций, изменяющих плату «в обход» `update_items` (например, после `flip_selected`). |
| `get_bounding_boxes(items)` | Оптимизированный батч-запрос для получения Box2 для нескольких элементов. |
| `get_field_value(footprint, field_name)` | Ищет в `footprint.texts_and_fields` объект типа `Field` с именем `field_name` и возвращает его значение (или `None`). |

---

### `__init__.py`

**Назначение:**  
Экспортирует публичные классы для удобного импорта из других модулей.

```python
from .adapter import KiCadBoardAdapter
from .interfaces import IBoardAdapter
```

## Взаимосвязи с другими модулями

Адаптер используется во всех частях проекта, которые взаимодействуют с KiCad:

- **`placement/planner.py`** – для получения целевого компонента (IC) и его падов.
- **`placement/executor.py`** – для выполнения перемещений, флипа и создания via (включая логирование и undo).
- **`placement/services/manual_position_calculator.py`** – для поиска футпринтов по refdes и чтения ролей (`get_field_value`).
- **`placement/services/via_planner.py`** – для чтения падов, создания via и проверки существующих via (идемпотентность).
- **`placement/services/component_pool.py`** – для построения пула компонентов по полю `Role` и цепи.
- **`validation.py`** – для проверки существования падов и шаблонов.
- **`diagnostics/`** – для диагностических скриптов (получение bbox, отладка).

Благодаря интерфейсу `IBoardAdapter`, код не зависит от конкретной реализации и может быть легко протестирован с мок-объектами.

## Примечания

- Все координаты и размеры в адаптере ожидаются в **нанометрах** (внутренние единицы KiCad). Для перевода из мм в нм используется константа `MM = 1_000_000` из `utils/units.py`.
- В транзакциях важно перехватывать исключения и корректно откатывать `commit`, чтобы не оставлять незавершённых изменений.
- Метод `flip_selected` не обновляет локальный объект футпринта – после его вызова необходимо перечитать футпринт через `refresh_board()` и повторно получить объект.
- При чтении пользовательских полей через IPC важно, чтобы поле было добавлено в схеме (Eeschema) и перенесено на плату через **Update PCB from Schematic**. Поле `Role` используется в DecapPlacer 4.0 для автоматического подбора компонентов.
- Адаптер использует `kipy.KiCad` с таймаутом по умолчанию 20000 мс, но это значение можно изменить через параметр `timeout_ms` при создании экземпляра.

- `kipy` – официальная Python-библиотека для IPC KiCad (устанавливается отдельно).
- `kipy.board_types` – типы данных KiCad (FootprintInstance, Pad, Via, Net, Field и др.).
- `kipy.geometry` – геометрические примитивы (Vector2, Box2, Angle).
- `kipy.errors.ApiError` – для обработки ошибок IPC.

# `placement/` – Планирование и исполнение расстановки

## Назначение

Директория `placement/` содержит основную логику расстановки компонентов и создания переходных отверстий. Она координирует все этапы процесса:

1. **Планирование** – вычисление целевых позиций компонентов на основе шаблонов и автоматического подбора refdes через пул ролей.
2. **Исполнение** – применение перемещений и создание via на плате через адаптер KiCad, с разделением на две фазы (сначала перемещения, затем via) и обязательным перечитыванием платы между ними.
3. **Логирование и откат** – сохранение информации об операции в JSON для команды `undo`.
4. **Проверка коллизий** – упрощённая проверка перекрытий (опционально).
5. **Идемпотентность** – пропуск уже существующих via и компонентов, уже стоящих на целевых позициях (при включённом флаге `skip_existing_components`).

Все сервисы используют адаптер `kicad/adapter.py` и геометрические утилиты `geometry/`.

**Используется в:** `planner.py`, `executor.py`, `manual_position_calculator.py`, `via_planner.py`.

### `collision.py`

**Назначение:**  
Упрощённая проверка коллизий между компонентами (по кругам-приближениям). Использует реальные bounding box'ы через адаптер для вычисления радиусов (половина диагонали bbox).

**Основные функции:**

| Функция | Описание |
|---------|----------|
| `compute_radii(footprints, adapter)` | Вычисляет радиусы для списка футпринтов (батч-запрос через адаптер). |
| `footprints_overlap(pos1, r1, pos2, r2, margin_mm)` | Проверяет перекрытие двух кругов с запасом. |
| `check_collisions(moves, all_footprints, adapter, ignore_refs, margin_mm)` | Проверяет коллизии между перемещаемыми компонентами и остальными. Возвращает список конфликтных пар (ref1, ref2, расстояние). |

**Используется в:** `executor.py` (опционально, при включённой проверке).  
**Примечание:** проверка может давать ложные срабатывания, поэтому отключается флагом `--no-collision-check`.

---

### `executor.py`

**Назначение:**  
Применяет команды перемещения и создания via к реальной плате через адаптер. Разделяет фазы (сначала перемещения, потом via) с обязательным перечитыванием платы между ними для корректного расчёта GND via по реальным позициям.

**Класс `BatchExecutor`:**

| Метод | Описание |
|-------|----------|
| `__init__(adapter, config, batch_size)` | Инициализация с размером батча для коммитов. |
| `execute_moves(moves, check_collisions, collision_margin_mm)` | Применяет только перемещения. Возвращает список refdes, которые не удалось применить. |
| `execute_vias(vias)` | Создаёт via. Возвращает список owner_ref'ов, для которых via не создались. |
| `execute(moves, vias, ...)` | Обратно совместимая обёртка (не рекомендуется для боевого использования – не перечитывает плату между фазами). |
| `_needs_flip(cmd, fp_by_ref)` | Проверяет, нужно ли перевернуть компонент на целевой слой. |
| `_flip_in_batches(refs, fp_by_ref)` | Переворачивает компоненты батчами через GUI-action. |
| `_write_operation_log(move_log, via_log)` | Сохраняет JSON-лог операции для undo. |

**Важно:**  
После `execute_moves()` вызывающий код обязан выполнить `adapter.refresh_board()`, чтобы `execute_vias()` работала с актуальными данными (GND via рассчитываются от реально размещённых компонентов).

**Используется в:** `decap_placer.py` (главный скрипт) для применения плана.

---

### `planner.py`

**Назначение:**  
Главный планировщик – координирует расчёт позиций (через `ManualPositionCalculator`) и via (через `ViaPlanner`). Содержит двухфазную логику: сначала планируются перемещения, затем via. Также реализует логику пропуска уже стоящих на месте компонентов (`skip_existing_components`).

**Класс `PlacementPlanner`:**

| Метод | Описание |
|-------|----------|
| `__init__(adapter, config)` | Инициализация, поиск целевого компонента (IC), определение слоя. |
| `_already_in_place(ref, dest, angle_deg)` | Проверяет, находится ли компонент уже на целевой позиции (с учётом слоя, позиции и угла). Допуски: 0.01 мм по координатам, 0.1° по углу. |
| `plan_moves()` | Использует `ManualPositionCalculator` для расчёта позиций компонентов. Возвращает список `MoveCommand`. Учитывает `place_components` и `skip_existing_components`. |
| `plan_vias()` | Использует `ViaPlanner` для расчёта via. Возвращает список `ViaCommand`. Вызывается **после** применения перемещений и перечитывания платы. |
| `plan()` | Обратно совместимая обёртка (вызывает `plan_moves()` и `plan_vias()` подряд). Для боевого использования не рекомендуется (см. `decap_placer.py`). |

**Используется в:** `decap_placer.py` для получения плана.

### `services/`

#### `services/__init__.py`

Экспортирует публичные классы сервисов:
```python
from .component_pool import ComponentPool
from .manual_position_calculator import ManualPositionCalculator
from .via_planner import ViaPlanner
```

#### `services/component_pool.py`

**Назначение:**  
Подбирает конкретные refdes компонентов для ролей шаблона. Пул строится один раз на каждое правило (`rule.net`) и разбирается спицами этого правила по очереди.

**Класс `ComponentPool`:**

| Метод | Описание |
|-------|----------|
| `__init__(adapter, net_name, roles)` | Строит пул для заданной цепи: читает все футпринты, у которых есть поле `Role` и которые подключены к `net_name`. |
| `_build()` | Внутренний метод, собирает списки refdes по ролям, сортирует их в естественном численном порядке (`C5` < `C10`). |
| `pop(role, spoke_pad)` | Забирает следующий компонент с указанной ролью. Если пул исчерпан – выбрасывает `ValidationError`. |
| `remaining_count(role)` | Возвращает количество оставшихся компонентов с данной ролью. |

**Используется в:** `manual_position_calculator.py` для получения refdes для каждой спицы.

---

#### `services/manual_position_calculator.py`

**Назначение:**  
Расчёт абсолютных позиций компонентов на основе шаблонов спиц (`SpokeTemplate`) и конкретных данных спицы (`ManualSpoke`). Не использует геометрию зоны – только пад FPGA + сдвиг/поворот + локальные координаты шаблона. Подбирает refdes через `ComponentPool`.

**Класс `ManualPositionCalculator`:**

| Метод | Описание |
|-------|----------|
| `__init__(adapter, config)` | Инициализация с адаптером и конфигом. |
| `compute_raw_positions(target_fp, rules, side)` | Для каждого правила строит `ComponentPool`, затем для каждой спицы вызывает `apply_spoke_geometry` с полученным сопоставлением роль→ref. Возвращает список `PlacedComponentInfo`. |

**Используется в:** `planner.py`.

---

#### `services/via_planner.py`

**Назначение:**  
Планирует создание всех via:
- **Power via** – из шаблона спицы (локальные координаты, повёрнутые на `rotation_deg`).
- **GND via** – по локальному смещению от **реального** земляного пада компонента (используя уже размещённый компонент).
- **Thermal via** – сетка под термопадом (поиск свободных точек через keepout).

Поддерживает **идемпотентность** – если включён флаг `skip_existing_components`, via, уже существующие на плате (в пределах допуска 0.01 мм), пропускаются.

**Класс `ViaPlanner`:**

| Метод | Описание |
|-------|----------|
| `__init__(adapter, config)` | Инициализация. |
| `_via_already_exists(existing_vias, position, net_name)` | Проверяет, существует ли уже via с данной цепью в указанной позиции (с допуском 0.01 мм). |
| `plan_vias(planned, target_fp, rules, target_layer)` | Основной метод. Строит keepout (для термовиа), вызывает `_plan_power_vias`, `_plan_gnd_vias`, `_plan_thermal_vias`. Возвращает список `ViaCommand`. |
| `_plan_power_vias(target_fp, rules, existing_vias)` | Power via – из шаблонов спиц. |
| `_plan_gnd_vias(planned, existing_vias)` | GND via – от реального GND-пада компонента + локальное смещение, повёрнутое на `rotation_deg`. |
| `_plan_thermal_vias(planned, target_fp, keepout, existing_vias)` | Термовиа – сетка с поиском свободных точек через `find_free_point`. |
| `_build_keepout(target_fp, planned, exclude)` | Строит keepout из падов IC и компонентов (для термовиа). |

**Используется в:** `planner.py` (после перемещений).

## Взаимосвязи с другими модулями

- **`kicad/adapter.py`** – используется для всех операций с платой (чтение, запись, транзакции).
- **`geometry/spoke_layout.py`** – используется в `manual_position_calculator.py` и `via_planner.py` для преобразования локальных координат в глобальные.
- **`geometry/thermal_grid.py`** и **`geometry/keepout.py`** – используются в `via_planner.py` для термовиа и keepout.
- **`config.py`** – предоставляет структуры данных (Config, ManualSpoke, SpokeTemplate и т.д.).
- **`validation.py`** – выполняет предварительные проверки перед вызовом планировщика.
- **`utils/units.py`** – константа `MM` для перевода миллиметров в нанометры.

---

## Примечания по использованию

- **Двухфазный процесс** (обязателен для корректного расчёта GND via):  
  1. Выполнить `plan_moves()` → `execute_moves()`.  
  2. Выполнить `adapter.refresh_board()`.  
  3. Выполнить `plan_vias()` → `execute_vias()`.  
  Это реализовано в `decap_placer.py:cmd_apply()`.

- **Коллизии** – проверка включена по умолчанию, но может давать ложные срабатывания. Отключается флагом `--no-collision-check`.

- **Логирование операции** – сохраняется в `logs/operation_*.json` и используется командой `undo`.

- **Dry-run** – отображает только перемещения; via не показываются, так как их позиции зависят от реального размещения компонентов (которое в dry-run не происходит).

- **Идемпотентность** – включение `skip_existing_components: true` позволяет безопасно перезапускать скрипт для добавления новых via без повторного перемещения уже стоящих компонентов и без создания дублирующих via.

- **Автоматический подбор refdes** – через `ComponentPool` по полю `Role` в схеме. Убедитесь, что поле добавлено в Eeschema и перенесено на плату через **Update PCB from Schematic**.

# `utils/` – Вспомогательные утилиты

## Назначение

Директория `utils/` содержит общеиспользуемые вспомогательные модули и константы, которые не относятся к основной логике расстановки, но необходимы для работы проекта. В основном это конвертация единиц измерения и вспомогательные функции для упрощения кода в других модулях.

---

## Структура

```
utils/
├── __init__.py        # Экспорт публичных функций
└── units.py           # Константы для перевода единиц измерения
```

---

## Файлы и функции

### `units.py`

**Назначение:**  
Содержит константу для перевода миллиметров в нанометры – внутреннюю систему координат KiCad (и библиотеки `kipy`). Все координаты и размеры в проекте хранятся в нанометрах, а пользовательские значения в конфигурации задаются в миллиметрах.

**Константа:**

| Имя | Значение | Описание |
|-----|----------|----------|
| `MM` | `1_000_000` | Количество нанометров в одном миллиметре. Используется для перевода ввода/вывода и в геометрических расчётах. |

**Используется во всех модулях, работающих с координатами:**  
- `kicad/adapter.py` – при создании via, обновлении позиций.
- `geometry/*.py` – во всех геометрических вычислениях.
- `placement/*.py` – при расчёте позиций и via.
- `diagnostics/*.py` – для отладки и вывода значений в мм.
- `undo.py` – для восстановления позиций.

---

### `__init__.py`

**Назначение:**  
Экспортирует публичные функции (пока только `MM`).  
Обычно содержит:
```python
from .units import MM
```

Это позволяет использовать `from decap_placer.utils import MM` вместо полного пути.

---

## Использование в проекте

Во всех модулях, где требуется пересчёт между миллиметрами и нанометрами, используется константа `MM`:

```python
from decap_placer.utils import MM

# Перевести миллиметры в нанометры
value_nm = value_mm * MM

# Перевести нанометры в миллиметры (для вывода)
value_mm = value_nm / MM
```

---

## Примечания

- Все координаты в `kipy` (и в адаптере) ожидаются в нанометрах, поэтому конвертация выполняется **всегда** при передаче данных из конфига (в мм) в API KiCad.
- При отладке и логировании для удобства чтения координаты часто выводятся в мм, для чего используется деление на `MM`.
- Константа `MM` является целым числом, поэтому умножение/деление выполняется точно без потери точности (если значения заданы в мм с достаточной точностью).

---

## Взаимосвязи

- `utils` не зависит от других модулей проекта, но используется практически всеми.
- В `geometry/`, `placement/`, `kicad/` и `diagnostics/` обязательно импортируют `MM` для работы с координатами.

# Отчёт о тестировании DecapPlacer 4.0

## Общая информация

В проекте используется **pytest** для автоматизированного тестирования. Все тесты выполняются **без подключения к реальному KiCad** – используется `unittest.mock` для имитации адаптера и объектов платы. Это обеспечивает быстрое выполнение и независимость от внешнего окружения.

Тесты покрывают все ключевые аспекты новой архитектуры: шаблоны, роли, автоматический подбор компонентов, двухфазное выполнение, идемпотентность, валидацию и откат операций.

---

## Структура тестов

Все тесты находятся в папке `tests/` в корне проекта. Каждый файл тестирует отдельный модуль или функциональный блок.

## 1. Юнит-тесты геометрии и вспомогательных модулей

### `test_spoke_layout.py`

**Назначение:**  
Проверяет корректность преобразования локальных координат шаблона спицы в абсолютные координаты платы (модуль `geometry/spoke_layout.py`).

**Что тестируется:**
- Поворот локального вектора (`rotate_local_offset`) – сверка с формулой `kipy`.
- Применение сдвига и поворота шаблона к конкретной спице (`apply_spoke_geometry`).

**Зачем:**  
Геометрия – основа расстановки. Эти тесты гарантируют, что все компоненты и via попадут точно в расчётные позиции, независимо от поворота и сдвига.

**Назначение:**

**Что тестируется:**
- Отсутствие изменений при нулевом переносе/повороте.
- Заметное изменение позиции пада при повороте на 90°.
- Зеркалирование локального X при флипе (`needs_flip=True`).

**Зачем:**  
Эта логика критична для правильного расчёта GND via (via ставится относительно земляного пада). Тест защищает от регрессий, которые ранее приводили к via на площадках конденсаторов.

### `test_kicad.py`

**Назначение:**  
Проверяет, что адаптер `KiCadBoardAdapter` имеет все методы, объявленные в интерфейсе `IBoardAdapter`, и что импорты работают.

**Что тестируется:**
- Наличие всех методов интерфейса.
- Конструктор адаптера не падает при создании (без вызова `refresh_board`).

**Зачем:**  
Это «дымовой» тест, гарантирующий, что адаптер соответствует контракту и может быть использован в основном коде.

## 3. Регрессионные тесты на исправленные баги

### `test_two_phase_execution.py`

**Зачем:**

### `test_undo_layer.py`

**Назначение:**  
Регрессионный тест на баг: `undo.py` не сохранял исходный слой компонента в JSON-лог, из-за чего при откате компонент возвращался на неверный слой.

**Что тестируется:**
- Корректное сохранение слоя в JSON-логе (явное преобразование в строку `"F.Cu"`/`"B.Cu"`).
- Восстановление слоя при откате (проверяется через логику `undo.py`).

**Зачем:**  
Undo – важная функция безопасности. Этот тест предотвращает потерю информации о слое, что могло бы привести к неправильному восстановлению платы.

### 🚨 `run_action` — официально нестабильный API

Самый критичный пункт. Это **единственная функция, которую разработчики KiCad официально маркируют как нестабильную** в документации.

*   **Что делает**: Используется в `adapter.flip_selected()` для выполнения GUI-действия `pcbnew.InteractiveEdit.flip`, которое переворачивает компонент на другую сторону платы.
*   **В чём нестабильность**: Это **нестабильный API (unstable API)**. Разработчики KiCad прямо предупреждают, что он «не предназначен для использования кем-либо, кроме разработчиков API», а имена действий и их поведение **не гарантируются** и могут измениться в любой версии.
*   **Риск**: Ваш скрипт может полностью сломаться при обновлении KiCad до новой версии, если изменится имя действия или его работа.

### ⚠️ Недокументированные или нестабильные структуры данных

Следующие элементы не имеют стабильной документации и их поведение может меняться.

*   **`fp.texts_and_fields`**: Используется в `adapter.get_field_value()` для чтения пользовательского поля `Role`. Это свойство не документировано. Его структура (список, содержащий объекты `Field` и `BoardText`) была определена эмпирически.
*   **`fp.definition.items`**: Используется в `adapter.get_footprint_pads()` для получения падов компонента. Это свойство также не документировано.
*   **`board.remove_items_by_id()`**: Используется в `undo.py` для удаления via по UUID. Этот метод был добавлен недавно (мерж-реквест от 30 июня 2025 года) и, вероятно, ещё не имеет стабильной документации.
*   **`board.get_item_bounding_box()`**: Используется в `adapter.get_bounding_boxes()` для получения ограничивающих прямоугольников. Хотя это часть API, его поведение с некоторыми объектами (например, с `GraphicShape`) может быть неочевидным.
*   **`Field` и `TextAttributes`**: Класс `Field` активно используется. Согласно коммитам в репозитории, некоторые его свойства (например, `visible`) были объявлены устаревшими и перенесены в `TextAttributes`. Это говорит о том, что даже документированная часть API может меняться.

### 💡 «Тонкие» моменты в работе с API

Это не прямые баги, но особенности, которые могут привести к неожиданному поведению, если о них не знать.

*   **Getter’ы возвращают копии**: При работе с объектами `kipy` (например, `fp.position`) важно помнить, что геттеры часто возвращают **копии** данных. Изменение атрибута полученной копии (`fp.position.x = ...`) не повлияет на реальный объект на плате. Необходимо присваивать новый объект целиком (`fp.position = Vector2(...)`). Это не ошибка, но частая ловушка.
*   **`begin_commit`**: Эта функция официально документирована, но в вашем коде есть важный паттерн. В методе `commit_with_retry` вы правильно обрабатываете случай, когда `begin_commit()` может упасть, и это критично для надёжности.

### 💎 Резюме и рекомендации

Наибольший риск для стабильности вашего проекта представляет использование **`run_action`** для переворота компонентов. Остальные пункты — это скорее «серые зоны» API, которые требуют осторожности, но не являются официально нестабильными.

Чтобы снизить риски, вы можете:
1.  **Изолировать `run_action`**: Обернуть его вызов в отдельную функцию и документировать как потенциально нестабильный.
2.  **Следить за обновлениями**: При переходе на новую версию KiCad в первую очередь проверять работу функции `flip_selected`.
3.  **Искать альтернативы**: По возможности, искать другие способы выполнения операций, которые используют более стабильные части API. Например, изменение слоя компонента через `fp.layer = ...` (если это будет работать без `flip`).
